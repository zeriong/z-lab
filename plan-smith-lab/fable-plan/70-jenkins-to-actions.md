# 수년 된 Jenkins → GitHub Actions 점진 이전
- 추론 프레임: min-diff
- 한 줄 요약: 이전 단위를 "Jenkinsfile"에서 "계약(입력→산출물+부수효과)"으로 재정의하고, 모든 단계를 롤백 한 문장 게이트로만 통과시킨다. 수년 묵은 파이프라인의 최대 적은 기술 부채가 아니라 "하는 김에"다.

## 프레임이 요구하는 시작점 — 롤백 한 문장이 입장권이다

이 플랜의 모든 단계는 실행 전에 다음 문장을 완성해야 한다: **"이 단계를 되돌리려면 ___한다."** 빈칸이 한 문장으로 안 채워지면 그 단계는 너무 크다 — 쪼개서 다시 온다. 이것은 선언이 아니라 게이트다: 아래 각 단계에 롤백 문장을 병기했고, 문장이 안 나오는 단계는 이 문서에 실리지 못했다.

게이트가 즉시 강제한 구조적 결정 세 가지:
1. **전환은 커밋 1개다.** "Jenkins 트리거 비활성 + Actions 실모드 활성"을 한 커밋으로 묶는다. 롤백 = "그 커밋을 revert한다."
2. **삭제는 전환이 아니다.** Jenkins 잡 삭제는 한 문장으로 롤백되지 않으므로 게이트를 통과하지 못한다 → 삭제는 전 파이프라인 전환 후의 별도 최종 단계로 격리된다.
3. **공유 축은 파이프라인별 소비만.** 시크릿·러너를 축 단위로 재설계하는 순간 롤백이 전 파이프라인에 걸린다. 이전 기간엔 각 파이프라인이 자기 몫(자기 시크릿 사본, 자기 러너 용량)만 소비하고, 축 자체의 개편은 최후로 민다.

## 층위 이동 — 이전 대상은 파일이 아니라 계약이다

"Jenkinsfile을 워크플로 YAML로 옮긴다"는 표현이 이 작업의 함정이다. 수년 된 Jenkins 잡의 실체는 파일이 아니라 **암묵 계약**이다: 어떤 트리거에 대해, 어떤 산출물(아티팩트·태그·배포)과 어떤 부수효과(알림·후속 잡 트리거·캐시 갱신)를 내놓는가. 파일은 옮겼는데 계약을 못 옮긴 이전이 "전환 후 미묘하게 다르다"는 사고의 근원이다.

따라서 파이프라인마다 이전의 첫 산출물은 YAML이 아니라 **계약서 1장**이다: 트리거 / 산출물 목록 / 부수효과 목록 / **동등성 판정 방법**(아래) / 의도적 차이(Actions에서 재현 불가한 플러그인 동작의 최소 대체 — 판정 제외 항목으로 등재). 검증의 정의도 여기서 바뀐다: "Actions가 초록불"이 아니라 **"같은 입력에 대해 두 시스템이 같은 계약을 이행한다."**

## 문제 정의 / 목표

수년 운영된 Jenkins(파이프라인 다수, 상호 트리거 체인, 오래된 크론 잡 포함)를 GitHub Actions로 이전한다. 목표는 속도가 아니라 **어느 시점에 중단해도 안전한 상태**다: 이전이 6개월 걸려도, 도중에 멈춰도, 절반은 Actions·절반은 Jenkins인 상태가 정상 운영 가능해야 한다.

## 명시적 가정 목록

- **가정 1 (전체가 걸린 가정): 각 파이프라인의 계약이 유한하게 문서화 가능하고, 동등성을 실측할 만큼 실행 빈도가 있다.** 틀리면(부수효과가 발굴 불가능하거나 연 1회 잡) 병행 검증이 성립하지 않는다 — 저빈도 잡은 수동 리허설로 판정 방법을 대체하되, 그 약화를 계약서에 명시한다.
- 가정 2: 병행 기간의 러너 이중 비용(같은 커밋을 두 시스템이 빌드)을 감당할 수 있다. 틀리면 동시 병행 파이프라인 수를 줄이고 기간을 늘린다 — 일정이 늘어질 뿐 구조는 불변.
- 가정 3: 저장소 정본이 이미 GitHub이다. 틀리면(SVN 등) 저장소 이전이 선행되어야 한다 — 이 플랜의 범위 밖, 별도 플랜.
- 가정 4: 파이프라인 대부분의 Jenkins 플러그인 의존 동작이 Actions에서 재현 또는 최소 대체 가능하다. 틀린 파이프라인은 "이전 불가"로 분류해 유예한다 — 이 경우 완료 정의가 "Jenkins 제로"에서 "Jenkins 최소 격리 운영"으로 바뀐다(완료 기준에 반영).

## 접근 및 단계

0. **인벤토리와 체인 지도.** 전체 잡 목록 + 마지막 실행 시각 + 잡 간 트리거 체인을 그린다. 6개월 이상 미실행 잡은 소유자 확인 후 "이전 제외" 목록으로(삭제가 아니라 분류 — 결정이지 누락이 아니다). → 검증: 목록을 Jenkins 설정 export와 대조해 누락 0. 롤백: 문서 작업이라 불필요.
1. **파이프라인 1개 선정 — 저위험·체인 독립·고빈도.** 첫 파이프라인은 학습 수단이다. → 검증: 체인 지도에서 상하류 간선 0.
2. **계약서 작성 → 동작 보존 이식.** 이식 커밋은 동작 보존만 — **버그도 계약의 일부다**(고치면 동등성 판정이 죽는다). 발견한 개선거리는 전부 **유혹 로그**에 적는다: "하는 김에" 충동을 항목별로 기록·기각하고, 전환 후 개선 백로그로 이관한다. 이전 커밋과 개선 커밋은 절대 한 diff에 살 수 없다 — 리뷰 체크 항목: "이 diff에 동작 변화가 있는가?" → 검증: 리뷰 통과 + 섀도 실행 성공. 롤백: "워크플로 파일을 삭제한다."
3. **섀도 병행 운영.** Actions는 같은 트리거로 돌되 부수효과를 차단하고(배포 dry-run, 알림 무음), 산출물은 격리 저장소로 보낸다.
   **숫자 전에 의미:** "일치"는 계약서의 판정 방법이 정의한다 — 재현 가능한 빌드는 체크섬, 비결정 빌드(타임스탬프 등)는 내용 목록+스모크 테스트로 대체하고 그 사실을 기록한다.
   **전환 게이트(기본값 있는 게이트):** 고빈도(일 1회 이상) = 연속 20회 일치. 중빈도 = 2주간 전 실행 일치·최소 5회. 저빈도 = 실입력 재현 리허설 2회. 불일치 1회 → 카운터 리셋 + 원인 문서화 전 전환 금지. **병행 4주 초과 시 기본값은 "전환 보류" 판정** — 보류도 유효한 결과다.
4. **전환 커밋 1개.** Jenkins 트리거 오프 + Actions 실모드 온. 전환 직후 첫 실사이클을 부수효과 감시 창으로 삼는다(섀도에선 부수효과를 실측하지 못했으므로). 롤백: "전환 커밋을 revert한다." Jenkins 잡은 비활성 상태로 보존.
5. **반복 — 그리고 세 번째 중복에서만 추출.** 파이프라인 2~N에 2~4단계를 반복한다. 동시 병행은 최대 3개(비교 노동과 러너 비용의 상한). 공통 워크플로/컴포지트 액션 추출은 **같은 패턴이 세 번째로 복붙될 때만** 허용 — 첫 두 번의 복붙은 정당한 행위다. 추출 자체도 min-diff로: 추출 커밋은 동작 보존, 파이프라인을 하나씩 갈아타며, 롤백은 "해당 파이프라인을 인라인 버전으로 되돌린다."
6. **체인 파이프라인.** 상호 트리거 체인은 말단(하류)부터 이전하고, 절단면에는 임시 브리지(Jenkins→Actions 웹훅 디스패치)를 둔다. 브리지는 부채로 등재하고 철거 조건(상류 전환 완료)을 병기한다. 롤백: "브리지 훅을 제거하고 원래 트리거를 되살린다."
7. **공유 축 — 최후에, 이것도 min-diff로.** 전 파이프라인 전환 후: 시크릿은 "사본 존재 확인 → Jenkins credentials 폐기 → 로테이션" 순서로, 시크릿 그룹별로 쪼개 진행한다(수년 묵은 저장소에 살았던 값은 유출 인접으로 간주 — 이것은 개선이 아니라 이전의 마감으로 분류한다). 러너 재설계(임시 러너·컨테이너화)는 개선 백로그로 — 이전 기간의 러너 원칙은 "Jenkins 에이전트와 최대한 동일한 환경" 하나뿐이다.
8. **좀비 기간과 해체.** Jenkins 컨트롤러를 30일간 정지 상태로 보존한다(분기·저빈도 잡의 누락 탐지 창). 무사고 확인 후 백업을 뜨고 해체한다.

## 대안과 기각 사유

- **빅뱅 전환(freeze 후 일괄 이전 스프린트):** 기각 — 롤백이 한 문장이 아니라 복구 프로젝트가 된다. 부활 조건: 파이프라인 5개 이하·전부 저위험·1주 freeze를 조직이 감당할 때.
- **공통 CI 템플릿 선행 구축("플랫폼 먼저"):** 기각 — 실측 없는 추상화 + 첫 전환 지연. 세 번의 법칙 위반. 부활 조건은 플랜에 내장되어 있다: 세 번째 중복이 실증되는 순간이 곧 부활 시점이다.
- **변환 도구(GitHub Actions Importer 등) 일괄 신뢰:** 절반 기각 — 이식 초안 생성기로는 채용하되, 도구 산출물도 동일한 계약서·섀도 게이트를 통과해야 한다. 도구는 2단계의 노동을 줄일 뿐 3단계를 면제하지 않는다.

## 리스크 & 완화

- **R1 병행 중 부수효과 이중 발생(배포 2회, 알림 2회).** → 섀도 모드의 부수효과 차단이 구조적 방어. 잔여 위험은 전환 직후 감시 창으로 흡수.
- **R2 "하는 김에"의 침투 — 이 프레임의 사망 원인 1위.** → 유혹 로그 의무화 + 커밋 분리 + 리뷰 체크 항목. 완료 기준에서 실측한다(아래).
- **R3 체인 절단으로 인한 사일런트 미실행.** → 체인 지도(0단계)와 브리지 철거 조건. 좀비 기간이 최종 안전망.
- **R4 병행 피로 장기화.** → 동시 병행 ≤3, 파이프라인당 병행 ≤4주(초과 시 보류 판정). 보류가 누적되면 이전 자체의 전제(가정 4)를 재심한다.
- **R5 저빈도 잡의 검증 공백.** → 리허설 2회 + 좀비 기간 30일. 그래도 연 1회 잡은 커버하지 못한다 — "리허설 기록으로 판정"을 계약서에 명시하는 것이 정직의 한계선이다.

## "완료"의 판정 기준

- **실측 문장: Jenkins 컨트롤러를 30일간 정지시킨 상태에서, 빌드·배포·정기 잡·알림이 단 한 건도 누락되지 않았다.** (30일로 커버되지 않는 저빈도 잡은 리허설 기록을 판정 대체물로 명시.)
- 모든 이전 파이프라인이 계약서 + 게이트 통과 기록(연속 일치 횟수 또는 리허설 2회)을 보유한다.
- 이전 기간의 전 커밋이 "이전/개선" 라벨 중 하나를 달고 있고, 이전 커밋에 동작 변화가 섞인 사례 0건(리뷰 기록으로 검증).
- 유혹 로그의 전 항목이 "기각 사유 명시" 또는 "개선 백로그 이관" 상태다 — 로그가 비어 있으면 성공이 아니라 기록 실패를 먼저 의심한다.
- Jenkins credentials store가 폐기되고, 그 안에 살았던 시크릿 전량이 로테이션되었다.
- "이전 제외"·"이전 불가" 목록이 소유자 확인과 함께 문서화되어 있다 — 남은 Jenkins가 있다면 그것은 누락이 아니라 결정이다.

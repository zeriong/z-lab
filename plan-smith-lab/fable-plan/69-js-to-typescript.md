# 대형 JS 코드베이스의 TypeScript 점진 전환
- 추론 프레임: min-diff
- 한 줄 요약: 전환을 "사람이 따르는 일정"에서 "CI가 강제하는 단조 감소 래칫"으로 옮긴다. 모든 단계는 롤백을 한 문장으로 써낼 수 있어야만 플랜에 입장하며, 타입 검사기가 드러낸 진짜 버그조차 전환 커밋에서는 고치지 않는다.

## 프레임이 요구하는 시작점 — 1문장 롤백은 선언이 아니라 입장 게이트다

min-diff의 실패 모드는 "겸사겸사"의 침투다. 그리고 TS 전환은 이 실패 모드의 최악의 서식지다: **타입 검사기 자체가 유혹 생성기**이기 때문이다. 파일을 .ts로 바꾸는 순간 죽은 코드, 이상한 패턴, 그리고 진짜 버그가 눈앞에 드러난다. 드러난 버그를 그 자리에서 고치는 것은 미덕처럼 보이지만, 그 순간 커밋은 "동작 보존"이라는 성질을 잃고, 회귀가 터졌을 때 전환 탓인지 수정 탓인지 분리할 수 없게 된다. 이 플랜의 첫 규칙은 그래서 유혹의 원천을 겨눈다.

**입장 게이트:** 어떤 단계든 롤백을 한 문장으로 쓰지 못하면 플랜에 들어올 수 없다. 게이트가 실제로 작동한 기록 — 초안에는 "핵심 도메인 모듈 일괄 전환(~200파일)" 단계가 있었다. 롤백 문장을 써 보니 "3주에 걸친 커밋 수십 개를 역순으로 revert하고 그 사이 merge된 타 변경과의 충돌을 수동 해소한다"가 되어 한 문장이 아니었다. 기각 후 분할: PR당 파일 10개 이하, 의존 그래프의 잎(leaf)부터.

**층위 이동:** 이 플랜은 일정표가 아니라 **불변식**이다. "몇 월까지 몇 %"가 아니라 "미전환 파일 수와 태그된 any 수는 절대 증가하지 않는다"를 CI가 기계로 강제한다. 전환 속도는 경영의 입력이고, 플랜이 보장하는 것은 방향의 비가역성이다. 래칫이 있으면 전환이 6개월 멈춰도 후퇴는 없다 — 롤백 가능한 전진만이 유일한 이동 방향이 된다.

## 문제 정의 / 목표

수십만 라인·수천 파일 규모의 운영 중인 JS 코드베이스를, 배포를 멈추지 않고, 회귀의 원인 추적 능력(bisect 가능성)을 잃지 않으면서 TS로 옮긴다. 목표는 "타입이 있는 코드"가 아니라 **"어느 시점에 중단·롤백해도 그 상태로 운영 가능한 전환 경로"**다.

## 의미 고정 — 숫자보다 먼저

- **"전환됨":** 파일이 (1) .ts/.tsx로 개명되고, (2) 최종 목표 strict 플래그를 전부 켠 tsconfig에서 컴파일되며, (3) 남은 any가 전부 명시적이고 `ts-debt` 태그가 붙어 있다.
- **"동작 보존":** 전환 커밋의 트랜스파일 산출물 diff가 타입 소거 이외의 변경 0. 이것은 정신 무장이 아니라 CI가 판정하는 기계적 성질이다.
- **any의 지위:** 실패가 아니라 **가격표 붙은 부채**. 전환 시점의 명시적 any+태그는 허용(정밀 타입 작성은 "개선"의 일이다), 암묵적 any는 금지(noImplicitAny는 첫날부터 on), 추측으로 쓴 거짓 타입은 any보다 나쁘므로 금지 — "확신 없으면 any+태그"가 규칙이다.
- **커밋 3계급:** ① 전환(동작 보존, 산출물 diff 0), ② 상환(태그 any를 정밀 타입으로 — 런타임 무변경), ③ 수정(tsc가 드러낸 버그의 수정 — 동작 변경이므로 테스트 선행, 별도 PR). 계급은 PR 라벨로 기계 식별 가능해야 한다.

## 명시적 가정 목록

- **가정 1 (전체가 걸린 가정): 타입 소거는 런타임 의미를 바꾸지 않는다.** 이를 위해 enum·namespace·parameter property 등 런타임 코드를 생성하는 TS 문법을 금지한다(`erasableSyntaxOnly` 강제). 틀리면(금지 문법이 필요해지면) "산출물 diff 0" 판정이 무너지고 동작 보존 검증이 테스트 의존으로 격하된다.
- 가정 2: 빌드 파이프라인이 .js/.ts 혼재를 허용한다(`allowJs`). → 틀리면 0단계 비용이 커지지만, 여전히 설정 커밋 revert로 복귀 가능해야 한다는 게이트는 유지.
- 가정 3: 테스트 스위트가 주요 경로를 덮는다. → 틀리면 산출물 diff가 유일한 동작 보존 기둥이 되고, 상환·수정 커밋의 안전망이 얇아진다. 완화: 커버리지가 낮은 모듈은 전환 순서를 뒤로 미룬다(전환이 테스트 작성을 강요하게 두지 않는다 — 그것도 겸사겸사다).
- 가정 4: 주요 외부 의존성에 @types가 존재한다. → 틀리면 스텁 .d.ts 부담 증가. 완화는 4단계의 rule of three.
- 가정 5: 팀이 커밋 3계급 규율을 수용한다. → 틀리면 bisect 가치가 소멸. 완화: 규율을 선의에 맡기지 않고 CI가 판정한다(전환 라벨 PR에서 산출물 diff가 0이 아니면 거부).

## 접근 및 단계

프레임의 재귀 적용: 전환 대상 코드뿐 아니라 **전환 도구 자체(tsconfig, 래칫 스크립트, lint 규칙)도 같은 입장 게이트를 통과**한다.

0. **공존 기반.** tsc를 `allowJs: true, noEmit` 검사기로 빌드 옆에 병렬 추가. strict 플래그는 처음부터 전부 on — "나중에 켜기"는 전 코드베이스 재방문을 예약하는 시간차 빅뱅이므로 기각(유혹 로그 T6). 검증: 빌드 산출물 바이트 동일, 테스트 전량 통과. 롤백: 설정 커밋 1개를 revert하면 순수 JS 빌드로 복귀한다.
1. **래칫 도입.** 미전환 파일 목록과 태그 any 수를 리포지토리에 체크인하고, 둘 중 하나라도 증가하면 CI 실패. 실패 메시지에는 고치는 명령을 그대로 출력(운영 부담 감사: 래칫은 스크립트 1개·숫자 2개를 넘지 않는다). 검증: 증가 PR을 일부러 올려 CI가 막는지 확인. 롤백: CI 잡 1개 비활성화로 끝, 코드 무영향.
2. **신규 코드 TS 강제.** D-day 이후 새 파일은 .ts만 허용(lint 규칙), 신규 파일의 any 예산은 0. 구멍을 파는 손부터 멈춘다. 검증: .js 신규 파일 PR이 거부되는지. 롤백: lint 규칙 1개 off.
3. **파일 단위 전환.** 의존 그래프의 잎부터, PR당 10파일 이하. 내용은 개명 + 컴파일에 필요한 최소 주석 + any 태그만. tsc가 드러낸 버그는 `@ts-expect-error`+티켓으로 **버그째 보존**하고 수정은 ③계급 PR로 분리한다. 검증: 산출물 diff 0(CI 게이트) + 테스트 통과. 롤백: 이 PR revert 하나로 해당 파일들이 .js로 돌아가며, import 경로·시그니처가 불변이므로 다른 파일은 무영향이다.
4. **경계 처리 — 단위 사이 탐색.** 위험은 파일 안이 아니라 전환/미전환의 import 경계에서 샌다(암묵 any 유입). 규칙: 미전환 모듈에 대한 스텁 .d.ts는 **세 번째 전환 파일이 같은 모듈을 import할 때** 비로소 작성한다(그 전엔 import 지점의 태그 any로 버틴다). 공용 타입 유틸도 동일 — 같은 타입 모양의 세 번째 중복 전까지 추출 금지(rule of three). 검증: 스텁 작성 시점이 래칫 로그로 추적 가능. 롤백: 스텁 파일 삭제 커밋 1개, 해당 import는 태그 any로 복귀.
5. **상환 트랙.** ②계급 PR로 태그 any를 정밀 타입으로 교체, 래칫 숫자가 내려간다. 검증: 런타임 산출물 무변경 + any 카운트 감소. 롤백: 상환 PR revert 하나로 any 상태로 복귀, 동작 무영향.

## 유혹 로그 — 기각 기록

- **T1 빅뱅 codemod:** "브랜치 하나에서 ts-migrate로 전부 전환." 기각 — 롤백이 "장수 브랜치 폐기 + 그동안의 본선 변경 재적용"이라 한 문장이 아니다. 부활 조건: 리포지토리 동결이 가능한 소규모(<300파일) 코드베이스.
- **T2 겸사겸사 리팩터링:** "파일 여는 김에 죽은 코드 제거·구조 개선." 기각 — 전환 커밋의 산출물 diff 0 성질을 파괴. 부활 조건: 없음. 언제든 별도 PR로는 환영.
- **T3 겸사겸사 버그 수정:** 가장 반직관적인 기각. tsc가 보여준 진짜 버그를 전환 커밋에서 고치면 "전환은 무해하다"는 bisect 전제가 죽는다. 버그는 보존·표식·발권하고 ③계급으로 고친다.
- **T4 완벽한 타입 먼저:** 전환 시점의 정밀 타입 작성. 기각 — 그것은 상환 트랙의 일이며, 전환 PR을 무한히 붙잡는다.
- **T5 공용 타입 라이브러리 선제 구축:** 기각 — rule of three 위반. 두 번째 중복까지는 중복이 정답이다.
- **T6 "strict는 나중에":** 기각 — 플래그 지연은 빅뱅의 시간차 변형(위 0단계).

## 대안과 기각 사유

- **JSDoc + checkJs로 개명 없이 타이핑:** 기각 — 타입을 두 번 쓰게 되고(주석→문법) 최종 상태에 도달하지 못한다. 부활 조건: 빌드 파이프라인을 만질 권한이 없는 환경(가정 2 붕괴 시 유력한 차선).
- **느슨하게 켜고 플래그를 점층 강화(loose-then-tighten):** 기각 — 플래그 하나를 켤 때마다 이미 전환된 전 파일이 다시 깨지는 코드베이스 전역 이벤트가 반복되며, 그 롤백은 한 문장이 아니다. 부활 조건: strict 상태 컴파일이 불가능할 만큼 동적 패턴이 지배적인 코드베이스임이 3단계 초기에 실측될 때.
- **빅뱅 일괄 전환:** 유혹 로그 T1 참조.

## 리스크 & 완화

- **R1 래칫 피로:** CI 실패가 잦으면 팀이 래칫을 우회·원망한다. → 래칫은 증가만 막고 동률은 통과시킨다. 실패 메시지가 해결 명령을 제시한다. 스프린트별 상환 쿼터 같은 강제 진도는 넣지 않는다 — 실행자의 인지 여유를 갉아먹는 규칙은 프레임보다 오래 못 산다.
- **R2 전환 정체:** 래칫은 후퇴를 막을 뿐 전진을 강제하지 않는다. min-diff는 속도 프레임이 아니므로 이를 숨기지 않는다. → 완화: 래칫 숫자의 주간 그래프를 자동 게시해 정체를 실측 가능하게 만들고, 속도 결정은 경영에 넘긴다.
- **R3 산출물 diff 판정 불능:** 번들러 변환이 개입해 diff가 시끄러워지는 경우. → 번들 전 단계(transpile-only) 비교 하니스를 별도로 두고, 그마저 불가하면 해당 모듈만 테스트+스모크 판정으로 격하하되 격하 사실을 PR에 명기.
- **R4 거짓 타입:** 추측으로 쓴 정밀 타입이 런타임과 다르면 any보다 해롭다(검사기가 거짓 안심을 준다). → "확신 없으면 any+태그" 규칙 + 상환 PR은 타입의 근거(호출부 실측·스펙 링크)를 요구.

## "완료"의 판정 기준

- **실측 문장 1:** `allowJs: false`로 전환한 빌드가 통과하고, 래칫의 미전환 파일 수가 0이다.
- **실측 문장 2:** 무작위 추출한 전환 커밋 20개의 트랜스파일 산출물 diff를 재검사했을 때 타입 소거 외 변경이 0건이다.
- **실측 문장 3:** 전환 기간 중 발생한 회귀의 원인 커밋 중 ①계급(전환) 커밋이 0건이다 — 1건이라도 나오면 3단계 게이트의 결함으로 간주하고 원인 분석을 완료 조건에 추가한다.
- 태그 any 잔량이 합의된 한도 이하이고, 잔여 목록 전부에 티켓이 연결되어 있다.
- 유혹 로그가 실제 기각 사례로 채워져 있다 — 비어 있는 유혹 로그는 규율의 증거가 아니라 기록 태만의 증거로 판정한다.

# 사내 데이터 플랫폼 셀프서브 쿼리 인터페이스
- 추론 프레임: api-first
- 한 줄 요약: 인터페이스를 화면이 아니라 "분석가가 이미 치는 질의 10줄 + 각 질의의 확정된 응답"이라는 실행 가능한 계약으로 정의한다. 애매한 것(PII, 조인 권한)일수록 뒤로 미루지 않고 예시의 응답 형식으로 지금 못 박는다.

## 프레임이 요구하는 시작점 — 분석가가 실제로 칠 질의 10줄

api-first의 실패 모드는 게으른 해피패스 예시 하나를 스케치하고 곧장 내부 설계로 도망가는 것이다. 이 문서의 방어: 아래 10줄이 유일한 스펙이며, 내부의 모든 결정은 "이 중 어느 줄을 참으로 만드는가"에 답해야 존재할 수 있다. 그리고 예시는 작성자의 취향이 아니라 분석가가 이미 가진 세계에서 와야 하므로 — 분석가들은 지금 리드온리 복제본에 SQL을 치고, 결과를 스프레드시트로 옮겨 산다 — 1단계는 이 10줄을 실측 질의로 교체·검증하는 절차다.

```
Q1  SELECT signup_week, count(*) FROM svc.users GROUP BY 1;
    → 수 초 내 표. 복제본에 치던 SQL이 방언 포함 그대로 동작한다.
Q2  SELECT email, last_login FROM svc.users LIMIT 20;
    → 에러가 아니다. email은 'k**@**.com'으로 마스킹된 값으로 반환,
      응답 메타에 masked_columns=[email] 명시.
Q3  SELECT count(DISTINCT email) FROM svc.users WHERE signup_week >= '2026-06-01';
    → 원본 기준 정확한 수치. 개별값 열람 권한과 집계 권한은 다르다 — 이 구분 자체가 계약.
Q4  SELECT u.plan, avg(o.amount) FROM svc.orders o JOIN svc.users u ON u.id = o.user_id GROUP BY 1;
    → 같은 도메인 안의 조인은 그냥 된다.
Q5  SELECT d.team, avg(s.base) FROM hr.salary s JOIN org.dept d ON s.dept_id = d.id GROUP BY 1;
    → 거절. 단, 막다른 골목이 아니라 항행 가능한 응답:
      {denied: "hr.salary", owner: "@hr-data", request_url: "...", sla: "1영업일"}
Q6  (스프레드시트 셀) =DATA("saved:weekly_kpi", "2026-07-06")
    → 시트 사용자의 호출 지점은 셀이다. 저장된 질의 결과를 시트가 직접 당겨온다.
Q7  SELECT * FROM metrics.weekly_active WHERE week = '2026-07-06';
    → 대시보드와 같은 숫자. 지표 정의는 조회 가능한 테이블이다 —
      "왜 대시보드랑 달라요?"의 구조적 봉쇄.
Q8  SELECT * FROM svc.events WHERE payload LIKE '%checkout%';
    → 즉시 실행되지 않는다: {estimated_scan: "2.3TB", limit: "100GB",
      options: [파티션 조건 추가 | 1% 샘플 실행 | 승인 요청]} — 기본값은 샘플 실행.
Q9  RERUN q_20260708_1432;
    → 어제 결과의 재현. 모든 질의는 ID를 갖고, 같은 ID는 같은 스냅샷을 읽는다.
Q10 SHOW TABLES LIKE '%order%';  /  DESCRIBE svc.orders;
    → 권한 없는 테이블도 존재·소유자·설명은 보인다(내용은 아님). 발견이 안 되면 신청도 없다.
```

Q2·Q3·Q5·Q8이 이 계약의 심장이다. PII와 조인 권한은 "보안팀과 추후 협의"가 아니라 응답의 형식으로 지금 확정된다: 마스킹은 에러가 아니라 값이고, 거절은 예외가 아니라 소유자·신청 경로·SLA를 실은 정규 응답이다. 단, 계약이 잠그는 것은 필드 구조이지 문구가 아니다 — 문구까지 잠그면 오탈자 수정이 파괴적 변경이 된다.

층위 이동 한 번: 이 10줄은 문서가 아니라 **구현 전에 red 상태로 먼저 존재하는 CI 테스트 스위트**다. 인터페이스 설계서가 아니라 실행 가능한 수락 조건 — 계약의 표현 형식 자체를 산문에서 테스트로 옮긴다.

## 문제 정의 / 목표

분석가가 데이터를 보려면 데이터팀에 슬랙 문의나 티켓을 넣는다. 숫자 앞에 의미부터: 이 플랜에서 "셀프서브"의 조작적 정의는 **질의 발상부터 결과 획득까지 데이터팀 인간의 개입이 0회**인 상태다. 목표는 이 정의를 위 10줄이 커버하는 질의 패턴에 대해 성립시키는 것.

## 명시적 가정 목록

- **가정 1: 분석가의 기존 세계는 SQL이다(시트는 소비 지점).** 틀리면 — 계약 언어가 저장질의+시트 수식 중심으로 바뀌고 Q1~Q5, Q7~Q10이 전부 재작성된다. 플랜 전체가 이 가정에 걸려 있으며, 1단계 수집이 판정한다.
- 가정 2: PII 정책이 컬럼 단위로 표현 가능하다. 틀리면(행 단위 정책, 조합 재식별) — Q2/Q3의 계약 재설계, 보안 서명 게이트 강화.
- 가정 3: 조회 대상 데이터는 이미 웨어하우스/복제본에 적재돼 있다(적재 파이프라인은 범위 밖). 틀리면 — 이 플랜은 시기상조, 적재 플랜이 선행돼야 한다.
- 가정 4: 쿼리 엔진 방언이 분석가의 기존 SQL과 호환되거나 얇은 번역이 가능하다. 틀리면 — Q1의 "그대로 동작" 약속이 붕괴, 마이그레이션 가이드가 계약 항목으로 추가돼야 한다.

## 접근 및 단계

1. **질의 수집(설계 금지 기간, 1주):** 최근 30일 쿼리 로그 + 데이터팀 슬랙 문의·티켓을 수집해 위 10줄을 실측 질의로 교체. 검증: 확정된 10줄이 실측 질의 패턴의 상위 80%를 커버함을 로그로 확인. 게이트: SQL 비중이 50% 미만이면 가정 1 기각 — 기본값은 계약 언어를 저장질의+시트 수식으로 전환.
2. **계약 서명:** Q2/Q3/Q5/Q8의 응답 구조에 보안팀·각 도메인 오너가 서명. 문구가 아니라 필드에 서명한다. 검증: 서명 기록과 반려 사유 목록.
3. **계약의 테스트화:** 10줄을 스테이징 CI 스위트로 변환, 구현 전 red 상태로 커밋. 검증: CI가 10건 전부 실패를 보고하는 것이 이 단계의 성공이다.
4. **최소 내부 구현:** 쿼리 게이트웨이(정책 검사 → 비용 견적 → 엔진 위임 → 응답 주석). 내부 설계 회의의 규칙: 10줄 중 어느 줄도 참으로 만들지 않는 결정은 보류함에 넣는다. 검증: CI green 개수의 단조 증가.
5. **파일럿:** 분석가 5명, 2주, 기존 쿼리 파일을 수정 없이 반입. 검증: 아래 "완료" 기준의 실측.
6. **확장과 계약 변경 절차:** 새 애매 케이스가 발견되면 11번째 질의로 스위트에 추가한다 — 논쟁은 문서가 아니라 테스트로 종결된다.

## 대안과 기각 사유

- **시맨틱 레이어/전용 DSL 선행 구축:** 기각. 분석가가 이미 가진 것을 승격하는 대신 작성자 취향의 새 언어를 학습시키는 것 — 프레임 요건의 정면 위반이고, 도입 비용을 사용자에게 전가한다. 부활 조건: 1단계 수집에서 "지표 정의 불일치" 문의가 권한 문의보다 많으면, Q7의 metrics.* 계층을 시맨틱 레이어의 시드로 확장한다.
- **상용 BI 툴 도입으로 갈음:** 기각. 조인 권한·PII 같은 애매 케이스가 툴 설정 화면 속으로 흩어져 계약이 소멸하고, 질의 발상은 여전히 티켓으로 남는다. 부활 조건: 파일럿에서 병목이 권한이 아니라 시각화 소비로 판명되면 소비 계층으로만 병행 도입.
- **자연어(LLM) 질의 인터페이스:** 기각. 같은 질문이 다른 SQL로 번역되는 순간 Q9의 재현 계약이 성립하지 않는다. 부활 조건: 계약 계층이 안정된 뒤, 계약 위의 입력 방식으로만.

## 리스크 & 완화

- **R1 집계를 통한 재식별(Q3의 뒷문):** count(distinct)는 안전해도 min/max·고특정성 group by는 위험. → 집계 허용 함수 화이트리스트 + 최소 그룹 크기 k 규칙을 11번째 질의 후보로 계약에 추가, 2단계 보안 서명 범위에 포함.
- **R2 계약 경직화:** 응답까지 테스트에 박으면 개선이 파괴적 변경이 된다. → 계약은 필드 구조만, 문구·정렬·부가 필드는 자유(하위 호환 추가 허용).
- **R3 비용 폭주:** Q8 게이트가 방어하되, 견적 불가 쿼리의 기본값은 "1% 샘플 실행" — 기본값 없는 게이트는 작동하지 않는 게이트다.
- **R4 승인 SLA는 사람의 약속(Q5):** 오너 부재 시 계약이 거짓말이 된다. → 3영업일 무응답 시 자동 에스컬레이션을 응답 필드(escalation)로 계약에 포함.
- **R5 스위트 유지보수 부담(운영부담 자기 감사):** 스키마 변경마다 CI가 깨지면 계약이 짐이 된다. → 계약 질의는 안정 스키마(metrics.*, 핵심 svc.*)만 참조하고, 실험 테이블은 계약 밖으로 명시.

## "완료"의 판정 기준

- **10줄 계약 스위트가 스테이징 CI에서 전부 green — 마스킹 값 형식(Q2), 거절 응답 필드(Q5), 비용 게이트 발동(Q8)까지 명시된 응답 그대로.** (실측 문장)
- 파일럿 분석가 5명이 기존 쿼리 파일을 수정 없이(또는 응답이 안내한 수정만으로) 실행해 성공률 80% 이상.
- 파일럿 2주간 데이터팀 인간 개입(슬랙 문의·티켓) 건수가 동일 인원의 직전 2주 대비 50% 이하 — "셀프서브" 조작적 정의의 실측.
- Q5 경로로 발생한 권한 신청의 90% 이상이 SLA(1영업일, 에스컬레이션 포함 3영업일) 내 처리.
- 포기 목록(자연어 입력, 전용 DSL, 실험 테이블의 계약 편입)이 부활 조건과 함께 문서화되어 있음.

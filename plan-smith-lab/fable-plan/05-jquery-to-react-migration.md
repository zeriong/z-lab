# 10년 된 jQuery 관리자 페이지의 점진적 React 전환
- 추론 프레임: 최소 diff 보수형
- 한 줄 요약: 모든 단계에 "롤백 방법"을 필수 필드로 강제한다. 롤백을 한 문장으로 못 쓰는 단계는 계획에 들어올 수 없다.

## 프레임이 요구하는 시작점 — 단계의 입장 자격

이 플랜의 모든 단계는 세 필드를 채워야 입장할 수 있다: **[배포 단위] [롤백 방법(1문장)] [이 지점의 빅뱅 유혹과 기각]**. 롤백 방법이 한 문장으로 안 써지면 그 단계는 너무 크다 — 쪼개서 다시 온다. 이 자격 심사가 계획 그 자체다.

또 하나의 헌법: **전환(porting)과 개선(refactoring)은 같은 배포에 섞이지 않는다.** 전환 커밋의 정의는 "동작이 이전과 동일함"이고, 동작이 달라지는 순간 그것은 전환이 아니라 기능 변경으로 재분류되어 별도 트랙을 탄다.

## 문제 정의 / 목표

매일 내부 운영에 쓰이는 10년치 jQuery 관리자 페이지를 React로. 목표는 "React가 된다"가 아니라 **전환 기간 내내 단 하루도 운영이 멈추거나 되돌릴 수 없는 상태가 없는 것**. 전환 완료일보다 전환 중 안전이 상위 목표다.

## 명시적 가정 목록 (틀렸을 경우의 영향 포함)

- **가정 1:** 페이지들이 서버 라우팅 기반의 개별 화면이고, 화면 간 공유 상태는 세션/서버에 있다. → 틀리면(프론트 전역 상태 공유) 화면 단위 격리 전략 자체가 붕괴 — 이 가정은 0단계에서 검증부터 한다.
- **가정 2:** 기존 페이지에 React를 부분 마운트할 DOM 앵커를 잡을 수 있다(빌드 파이프라인이 스크립트 1개 추가를 허용). → 틀리면 인프라 정비가 선행 과제로 승격.
- **가정 3:** 화면별 사용 빈도·중요도 데이터를 얻을 수 있다(접속 로그). → 틀리면 전환 순서를 감으로 정하게 됨 — 리스크 오름차순 원칙이 약화.

## 접근 및 단계 (자격 필드 포함)

**0. 안전망 확인 스파이크 (전환 아님)**
- 배포 단위: 없음(조사만). 롤백: 해당 없음.
- 내용: 가정 1·2 검증 + 화면 목록에 (사용 빈도 × 장애 시 피해) 점수로 전환 순서표 작성. 가장 중요한 산출물은 "절대 먼저 건드리면 안 되는 화면" 목록(정산·권한 관리 등).
- 유혹: "조사하는 김에 공통 유틸도 정리하자." → 기각. 조사 스파이크에 diff가 생기면 스파이크가 아니다.

**1. 파이프라인에 React 주입 (사용자 무영향 배포)**
- 배포 단위: 번들 1개 추가, 아무 화면도 안 바뀜. 롤백: 스크립트 태그 1줄 제거.
- 내용: React 런타임 + 마운트 헬퍼를 로드만 한다. 렌더링 0. 이 "무의미해 보이는 배포"가 이후 모든 단계의 배포 경로를 미리 검증한다.
- 유혹: "이왕이면 빌드 시스템도 현대화(webpack→vite)하자." → 기각. 빌드 교체는 전 화면 동시 영향 = 사실상 빅뱅. 전환이 다 끝난 뒤의 별도 프로젝트.

**2. 최저위험 화면의 위젯 1개를 React 섬으로**
- 배포 단위: 위젯 1개. 롤백: 마운트 호출 1줄을 끄면 기존 jQuery 코드가 그대로 동작(기존 코드는 **삭제하지 않고 비활성화만** — 안정화 기간 후 삭제).
- 유혹: "위젯만 하지 말고 이 화면 통째로 하자, 금방인데." → 기각. 첫 섬의 목적은 화면 전환이 아니라 '섬 패턴 자체의 검증'(스타일 충돌, 이벤트 간섭, 전역 오염). 검증 대상을 최소로.

**3. 화면 단위 전환 반복 (순서표의 저위험부터)**
- 배포 단위: 화면 1개 = PR 1개 = 배포 1개. 롤백: 화면별 플래그로 구버전 즉시 복귀(플래그는 URL 파라미터로도 강제 가능하게 — 운영자가 장애 시 스스로 탈출).
- 규칙: 전환 전 해당 화면의 현재 동작을 스크린샷·주요 시나리오로 기록(동작 보존의 비교 기준). 신규 화면은 구버전과 **같은 서버 API만** 사용 — API 변경 욕구는 별도 트랙.
- 유혹: 화면마다 반복될 것 — "이 못생긴 로직, 옮기면서 고치자." → 기각. 고치고 싶으면 전환 배포가 안정화된 뒤 별도 diff로. 버그가 나면 "전환 탓인지 개선 탓인지"를 즉시 알 수 있어야 한다.
- 공통 컴포넌트는 **세 번째 중복이 나올 때까지 추출 금지**. 두 화면짜리 추상화는 대개 틀린 추상화다.

**4. 고위험 화면(정산·권한)은 병행 운영으로**
- 배포 단위: 신구 화면 동시 존재, 기본은 구버전. 롤백: 기본값이 이미 구버전이므로 롤백 개념 자체가 없음(전진만 opt-in).
- 내용: 신버전을 운영자 일부에게만 opt-in으로 열고, 핵심 수치(정산 합계 등)를 신구 화면에서 교차 검증하는 기간을 명시적으로 둔다.
- 유혹: "여기까지 왔는데 그냥 바꾸자." → 기각. 피해 최대 화면일수록 전환은 opt-in → 검증 → 기본값 변경 → 구버전 제거의 4박자를 다 밟는다.

**5. 라우팅·셸 전환은 최후, 그리고 별도 판단**
- 모든 화면이 React가 된 후에야 SPA 셸/클라이언트 라우팅을 검토. 이것은 전 화면 공유 축이라 이 플랜의 자격 심사(1문장 롤백)를 통과하기 어렵다 — **통과 못 하면 영구히 안 한다.** MPA로 남는 것은 실패가 아니다.

## 대안과 기각 사유

- **빅뱅 재작성:** 기각. 롤백 단위 = 전체. 자격 심사 통과 불가. 게다가 10년치 화면에는 문서화 안 된 동작(=현장의 업무 절차)이 묻혀 있어, 재작성은 그 암묵 스펙을 전부 재발견하는 비용을 숨기고 있다.
- **iframe 격리:** 기각. 섬 방식 대비 인증·스타일·통신 경계 비용이 크고, "점진적으로 사라지는 구조"가 아니라 "이중 구조의 고착"이 되기 쉽다.
- **전환 없이 유지:** 진지하게 검토할 가치 있음(전환의 기대 이익이 비용을 넘는지). 기각 사유: 채용·라이브러리 생태계·보안 패치의 장기 비용. 단, 이 판단은 0단계 조사 후 재확인한다 — 조사 결과 "안 하는 게 답"이면 그것도 유효한 종료다.

## 리스크 & 완화

- **R1: 신구 공존 장기화(영원한 하이브리드).** → 순서표에 화면별 목표 분기(quarter)를 붙이고, 분기마다 "제거된 jQuery 라인 수"를 추적 — 전진의 리듬을 측정 가능하게.
- **R2: 전역 오염(구코드의 window 변수·이벤트 위임과 섬의 충돌).** → 2단계 첫 섬에서 의도적으로 이 충돌을 찾는 테스트를 한다(충돌 카탈로그 작성). 문제를 저위험 지대에서 미리 만난다.
- **R3: 플래그 부채.** → 화면 전환 안정화(예: 2주 무이슈) 시 플래그와 구코드 제거를 그 화면의 "완료 정의"에 포함 — 제거까지가 전환이다.

## "완료"의 판정 기준

- 전환 기간의 **모든 배포가 1문장 롤백 필드를 갖고 배포됐고**, 실제 롤백 발동 시 그 문장대로 동작했다.
- 전환 커밋과 개선 커밋이 **한 번도 섞이지 않았다**(리뷰 기록으로 증명 가능).
- 각 화면의 구코드·플래그가 안정화 후 제거되어, 종료 시점에 하이브리드 잔재가 0.
- 고위험 화면은 신구 교차 검증 기간을 거쳤고, 그 수치 대조 기록이 남아 있다.

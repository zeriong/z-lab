# 엑셀 파워유저의 SQL 전환 8주 학습
- 추론 프레임: misconception-first
- 한 줄 요약: 이 플랜은 커리큘럼이 아니라 멘탈모델의 회귀 테스트 스위트다. 엑셀 숙련이 SQL을 오역할 지점을 먼저 예측해 각각에 재현 시험을 붙이고, 8주를 그 시험들을 "예측→충돌→재구성"으로 통과시키는 교정 일정으로 짠다.

## 프레임이 요구하는 시작점 — 오개념 예측 목록

프레임의 실패 모드부터 심문한다. misconception-first는 오개념 목록을 만들어 놓고 결국 문법 난이도순으로 가르치는 순간 장식이 된다. 이를 봉쇄하기 위해 표현 형식 자체를 한 층 옮긴다 — **학습 계획을 "오개념별 재현 시험(reproduction test)의 스위트"로 정의하고, 주차 배열의 근거를 SQL 문법 난이도가 아니라 오개념 사이의 의존·간섭 관계로 제한한다.** 각 오개념은 버그처럼 다룬다: 재현 조건(어떤 과제가 오답 예측을 유발하는가)과 폐쇄 판정(무엇이 관측되면 고쳐진 것인가)이 없으면 목록에 올리지 않는다.

전제 검사(프레임 유효성): misconception-first는 간섭할 선지식이 있을 때만 작동한다. 대상은 VLOOKUP·피벗·배열수식을 감으로 다루는 파워유저다 — 간섭원이 충분히 깊으므로 프레임 유효. 얕으면 이 플랜 전체가 무효다(가정 1).

예측 목록. 뿌리는 하나 — **M0 "표는 격자다"**: 셀은 주소를 갖고, 행은 위치를 갖고, 데이터 전체가 눈에 보인다. 나머지는 파생이다.

- **M1 행 순서 신앙** (M0 직계): "데이터는 순서대로 있다." ORDER BY 없는 SELECT의 순서를 신뢰하고 "위에서 3번째 행" 같은 위치 지칭을 시도한다. 재현: 같은 쿼리 재실행의 순서 예측.
- **M2 VLOOKUP식 조인관**: VLOOKUP은 첫 매치 하나를 가져오고 행 수를 보존한다. JOIN은 매치 수만큼 곱하고(fan-out), INNER는 비매치를 소리 없이 버린다. 재현: 중복 키가 심어진 조인의 결과 행 수 예측 — 1,000행이 1,000행으로 나온다고 예측하면 충돌.
- **M3 절차 가공 습관** (M0 파생): "이전 행을 참조한다"(C2=C1+B2식 누계). 기본 SQL엔 '이전 행'이 없다. 재현: 누계 과제.
- **M4 빈 셀 = NULL 오역**: 엑셀의 빈 셀은 문맥에 따라 0이나 ""처럼 행동하지만 NULL은 3치 논리다. 재현: `WHERE col <> 'A'`의 결과 행 수 예측 — NULL 행이 조용히 빠진다.
- **M5 집계관**: SUMIF는 데이터 옆에 결과를 놓지만 GROUP BY는 행을 붕괴시킨다. 재현: "그룹에 없는 열은 왜 SELECT 못 하나" 과제.

단위 사이 탐색에서 나온 것 하나 — 오개념 목록 바깥, **"쿼리 작성"과 "결과 신뢰" 사이의 틈**: 엑셀 유저의 검증 도구는 '전체 격자를 눈으로 훑기'인데 SQL 결과창은 그 도구를 몰수한다. 잘못된 쿼리를 잡아낼 본능적 수단이 없는 채로 8주를 보내게 된다는 뜻이다. 그래서 대조 쿼리(조인 전후 COUNT 대조, 합계 불변 검사)를 1주차에 문법보다 먼저 가르친다. 이것은 오개념 교정이 아니라 **몰수당한 검산 본능의 재무장**이며, 이후 모든 충돌 과제의 판정 도구가 된다.

봉인 목록(의도적으로 아직 안 가르치는 것 — 해제 조건 포함):
- **윈도우 함수**: ORDER BY가 프레임 안에서 부활하는 문법이라, M1 교정 중에 노출되면 "거 봐, 순서 있잖아"로 회귀시킨다. 해제 조건: M1 폐쇄 후(6주차 예정).
- **정규화·데이터 모델링**: "한 장짜리 넓은 시트" 관성을 건드리는 큰 주제지만 조인의 의미(M2)가 서기 전에는 왜 쪼개는지 이해 불가. 해제 조건: M2 폐쇄 후 조인 과제 안에서 자연 도입.
- **절차형 SQL(스토어드 프로시저·커서)**: 절차 습관(M3)의 밀수 통로. 8주 내 해제 없음. 해제 조건: 동일 문제를 집합 연산으로 먼저 푼 이력이 있을 때만.
- **인덱스·성능 튜닝**: 정확성 모델이 서기 전의 성능 논의는 소음. 해제 조건: 8주 이후.

## 문제 정의 / 목표

엑셀 파워유저 1인이 8주(주 5~6시간) 안에 자기 실무 데이터를 SQL로 다루되, **엑셀 멘탈모델의 오역 없이** 집합·선언형 사고로 질의하게 만든다. 목표는 문법 커버리지가 아니라 오개념 폐쇄다 — 문법은 알지만 JOIN을 VLOOKUP처럼 쓰는 상태가 이 플랜이 정의하는 실패다.

## 명시적 가정 목록

- **가정 1 (전체가 걸림): 학습자의 엑셀 숙련이 실제로 깊다.** → 틀리면 간섭할 선지식이 없어 misconception-first가 무력하다(백지에는 예측할 충돌이 없다). 프레임을 버리고 표준 커리큘럼으로 교체해야 한다.
- 가정 2: 실행 환경(PostgreSQL 등 실 DB + 학습자 업무와 닮은 데이터셋)이 1주차 전에 준비된다. → 틀리면 충돌이 지면 위 상상이 되어 교정력이 사라진다. 충돌은 실행 결과가 예측을 배신할 때만 일어난다.
- 가정 3: 학습자가 "실행 전 예측을 적는" 의식을 수용한다. → 틀리면(귀찮아서 바로 실행) 충돌 없는 노출 학습으로 퇴행 — 플랜의 엔진이 꺼진다. 완화는 R2.
- 가정 4: 충돌 과제를 학습자가 미리 알지 못하게 준비할 수단(문제 은행, 동료 출제, 생성 도구)이 있다. → 틀리면 자기 출제·자기 예측이 되어 충돌이 오염된다 — 답을 아는 예측은 예측이 아니다.
- 가정 5: 주 5~6시간이 8주간 유지된다. → 틀리면 주차를 밀지 않고 게이트 규칙에 따라 프로젝트 주가 교정 주로 전환된다 — 일정보다 폐쇄가 우선.

## 접근 및 단계

숫자 전에 의미부터 — **"폐쇄"의 조작적 정의**: 해당 오개념을 자극하도록 설계된 *처음 보는 변형 과제*에서, 실행 전 예측(행 수·순서 유무·NULL 처리)을 적고 실행 결과와 일치. **최소 1주 간격으로 2회.** 1회 정답은 폐쇄가 아니라 후보다 — 오개념은 재발한다.

1. **1주 — 검산 재무장 + M1 충돌.** 대조 쿼리 습관(COUNT·합계 불변 검사)을 문법보다 먼저. 이어 M1 충돌: 순서 예측 과제 → 배신 → 재구성("순서는 데이터의 속성이 아니라 출력의 요청이다"를 학습자가 자기 문장으로 쓰게 한다). 검증: 변형 과제 예측 정답 → 폐쇄 후보 등록.
2. **2주 — M4 충돌.** NULL이 심어진 데이터로 `<>`, NOT IN, COUNT(col) 예측 과제. 검증: "이 결과에서 몇 행이 조용히 빠졌나"를 실행 전에 답하기.
3. **3~4주 — M2 충돌(플랜의 무게중심).** 중복 키·비매치 키를 공학적으로 심은 데이터로 INNER/LEFT의 행 수 예측 → fan-out과 무음 탈락을 각각 정면 충돌 → 재구성: "JOIN은 찾기가 아니라 곱하기다." 모든 조인 뒤 COUNT 대조를 규칙화. 4주 말: M1·M4 재검(2회차 폐쇄 판정).
4. **5주 — M5 충돌.** SUMIF 관성 대 행 붕괴. 피벗으로 하던 실무 집계 하나를 GROUP BY로 재현하고 수치 대조. 검증: 그룹 열 제약을 오류 메시지 없이 사전 예측.
5. **6주 — M3 충돌 + 윈도우 함수 봉인 해제**(M1 폐쇄 확인이 해제 조건). 누계 과제를 기본 SQL로 먼저 시도해 절차 습관을 충돌시킨 뒤 OVER() 도입. 여기서 **프레임 재귀 적용** — 교정 자체가 낳는 2차 오개념을 예측해 둔다: "순서는 없다"를 과잉 학습한 학습자는 OVER(ORDER BY ...)에서 혼란하거나 M1로 회귀한다. 전용 충돌 과제("순서는 존재하지 않지만 요청할 수 있고, 윈도우는 요청된 순서 안에서 계산한다")를 미리 배치. fan-out 공포로 조인을 회피하고 죄다 사전 집계하는 과잉 교정도 2차 목록에 올려 7주에 검사.
6. **7주 — 통합 프로젝트.** 자기 실무 엑셀 리포트 1개를 SQL로 재구현, 원본과 수치 대조. **게이트(기본값 내장):** 7주 시작 시점에 미폐쇄 오개념이 2개 이상이면 프로젝트를 취소하고 교정 주로 전환한다 — 기본값은 전환이고, 프로젝트 강행이 예외다.
7. **8주 — 회귀 검사.** 전체 오개념(2차 포함)을 새 변형 과제로 재검. 재발 항목은 폐쇄 취소 후 재충돌. 검증: 장부의 모든 항목에 판정 날짜 2개.

## 대안과 기각 사유

- **문법 난이도순 표준 커리큘럼(SELECT→WHERE→JOIN→…):** 기각. 프레임의 실패 모드 그 자체다. 엑셀 지식은 침묵 속에서 오역된다 — JOIN을 VLOOKUP처럼 써도 데이터가 우연히 착하면 결과가 그럴듯하고, 틀린 모델이 성공 경험으로 굳는다. 부활 조건: 가정 1이 틀렸을 때(간섭원이 없으면 이쪽이 낫다).
- **엑셀 내부 경유(Power Query·피벗으로 다리 놓기):** 기각. 충돌을 지연시켜 오개념을 보존한다 — 다리는 편하지만 강을 안 건너게 만든다. 부활 조건: 학습 시간이 주 3시간 이하로 붕괴해 충돌 설계를 감당 못 할 때, 보조 수단으로만.
- **"SQL은 엑셀 수식과 비슷하다"식 유추 우선 교수:** 기각. 유추는 간섭을 완화하는 게 아니라 강화한다. 이 플랜은 유사점이 아니라 충돌점을 자산으로 쓴다.

## 리스크 & 완화

- **R1 충돌 불발:** 과제 데이터에 우연히 중복 키·NULL이 없으면 오답 예측이 안 나온다. → 충돌은 우연이 아니라 공학이다 — 모든 충돌 과제 데이터에 해당 오개념의 기폭제(중복 키, NULL, 비매치 키)를 의무 삽입하고 과제 제작 체크리스트에 명시.
- **R2 예측 의식의 운영 부담(자기 감사):** 매 쿼리마다 예측을 적게 하면 학습자가 장부에 깔린다. → 예측 장부는 주 2개의 지정 충돌 과제에만 적용, 나머지는 자유 실습. 규칙이 학습을 이기기 시작하면 규칙을 줄인다.
- **R3 정서 비용:** 8주 내내 "네 직관이 틀렸다"를 반복하면 자신감이 무너진다. → 모든 충돌 세션은 재구성 성공으로 마감(충돌로 끝나는 세션 금지). 엑셀 숙련이 자산인 지점 — 도메인 감각, 데이터 냄새 맡기, 검산 본능 — 을 주차마다 명시적으로 사용하게 한다.
- **R4 재발:** 폐쇄는 영구가 아니다. → 1주 간격 2회 판정 + 8주차 전량 회귀 검사가 구조적 방어.

## "완료"의 판정 기준

- **실측 문장:** 처음 보는 스키마에서, 중복 키와 NULL이 심어진 조인 과제의 결과 행 수를 실행 전에 예측해 정확히 맞힌다 — 최소 1주 간격으로 2회.
- 자기 실무 엑셀 리포트 1개가 SQL로 재구현되어 원본과 수치 대조 일치.
- 오개념 장부에 M0~M5와 2차 오개념 각각의 폐쇄 판정(날짜 2개)이 기록되어 있고, 재발 항목은 재충돌 이력이 남아 있다.
- 봉인 목록의 미해제 항목을 8주간 건드리지 않았고, 해제된 항목에는 해제 조건 충족 기록이 있다.

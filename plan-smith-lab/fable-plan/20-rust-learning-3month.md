# 백엔드 개발자의 3개월 Rust 학습 계획
- 추론 프레임: 오개념 예측 우선
- 한 줄 요약: 오개념을 "깨지는 시점"별로 배치한다 — 즉시 깨야 하는 것, 중반에 깨야 하는 것, 그리고 3개월 안에는 일부러 안 깨는 것(async)까지가 커리큘럼이다.

## 프레임이 요구하는 시작점 — 오개념 지도 (기존 지식의 간섭 예측)

학습자 프로파일: GC 언어(Java/Python/Go류) 백엔드 5년차. 이 사람의 장애물은 Rust의 낯섦이 아니라 **이미 아는 것의 간섭**이다. 간섭을 예측하고, 각 오개념에 "교정 시점"과 "교정 방법"을 붙인다.

| # | 오개념 (기존 지식의 문장) | Rust의 현실 | 교정 시점 | 왜 그 시점인가 |
|---|---|---|---|---|
| M1 | "대입은 복사나 참조 공유다" | 대입은 기본이 **이동(move)** — 원본은 무효가 된다 | **1주차 즉시** | 모든 코드 줄에서 부딪힘 — 이게 안 깨지면 아무것도 못 씀 |
| M2 | "컴파일 에러는 내가 뭘 잘못 쓴 것" | borrow checker 에러는 문법 오류가 아니라 **설계 피드백** | 1~2주차 (M1과 동시) | 에러의 의미를 재해석 못 하면 정서적으로 이탈(가장 흔한 포기 지점) |
| M3 | "문자열은 하나의 타입이다" | `String`/`&str`의 이중 구조 (소유/빌림의 문자열 버전) | 2~3주차 | 실코드 작성 시작점에서 즉시 부딪히는 최초의 실용 벽 |
| M4 | "에러는 던지고 위에서 잡는다" | 에러는 **값**(`Result`) — 타입이 전파를 강제 | 4~6주차 | 함수를 조합하기 시작할 때 깨야 함 |
| M5 | "재사용은 상속으로" | trait 합성 — is-a 계층이 아니라 능력의 조합 | 6~8주차 | 설계 단계 진입 시점. 너무 일찍 깨려 하면 추상적으로만 남음 |
| M6 | "동시성 안전은 런타임에 조심하는 것" | `Send`/`Sync` — 데이터 레이스가 **컴파일 타임에 불법** | 9~11주차 | 소유권 직관(M1)이 몸에 붙은 뒤라야 이 모델이 이해됨 |
| M7 | "async는 다른 언어의 async와 같겠지" | Rust async는 런타임 선택·Pin·수명 상호작용의 별도 세계 | **3개월 내 교정 안 함 (의도적 봉인)** | 아래 참조 |

**M7 봉인의 근거:** 오개념 교정에도 순서 의존성이 있다 — async의 난해함은 소유권+수명 직관이 완성된 뒤에야 "어렵지만 일관된 것"으로 보인다. 그 전에 열면 M1~M2의 교정까지 오염시킨다(모든 게 이상해 보임). 3개월 커리큘럼의 성숙한 결정은 "무엇을 배울까"만이 아니라 **"무엇을 아직 안 배울까"**다. 동시성 학습(M6)은 스레드+채널·`Arc<Mutex>`로 충분히 가능하다.

## 문제 정의 / 목표

3개월 후 상태 목표: "borrow checker와 싸우지 않고 **협상하는** 사람" — 에러를 보고 자기 설계의 소유권 구조를 다시 그릴 수 있는 상태. 문법 완주가 아니라 M1~M6의 교정 완료가 목표다.

## 교정 방법론 — 오개념은 강의가 아니라 충돌로 깨진다

각 오개념에 **"예측 → 충돌 → 재구성" 3단 과제**를 설계한다:
1. **예측:** 기존 멘탈모델로 결과를 예측하게 한다 ("이 코드, 될 것 같은가?").
2. **충돌:** 컴파일러가 예측을 깨뜨린다 — 이때 에러 메시지를 **끝까지 읽는 것**이 과제의 일부 (rustc의 에러는 교정 도구로 설계돼 있다).
3. **재구성:** 왜 언어가 이것을 금지하는지의 이유(어떤 버그 클래스를 원천 차단하는지)를 자기 언어로 설명 — 설명 못 하면 교정이 아니라 암기다.

예: M1 과제 — "Vec을 함수에 넘기고 다시 쓰는" GC식 코드를 먼저 쓰게 함 → move 에러 충돌 → "이 금지가 use-after-free와 이중 해제를 컴파일 타임에 소멸시킨다"를 설명하게 함.

## 명시적 가정 목록 (틀렸을 경우의 영향 포함)

- **가정 1:** 학습자의 병목이 문법이 아니라 멘탈모델 간섭이다(경력자 전제). → 신입이면 이 플랜은 과잉 — 오개념 프레임은 "간섭할 기존 지식"이 있을 때만 유효.
- **가정 2:** 주 10시간 내외의 학습 시간. → 절반이면 M5·M6을 4개월차로 밀고 M7 봉인 유지 — 순서는 불변, 속도만 조정.
- **가정 3:** 목표가 백엔드 실무(API 서버, CLI 도구)다. → 임베디드/게임이면 오개념 지도의 후반(M6 이후)이 달라짐.

## 접근 및 단계 (교정 시점 = 커리큘럼)

- **1~3주 (M1·M2·M3 — 소유권 집중기):** 3단 과제 반복. 목표는 진도가 아니라 "move/borrow 에러를 보고 놀라지 않게 되는 것". 작은 CLI 도구 1개(파일 처리)로 M3까지 관통.
- **4~6주 (M4 — 에러의 값화):** 기존에 짜둔 코드를 `Result`+`?`로 재작성하는 리팩터 과제 — 새 코드보다 "예외 습관이 밴 코드의 번역"이 교정 효율이 높다.
- **6~8주 (M5 — 상속의 장례):** 자기가 과거에 짠 (다른 언어의) 클래스 계층 하나를 trait 합성으로 재설계 — 자기 코드의 번역이라 간섭이 가장 세게 드러나고, 그래서 가장 잘 깨진다.
- **9~11주 (M6 — 동시성):** 데이터 레이스를 의도적으로 시도 → 컴파일러의 거부를 체험 → `Arc<Mutex>`/채널로 재구성. "런타임에 조심"이 "타입으로 불가능"으로 바뀌는 체험이 이 달의 전부.
- **12주 (통합):** 강점 도메인 재구현 — 작은 REST API 서버 (단, async 프레임워크가 아니라 스레드 기반로 — M7 봉인 유지). 6개의 오개념이 실전 압력에서 재발하는지 최종 점검.

## 대안과 기각 사유

- **문법 순서 진도표(공식 문서 챕터 순):** 기각. 챕터 순서는 언어의 논리이지 학습자의 간섭 순서가 아니다 — 경력자의 병목과 무관한 배치.
- **"일단 프로젝트부터" (부딪히며 배우기 전면화):** 부분 기각. 충돌은 이 방법론의 핵심이지만, **설계된 충돌**(3단 과제)과 무작위 충돌은 다르다 — 무작위 충돌은 M2(정서적 이탈)를 관리하지 못하고, async 같은 봉인 구역에 조기 진입해 좌절을 증폭시킨다.
- **async 포함 풀 커리큘럼:** 기각. M7 봉인 근거 참조 — 3개월의 가장 흔한 실패 경로가 "8주차에 async 열었다가 전부 뒤엉킴"이라고 판단.

## 리스크 & 완화

- **R1: M2 실패(에러와의 싸움을 견디지 못함) — 최대 이탈 위험.** → 1주차에 "에러는 피드백"의 재해석을 명시 과제로 + 에러 메시지 읽기 자체를 훈련 항목으로. 좌절 관리가 커리큘럼의 일부다.
- **R2: 예측한 오개념과 실제 걸림돌의 불일치.** → 주간 "이번 주 나를 막은 것" 기록 → 지도와 대조 → 지도를 학습자 실측으로 갱신 (오개념 지도도 가설이다).
- **R3: M7 봉인에 대한 조바심(실무 코드가 다 async라서).** → 봉인은 금지가 아니라 순서라는 것, 그리고 4개월차 해제 계획(소유권 완성 후)을 처음부터 명시.

## "완료"의 판정 기준

- M1~M6 각각에 대해 3단 과제의 "재구성" 설명(왜 언어가 금지하는가)을 자기 언어로 통과.
- 12주차 프로젝트에서 borrow checker 에러를 **스스로** 해결(검색 없이 에러 메시지만으로 1차 시도 가능).
- 예외/상속/런타임-조심 습관의 재발이 코드 리뷰에서 검출되지 않음.
- M7 봉인이 유지됐고, 해제 조건(소유권 직관 완성)이 충족 상태로 4개월차에 인계됨.

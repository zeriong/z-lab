# 하루 1억 이벤트 로그 수집 파이프라인
- 추론 프레임: 봉투 뒷면 계산 우선
- 한 줄 요약: "1억"은 형용사가 아니라 숫자다. 나눠보면 초당 ~1,200 — 아키텍처 등급을 계산이 정한다. 민감도 표까지가 봉투 계산의 완성이다.

## 프레임이 요구하는 시작점 — 계산 (기술 명사 금지 구간)

이 섹션이 끝날 때까지 Kafka·Flink·빅데이터라는 단어를 금지한다. 숫자만.

**처리량:**
- 10⁸ events/day ÷ 86,400 s ≈ **1,157 eps 평균**.
- 피크 배율 가정 ×8 (일간 트래픽 요철) → **~9,000 eps 피크**.
- 스파이크(장애 복구 후 클라이언트 일괄 재전송 등) ×5 추가 → 순간 **~45,000 eps**, 지속 수 분.
- 해석: 평균과 피크는 튜닝된 단일 노드의 영역(현대 서버는 수만 eps 수집이 가능). 스파이크는 **흡수 장치(버퍼)**의 문제이지 상시 용량의 문제가 아니다 — 상시 용량을 스파이크에 맞추는 것이 전형적 과잉설계.

**저장:**
- 이벤트 평균 1KB(JSON) 가정 → **100GB/day 원본**, 36.5TB/year.
- 컬럼형 + 압축(로그는 반복 구조라 8~12×) → **~10GB/day, ~3.7TB/year**.
- 해석: 연 단위 누적이 수 TB — 전량 검색 인덱스에 넣으면 비용이 지배하고, 오브젝트 스토리지면 무시 가능. **이 문제의 본질은 수집 속도가 아니라 저장 계층화다.**

**비용 자릿수 (월):**
- 수집 컴퓨트: 1~2 노드 = $100~300.
- 핫 검색 인덱스(최근 7일 = ~70GB 압축 전 기준): $100~300.
- 콜드(오브젝트 스토리지, 연 3.7TB 누적): $10~100.
- 합계 자릿수: **$10²~10³/month.** 관리형 스트리밍 풀스택($10³~10⁴)과 자릿수가 갈린다.

**민감도 표 (봉투 계산의 필수 마무리 — 어느 가정이 아키텍처를 뒤집는가):**

| 가정 | 기준값 | 뒤집힘 지점 | 뒤집히면 |
|---|---|---|---|
| 이벤트 크기 | 1KB | ~10KB | 저장 10× → 계층화 앞당김, 수집 대역폭 재계산 |
| 피크 배율 | ×8 | ×50+ | 단일 노드 수집 탈락 → 분산 수집 |
| 보존 요구 | 1년 | 무기한+전기간 검색 | 콜드 쿼리 엔진이 핵심 컴포넌트로 승격 |
| 유실 허용 | 소량 허용 | 0 (감사 로그) | 버퍼 복제·영속화 필수 → 비용 한 자릿수 상승 |

민감도 1행(이벤트 크기)이 가장 취약하다 → **런칭 첫 주에 실측으로 검증하는 것이 계획의 일부.**

## 문제 정의 / 목표

하루 1억 이벤트를 수집·보존·조회한다. 목표는 위 숫자가 정당화하는 **최저 등급의 아키텍처** + 숫자가 뒤집힐 때를 위한 **수치화된 재설계 트리거**.

## 명시적 가정 목록 (틀렸을 경우의 영향 포함)

민감도 표의 4행이 곧 가정 목록이다. 추가로:
- **가정 5:** 조회 패턴은 "최근 데이터 검색·집계가 대부분, 과거는 드문 배치 분석". → 틀리면 핫 윈도 확대 → 비용 구조 재계산.
- **가정 6:** 이벤트는 append-only이고 수정·삭제 요구 없음. → 틀리면(GDPR 삭제 등) 불변 파일 전략에 삭제 메커니즘 추가 필요 — 설계 복잡도의 숨은 점프.

## 접근 및 단계 (숫자 → 컴포넌트 매핑)

1. **수집 게이트웨이 (평균 1,157 / 피크 9,000 eps):** 무상태 HTTP 수집기 2대(이중화는 가용성 목적이지 용량 목적이 아님을 명시). 수신 즉시 디스크 지속 큐에 append 후 ACK — 요청 경로에서 어떤 원격 호출도 하지 않는다.
2. **스파이크 흡수 (45,000 eps × 수 분):** 위 디스크 큐가 흡수 장치. 필요 용량 = 45k eps × 1KB × 600s ≈ **27GB** — 로컬 NVMe로 사소함. 계산이 "스파이크용 분산 스트리밍"을 기각하는 순간.
3. **배치 적재 (10GB/day 압축):** 큐 → 5분 마이크로배치 → Parquet 변환 → 오브젝트 스토리지(일자/시간 파티션). 다운스트림은 평균 속도로만 설계.
4. **조회 계층 (가정 5):** 최근 7일만 검색 인덱스에 적재(핫), 전 기간은 오브젝트 스토리지 위 스캔 쿼리(콜드·저빈도). 핫 윈도는 비용 다이얼 — 늘리고 줄이는 게 아키텍처 변경이 아니라 설정 변경이 되게.
5. **계측과 재설계 트리거(수치 명시):** 이벤트 크기 실측 p50/p99, eps 추이, 큐 적체. 트리거 예: "피크 30,000 eps 3일 연속 → 수집기 증설", "일 5억 도달 → 분산 스트리밍 재평가". 트리거를 지금 숫자로 박는 것이 미래의 나에 대한 봉투 계산의 인수인계다.

## 대안과 기각 사유

- **Kafka + Flink 풀스택 선도입:** 기각. 계산 결과(피크 9k eps, 스파이크 27GB 버퍼)가 이 등급을 요구하지 않는다. "1억"이라는 단어의 인상에 대한 설계이지 숫자에 대한 설계가 아니다. 재평가 트리거(일 5억)에 예약.
- **전량 검색 인덱스 보관:** 기각. 연 36TB 원본을 인덱스에 → 비용 자릿수 상승. 가정 5(조회는 최근 위주)가 유지되는 한 낭비.
- **수집기에서 직접 스토리지 동기 쓰기(큐 생략):** 기각. 스파이크 흡수 장치가 사라져 45k eps가 상시 용량 요구로 전환 — 큐 27GB가 그 요구를 지운다는 계산과 모순.
- **서버리스 수집(함수 단가 과금):** 기각. 월 30억 호출의 단가 × 호출 수 계산이 상시 노드 대비 자릿수 열세. (봉투 계산은 기술 선택뿐 아니라 과금 모델 선택에도 적용된다.)

## 리스크 & 완화

- **R1: 이벤트 크기 실측이 1KB를 크게 상회 (민감도 1행).** → 첫 주 실측을 마일스톤으로. 초과 시 스키마 다이어트(불필요 필드 제거)를 먼저, 아키텍처 승급은 그 다음.
- **R2: 유실 허용 가정의 붕괴(감사 용도 편입).** → 해당 이벤트 유형만 복제 큐로 분리 — 전체 파이프라인 승급이 아니라 유형별 계층화로 대응.
- **R3: 성장으로 트리거 도달.** → 트리거가 수치로 존재하므로 "언제 재설계하나" 논쟁이 발생하지 않음 — 그것이 트리거의 존재 이유.

## "완료"의 판정 기준

- 모든 컴포넌트 선택 옆에 근거 숫자가 병기돼 있다(숫자 없는 선택 0개).
- 민감도 표의 각 행에 실측 검증 일정 또는 재설계 트리거가 붙어 있다.
- 실측 첫 주 후 봉투 계산이 실제 수치로 갱신됐고, 자릿수 이탈이 없거나 이탈 시 재계산이 수행됐다.
- 월 비용이 계산된 자릿수($10²~10³) 안에 있다.

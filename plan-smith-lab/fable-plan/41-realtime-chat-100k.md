# 동시접속 10만 실시간 채팅 서비스 인프라
- 추론 프레임: envelope
- 한 줄 요약: 자릿수 계산을 해 보면 "동시접속 10만"은 이 시스템에서 가장 싼 숫자다. 비싼 것은 방 크기 분포의 꼬리와 부속 신호(수신 확인·프레즌스)이며, 설계는 이 두 축의 플립 지점에 재설계 트리거를 1:1로 못 박는다.

## 프레임이 요구하는 시작점 — 자릿수 계산 (이 구간 기술 명사 금지)

envelope의 실패 모드는 "계산해 놓고 결론은 따로 노는 것"이다. 이를 막기 위해 이 구간에서는 제품·기술 명사를 금지하고, 뒤의 모든 설계 결정이 여기의 숫자 중 어느 것에서 연역됐는지 표기한다.

**의미 고정부터.** "동시접속 10만" = 피크 시점에 열려 있는 연결 10^5개(일 사용자 수 아님 — 그건 이보다 5~10배 크다). 활성 발화자 = 지금 이 순간 대화 중인 접속자, 비율 10%, 발화 간격 20초. 메시지 = 텍스트, 외피 포함 평균 0.5KB.

**① 유입 메시지 율:** 10^4명 ÷ 20초 = **5×10^2건/초**. 자릿수: 10^2~10^3. 작다.

**② 팬아웃 율:** 배달 수 = 유입 × 방 크기. 평균 방 10명이면 5×10^3건/초 — 여전히 작다. 그러나 팬아웃은 평균이 아니라 꼬리가 지배한다: 인원 10^4의 방 하나에서 초당 10건 발화 = 그 방 혼자 **10^5건/초**, 전체 평균 팬아웃의 20배. 설계 변수는 총 접속자가 아니라 **최대 방 크기**다.

**③ 부속 신호 율 (본 메시지 사이의 틈):** 수신 확인을 개인별 이벤트로 만들면, 메시지 1건 → r명이 확인 이벤트 발생 → 각각 r명에게 배달 = 메시지당 **r² 배달**. r=10에서 5×10^2 × 100 = **5×10^4건/초 — 본 메시지 팬아웃의 10배**. 타이핑 표시는 발화당 2~5건 × r = 본 메시지의 2~5배. 프레즌스는 10^5명 × 일 전환 10회 ≈ 12건/초 × 관계망 10^2 = 1.2×10^3건/초. 합산하면 **본 메시지는 이 시스템에서 가장 작은 트래픽이다.**

**④ 연결 유지 — 같은 숫자의 두 얼굴:** 총량으로 읽으면 연결당 상태 50KB × 10^5 = **5GB**. 상자 한두 대 분량, 사소하다. 율로 읽으면 다르다: 네트워크 단절 후 30초 창에 전원 재접속 = **3.3×10^3 접속 협상/초**, 각 재접속이 "밀린 메시지 조회" 1회를 동반하면 읽기 부하가 평시 유입의 6배로 동시에 몰린다. 10만이라는 헤드라인 숫자는 총량으로는 사소하고 율로는 이 시스템 최대의 순간 부하다.

**⑤ 저장 총량:** 저장은 팬아웃이 아니라 유입에 비례한다. 5×10^2/초 × 86,400 ≈ 4×10^7건/일 × 0.5KB ≈ **20GB/일, 연 7~8TB**. 율(초당 쓰기 500건)로는 사소, 총량으로는 기술 문제가 아니라 보존 정책 문제.

**⑥ 대역폭:** 부속 포함 배달 5×10^4/초 × 0.3KB ≈ 15MB/초 ≈ 120Mbps. 사소. 단 미디어가 본문 경로에 실리면 평균 크기가 100배로 뛰어 자릿수가 붕괴한다 — 미디어는 본문 경로에서 배제해야 한다.

**단위 선택 총괄:** 연결(율이 위험/총량 사소), 유입(율 사소/총량은 정책 문제), 팬아웃·부속 신호(율이 위험). 위험한 축은 셋 다 율이고, 그중 둘은 문제 문장("10만 접속")에 아예 등장하지 않는다.

## 계산이 강제하는 결론 (기술 명사 해금)

1. **배달 2-레짐** (②에서): 방 크기 임계 이하는 연결별 개별 푸시, 초과 방은 공유 피드를 클라이언트가 구독하는 풀 방식으로 배달 의미론 자체를 전환. 방이 커지면 "메시지를 보낸다"가 "피드에 게시한다"로 층위가 바뀐다. 임계 초과 시 전환이 기본값인 게이트로 운영.
2. **부속 신호 강등** (③에서): 수신 확인은 개인별 이벤트가 아니라 방 단위 워터마크(구성원별 최고 수신 시퀀스)로 집계 — r²을 r로 강등. 타이핑·프레즌스는 저장 없음·재전송 없음의 손실 허용 채널로 본 메시지 경로에서 분리.
3. **재접속 폭풍은 용량이 아니라 프로토콜 계약으로** (④에서): 클라이언트 재개 토큰(마지막 수신 시퀀스 내장) + 서버 지시 지터. 폭풍을 상시 6배 용량으로 받아내는 안은 기각.
4. **저장은 유입 기준 설계 + 보존 기간을 제품 계약으로 명문화** (⑤에서). **미디어는 포인터만 본문 경로에** (⑥에서).

## 문제 정의 / 목표

피크 동시접속 10^5의 실시간 채팅 인프라. 목표는 "위 계산의 위험 축 3개(최대 방 팬아웃, 부속 신호, 재접속 폭풍)가 각각 독립적으로 폭주해도 본 메시지 배달이 무너지지 않는 구조"다.

## 명시적 가정 목록

- **가정 1: 방 크기 분포 — 평균 ~10명, 최대 10^4 안팎.** 틀리면: 전부 소형 방이면 2-레짐이 과설계(단, 비용은 게이트 하나), 초대형 방이 일상이면 브로드캐스트 레짐이 기본이 되어 개별 푸시 쪽이 예외로 강등. **플랜의 절반이 이 가정 위에 있다.**
- 가정 2: 활성 발화율 10%, 간격 20초. 틀려서 10배여도 유입은 5×10^3/초 — 자릿수 결론 유지. 둔감.
- 가정 3: 평균 메시지 0.5KB(텍스트 중심). 틀리면 대역폭·저장 자릿수 붕괴 — 그래서 미디어 포인터화를 가정이 아니라 설계 규칙으로 승격해 이 가정의 파괴력을 제거했다.
- 가정 4: 재접속 폭풍 창 30초. 창이 10초로 줄면 협상 10^4/초 — 지터 계약만으로 부족, 입장 대기열 필요(트리거 표 참조).

## 접근 및 단계

1. **계측 먼저:** 가정 1·2·4를 실측하는 부하 모델 계기판(방 크기 분포, 발화율, 부속 신호/본 메시지 비율, 재접속 패턴). → 검증: 4주 내 가정 표 각 행에 실측값 기입.
2. **연결 계층:** 접속 유지 전담, 연결당 상태 최소화. → 검증: 노드당 목표 연결 수에서 메모리 실측이 계산치(④)의 2배 이내.
3. **배달 계층 2-레짐 구축.** → 검증: 합성 부하로 방 크기 10 / 10^3 / 10^5 각각에서 배달 p99 측정, 레짐 전환 게이트 동작 확인.
4. **부속 신호 채널 분리 + 워터마크 집계.** → 검증: 부속 채널 전체 차단 시 본 메시지 배달 무영향(격리 실증).
5. **재접속 계약(재개 토큰 + 지터) 구현.** → 검증: 전 연결 강제 절단 후 재접속 완료 시간과 메시지 유실 측정.
6. **저장 + 보존 정책 명문화.** → 검증: 일 증가량 실측이 20GB/일의 자릿수 내.

## 민감도 표 (결론이 뒤집히는 지점)

| 가정 | 기준값 | 플립 지점 | 뒤집히면 |
|---|---|---|---|
| 최대 방 크기 | 10^4 | ~10^3 초과부터 개별 푸시 불성립 | 브로드캐스트 레짐 필요(이미 반영) |
| 활성 발화율 | 10% | 100%여도 결론 유지 | 없음 — 둔감 축 |
| 평균 메시지 크기 | 0.5KB | > ~5KB | 미디어 경로 분리가 선택이 아니라 생존 조건 |
| 수신 확인 의미론 | 워터마크 | 개인별 요구 + 방 크기 > ~30 | r² 폭발 — 방 크기 캡 또는 배치 집계 재협상 |
| 재접속 창 | 30초 | < 10초 (협상 10^4/초) | 입장 대기열(승인 제어) 추가 |
| 동시접속 | 10^5 | 10^6 | 연결 계층 수평 분할 — 본 플랜 범위 밖, 재설계 |

## 재설계 트리거 (숫자로 선약정 — 민감도 표와 1:1)

- 실측 최대 방 크기가 5×10^3을 1주 이상 상회 → 브로드캐스트 레짐을 기본값으로 승격.
- 부속 신호/본 메시지 배달 비율 > 50 → 부속 채널에 샘플링 도입.
- 재접속 율 p95 > 5×10^3/초 → 입장 대기열 추가 착수.
- 저장 일 증가량 > 100GB → 보존 정책 재협상(설계 변경이 아니라 계약 변경).
- 동시접속 3×10^5 초과 추세 2주 지속 → 연결 계층 샤딩 설계 착수.

## 대안과 기각 사유

- **단일 균일 배달 경로(모든 방을 개별 푸시):** 기각 — ②에 의해 최대 방 하나가 10^5/초를 만들며 전체 용량을 인질로 잡는다. 부활 조건: 실측 방 크기 p100 < 500.
- **수신자별 사서함(전 메시지를 팬아웃 저장):** 기각 — 저장이 유입이 아니라 팬아웃에 비례하게 되어 20GB/일 → 2TB/일로 자릿수 2단 상승(⑤ 위반). 부활 조건: 방 크기가 균일하게 ≤10이고 오프라인 동기화가 지배적 요구일 때.
- **재접속 폭풍 대비 상시 6배 용량:** 기각 — 연중 수 분을 위해 상시 6배 비용. 프로토콜 계약이 상위 호환. 부활 조건: 클라이언트 배포 통제가 불가능해 계약을 강제할 수 없는 경우.

## 리스크 & 완화

- **R1: 2-레짐 임계값의 임의성.** 계산이 주는 것은 "10^3 부근"뿐, 정확한 값은 아니다. → 임계는 상수가 아니라 설정값이며, 1단계 계측 후 4주 내 재설정 게이트를 의무화.
- **R2: 부속 신호 강등이 제품 요구(개인별 읽음 표시)와 충돌.** → 방 크기 조건부로 정밀화: 소형 방은 개인별, 임계 초과는 워터마크. 이는 요구 완화가 아니라 "누가 그 신호를 실제로 읽는가"의 정밀화임을 제품 측에 공개.
- **R3: 입력 임의성 — 계산 자체가 틀림.** envelope의 고질 약점. → 1단계가 계측인 이유이며, 민감도 표의 모든 플립 지점이 트리거·대시보드 알림과 1:1로 연결되어 틀린 입력이 조용히 살아남을 수 없게 한다.
- **커버리지 자기 감사:** 지리적 분산(멀티 리전), 남용·스팸 트래픽, 보안은 이 계산의 축에 없다. 범위 밖 선언이지 해결이 아님을 명기한다.

## "완료"의 판정 기준

- **합성 부하로 동시접속 10^5, 방 크기 분포(평균 10/최대 10^4), 부속 신호 포함 배달 5×10^4건/초를 1시간 유지했을 때, 본 메시지 배달 p99 ≤ 1초, 유실 0.** (실측 문장)
- **전 연결 강제 절단 후 5분 내 95% 재접속 완료, 절단 중 발생 메시지 유실 0** — 재개 토큰 계약의 실측.
- 부속 채널 차단 실험에서 본 메시지 무영향(4단계 격리 실증) 통과.
- 민감도 표의 전 행에 실측값 기입 완료, 재설계 트리거 5개가 각각 대시보드 알림과 1:1 매핑.

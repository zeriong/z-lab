# 오프라인 우선 모바일 메모 앱 — 동기화 엔진 설계
- 추론 프레임: backward
- 한 줄 요약: 인수조건 5개를 문서 최상단에 못 박으면 엔진 선택의 자유도는 거의 남지 않는다 — 동시 편집 보존과 충돌 UX 금지가 병합을 사람에서 자료형으로 옮기고, 장기 오프라인과 복원이 저장·GC 구조까지 역산으로 확정한다.

## 프레임이 요구하는 시작점 — 인수조건 5개를 먼저 확정한다

backward의 실패 모드는 "정해둔 결론을 향한 역산"이다. 이 도메인에는 이미 유행하는 답(CRDT)이 있으므로, 인수조건이 그 결론을 위해 역설계될 위험이 어느 주제보다 크다. 방어 장치를 조건 자체에 심는다: 각 조건은 (a) 제품의 사실에서 정당화되고, (b) 실측 가능하며, (c) **자신이 어떤 대안을 죽이는지**를 옆에 명기한다. 그리고 앵커 자격 시험 — 깨졌을 때 재작업이 최대인 조건만 앵커로 인정한다. 장식용 조건은 탈락이다.

| # | 인수조건 (실측 문장) | 이 조건이 죽이는 대안 |
|---|---|---|
| A1 수렴 | 어떤 순서·경로로 동기화되든 모든 기기의 최종 상태가 동일하다(상태 해시 일치) | 사람 개입 병합(비결정적), 동기화 순서에 의존하는 병합 |
| A2 동시 편집 | 두 기기가 같은 메모를 오프라인에서 각각 수정해도, 동기화 후 양쪽 편집이 문자 수준에서 모두 남는다 | 메모 단위 LWW(한쪽 편집 통째 소실), 파일 단위 동기화 |
| A3 장기 오프라인 | 90일 오프라인 후 복귀해도 동기화가 성공하고 그 90일치 편집이 보존된다 | 짧은 보존창의 연산 로그 컴팩션, 오프라인 길이에 비례해 변환이 복잡해지는 OT |
| A4 복원 | 기기 분실 → 새 기기 로그인 → 5분 내 전체 메모 복원 | 순수 P2P(권위 저장소 부재), 서버가 릴레이만 하는 구조 |
| A5 충돌 UX 없음 | 충돌 해결 다이얼로그·"충돌 사본"이 사용자에게 단 한 번도 노출되지 않는다 | Dropbox식 conflicted copy, git식 3-way 수동 병합 |

앵커 자격 재점검: 가장 많은 구조를 움직이는 것은 A5다. 충돌 다이얼로그를 넣고 출시한 뒤 제거하려면 병합 계층 전체를 다시 쓴다 — 재작업 최대 조건이므로 앵커 자격이 있다. A1은 당연해 보이지만 실측 문장("해시 일치")이 있어야 앵커다 — "언젠가 수렴"은 장식이다.

**앵커 밖 요구의 실격 선언**: 실시간 공동편집, E2E 암호화, 협업 권한 모델은 인수조건에 없다. 따라서 이들은 아키텍처를 움직일 권리가 없다 — 논의에 올라오면 이 문장을 가리킨다. 각각의 부활 조건은 대안 섹션에 기록한다.

## 역산 — 층위 이동: 동기화는 절차가 아니라 자료형의 성질이다

A2∧A5를 겹치면 요구가 정확해진다: 병합은 자동이고, 결정적이고, 메모 내부 단위에서 무손실이어야 한다. 이것을 절차(동기화 프로토콜)의 성질로 보장하려 하면 순서·재시도·부분 실패의 조합마다 별도의 증명이 필요해진다. 층위를 한 번 옮긴다: **병합 가능성을 문서 자료형의 수학적 성질(교환·결합·멱등 병합)로 만들고, 엔진은 전송과 저장만 담당한다.** 이 층위에서 A1은 테스트로 접근하는 목표가 아니라 타입이 보장하는 정리이고, 전송 계층의 버그는 "느린 동기화"로 나타나지 "기기마다 다른 상태"로는 나타나지 않는다. 실패의 등급 자체가 내려간다.

역산이 확정하는 자료 모델:
- 본문: 문자 단위 시퀀스 CRDT — A2의 "문자 수준"이 강제.
- 메타데이터(핀, 폴더, 표시용 수정시각): LWW 레지스터, 순서는 hybrid logical clock(벽시계 금지).
- 태그: add-wins OR-Set.
- 메모 목록: OR-Set of note ID + 삭제는 툼스톤.

**재귀 적용 — 생존자에게 같은 인수조건을 다시 들이댄다.** CRDT가 살아남았으니 CRDT에 A3·A4를 재심문한다: 90일 오프라인이면 툼스톤과 버전 벡터가 팽창한다. GC가 필요한데, GC는 A3와 정면 충돌한다 — 아직 돌아오지 않은 기기가 필요로 하는 이력을 지울 수 있다. 이 판단을 미래의 운영자에게 미루지 않고 게이트로 박는다: **툼스톤 GC 게이트 — 계정의 모든 등록 기기가 버전 벡터 V를 ack했을 때만 V 이전을 수거한다. 180일 무접속 기기는 등록 해제가 기본값이며, 복귀 시 병합이 아니라 복원(A4 경로)으로 진입한다.** 기본값이 있어야 게이트다 — 무접속 기기 한 대가 GC를 영원히 인질로 잡는 것을 이 기본값이 막고, A4가 이미 복원 경로를 요구했기 때문에 탈출구는 공짜로 얻는다.

**단위 사이 탐색.** 충돌은 자료형 안이 아니라 자료형 사이에서 터진다: A가 메모를 삭제하고 B가 같은 메모를 오프라인에서 편집하는 사건은 목록 OR-Set과 본문 CRDT의 경계에 있다. 런타임에 사용자에게 묻는 순간 A5 위반이므로, 판정을 지금 내린다: 편집이 삭제를 이긴다(add-wins 부활). 대신 모든 삭제는 30일 휴지통을 거쳐 A5(질문 없음)와 무손실(A2의 정신)을 동시에 만족시킨다.

## 문제 정의 / 목표

개인용(단일 사용자, 다기기) 메모 앱의 동기화 엔진. 목표는 기능 목록이 아니라 인수조건 5개의 구조적 충족 — 한 문장으로 "사용자가 동기화의 존재를 잊고 살 수 있는 것".

## 명시적 가정 목록

- **가정 1: 메모 본문은 병합 가능한 텍스트다. 이미지·첨부는 불변 blob으로 격리되어 병합 대상이 아니다.** → 틀리면(그림·구조화 문서가 본질이면) 본문 CRDT 선택 전체가 재검토 — 이 플랜의 척추가 부러진다.
- 가정 2: 단일 사용자 다기기이며 실시간 공동편집이 아니다. → 틀리면 자료형은 살아남으나 presence·권한 계층 신설, 전송 주기 재설계.
- 가정 3: 서버가 병합·컴팩션 가능한 형태로 데이터를 저장할 수 있다(E2E 암호화 요구 없음). → 틀리면 서버측 스냅샷 재작성이 불가능해져 A3의 GC 설계와 A4의 복원 경로가 키 관리 문제로 전이.
- 가정 4: 계정당 기기 10대 이하. → 틀리면 버전 벡터 크기와 GC ack 추적 비용을 재계산, 게이트 재설계.

## 접근 및 단계

1. 자료 모델 확정 + 수렴 증명이 있는 기성 CRDT 라이브러리 선정(자체 구현 금지) → 검증: 무작위 연산 인터리빙 property 테스트에서 상태 해시 일치(A1 실측의 축소판).
2. 로컬 우선 저장 — 모든 쓰기는 로컬 커밋이 진실, 동기화는 전적으로 비동기 → 검증: 비행기 모드에서 전 기능 동작, 강제 종료 후 무손실.
3. 델타 동기화 전송(버전 벡터 기반, 재시도 멱등) → 검증: 2기기 동시 편집 시나리오에서 문자 수준 보존(A2 실측).
4. 서버 스냅샷 + 델타 저장, 신규 기기 복원 경로 → 검증: 재설치 후 로그인 5분 내 전체 복원(A4 실측).
5. GC 게이트 구현 + 장기 오프라인 시뮬레이션 → 검증: 90일 시뮬 후 수렴, 문서 오버헤드가 본문 크기의 3배 이하(A3 실측).
6. 경계 규칙(삭제 vs 편집, 휴지통) + UX 감사 → 검증: 전 시나리오에서 충돌 UI 노출 0회(A5 실측).

단계 배열은 역산 순서가 아니라 검증 의존성 순서다 — 역산은 설계를 정하고, 실행은 의존성을 따른다. 각 단계는 자기 검증을 통과해야 다음으로 넘어간다.

## 대안과 기각 사유

- **OT + 중앙 서버(구글 독스 방식):** A3가 죽임 — 오프라인 길이에 비례해 변환 체인이 자라고, 90일치 연산의 변환 정확성은 증명 부담이 감당 불가. 부활 조건: 제품이 "대부분 온라인 + 실시간 공동편집"으로 피벗하면 최우선 재검토(그 세계에서는 OT가 상위 호환).
- **메모 단위 LWW(최신 저장 승리):** A2가 죽임. 구현이 열 배 단순하다는 유혹이 크지만, **앵커가 죽인 대안은 난이도로 부활하지 않는다** — 부활 조건은 인수조건 자체의 폐기뿐이며 그것은 제품 재정의다.
- **충돌 사본 생성(Dropbox식):** A5가 죽임. 데이터는 무손실이지만 손실 처리를 사용자에게 떠넘긴다 — 메모 앱에서 "회의록 (충돌 사본 2)"는 그 자체로 데이터 손상이다.
- **CRDT 자체 구현:** 인수조건이 아니라 리스크가 죽임 — 수렴 증명·실전 검증이 축적된 라이브러리 대비 얻는 것이 없다. 부활 조건: 기성 라이브러리가 가정 1의 자료형을 지원하지 못할 때.

## 리스크 & 완화

- R1 CRDT 메타데이터 팽창(A3의 그림자) → 오버헤드 예산을 상한 규칙으로: 본문의 3배 초과 시 서버측 스냅샷 재작성 자동 트리거. 사람이 지켜보다 결정하는 구조 금지.
- R2 라이브러리 종속 → 저장 포맷과 전송 포맷을 분리하고, 탈출 경로(전 문서 최종 상태 스냅샷 → 신규 포맷 재주입)를 1단계에서 문서로 확정.
- R3 앱 버전 간 스키마 이행 — 구버전 기기가 신버전 델타를 받는 사건은 또 하나의 "단위 사이" 지점 → 델타에 스키마 버전 명기, 구버전은 병합 보류 + 업데이트 유도. 로컬 편집은 계속 가능하므로 사용자 피해 없음(오프라인 우선이 여기서도 방어선).
- R4 시계 왜곡 → LWW 순서는 hybrid logical clock으로만, 벽시계는 표시용으로만. 기기 시계를 신뢰하는 코드는 리뷰에서 차단.

## "완료"의 판정 기준

- 기기 3대가 각자 무작위 오프라인 구간을 갖고 총 1만 회의 편집·삭제·태그 연산을 섞어 수행한 뒤 전원 복귀 — 세 기기의 상태 해시가 일치한다(A1).
- 기기 A를 90일 오프라인으로 시뮬레이션(그동안 B는 매일 편집)한 뒤 복귀 — 동기화가 성공하고 A·B 양쪽 편집이 문자 수준에서 모두 보존된다(A2∧A3).
- 앱 삭제 → 재설치 → 로그인: 5분 내 전체 메모 복원(A4).
- 위 모든 시나리오에서 사용자에게 노출된 충돌 해결 UI 0회, 생성된 "충돌 사본" 0건(A5).
- 인수조건에 없는 요구(실시간 공동편집·E2E 등)가 아키텍처 결정의 근거로 등장한 흔적이 설계 리뷰 기록에서 0건 — 실격 선언이 지켜졌는지의 실측.

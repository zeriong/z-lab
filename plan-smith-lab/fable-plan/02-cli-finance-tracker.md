# CLI 기반 개인 재무 추적 도구
- 추론 프레임: 삭제 우선 / 미니멀리즘
- 한 줄 요약: 기능 후보를 전부 깔아놓고 "없으면 안 되는가"를 심문한다. 최종 생존은 2개 — 도구가 아니라 파일이 제품이다.

## 프레임이 요구하는 시작점 — 근본 목적 한 문장, 그리고 심문

삭제를 하려면 삭제의 기준이 먼저 서야 한다. 근본 목적을 한 문장으로 고정한다:

> **"지출을 기록하는 마찰을 0에 가깝게 만들고, 월 1회 어디에 얼마 썼는지 본다."**

이 문장에 대한 **필요조건**이 아니면 전부 삭제한다. 입증 책임은 기능 쪽에 있다(기능이 자기 존재를 증명해야 함). 애매하면 삭제.

핵심 선결 결정: **저장소는 사람이 직접 편집 가능한 플레인 텍스트 1파일**로 한다. 이것은 기능이 아니라 삭제의 지렛대다 — 아래 심문에서 대부분의 기능은 "파일이 이미 그 일을 한다"로 죽는다.

## 삭제 심문 기록

| 후보 | 판결 | 근거 |
|---|---|---|
| 기록 추가 `add` | **생존** | 마찰 0의 기록이 목적의 전반부 그 자체. 셸 히스토리·자동완성의 이점 때문에 "에디터로 직접 추가"보다 빠름 — 유일하게 파일 직접 편집을 이기는 경로. |
| 월간 요약 `report` | **생존** | 목적의 후반부. 카테고리 합산은 눈으로 못 하므로 계산기가 필요. |
| 수정/삭제 `edit` | **삭제** | 파일이 플레인 텍스트다. 오타는 `$EDITOR`로 그 줄을 고치면 끝. 도구가 편집 명령을 갖는 순간 행 번호 지정·확인 프롬프트·실수 방지 등 자체 UI를 떠안는다. **파일의 편집 가능성이 기능을 대체하는 대표 사례.** |
| 검색 | **삭제** | `grep`이 있다. |
| 예산 초과 경고 | **삭제** | report를 읽으면 안다. 목적 문장에 "경고"는 없다. |
| 반복 거래 자동 입력 | **삭제** | 월 몇 건의 30초 작업을 없애려 스케줄러 상태를 들이는 건 역마진. |
| 다중 통화/계좌 | **삭제** | 목적 문장 밖. 카테고리 문자열이 계좌 구분까지 흡수 가능. |
| 그래프/차트 | **삭제** | 숫자 표가 월 1회 회고에 충분. 시각화 욕구는 파일을 스프레드시트에 부으면 해결. |
| CSV export | **삭제** | 저장 포맷이 이미 구분자 있는 텍스트 — export는 `cat`이다. |
| 동기화/백업 | **삭제** | 1파일 + git/클라우드 드라이브. 사용자가 이미 가진 도구의 몫. |
| `edit` 재심 — "그래도 최근 1건 취소는?" | **기각 유지** | 가장 흔한 정정(방금 잘못 입력)조차 `$EDITOR`로 마지막 줄 삭제가 충분히 싸다. "있으면 편함"은 생존 사유가 아니다. |

**최종 생존: `add`, `report`. 명령 2개.** (목표치 5개의 절반 이하. 프레임의 성공 기준을 여유 있게 통과.)

## 문제 정의 / 목표

기록 마찰 최소화 + 월간 회고. 제품의 실체는 명령 2개가 아니라 **잘 설계된 텍스트 한 줄 포맷**이다. 도구는 그 파일에 대한 얇은 편의 층일 뿐이다.

## 명시적 가정 목록 (틀렸을 경우의 영향 포함)

- **가정 1:** 사용자는 터미널과 `$EDITOR` 사용이 일상인 개발자형 사용자다. → 틀리면 `edit` 삭제 논거(에디터로 고치면 됨)가 무너지고, 심문 결과의 절반이 뒤집힌다. **이 플랜 전체가 이 가정 위에 서 있다.**
- **가정 2:** 단일 사용자·단일 통화. → 틀리면 파일 동시 쓰기·통화 처리 필요, 최소 구성 붕괴.
- **가정 3:** 연간 기록 수천 건 이하 → 전체 스캔 성능이 무시 가능. → 틀리면 report가 느려지나, 수십만 건 전까지는 실측상 문제없을 것.

## 접근 및 단계

1. **포맷 확정 (제품의 핵심):** `YYYY-MM-DD<TAB>금액<TAB>카테고리<TAB>메모`. 탭 구분 — 사람이 읽고, awk/cut이 바로 파싱하고, 스프레드시트에 붙는다. 파일 위치는 고정 경로 1곳(`~/.fin/ledger.tsv`), 설정 파일 없음.
2. **`add`:** `fin 12000 식비 점심` → 오늘 날짜로 1행 append. 인자 순서 고정, 플래그 없음. 목표: 타이핑 5초.
3. **`report`:** `fin report 2026-07` → 카테고리별 합계·건수·총계. 출력도 탭 구분(파이프 친화).
4. 끝. 3단계 이후의 모든 아이디어는 심문 테이블로 회부.

## 대안과 기각 사유

- **SQLite 저장:** 기각. 심문에서 삭제된 기능들이 전부 "파일의 편집·검색·이식 가능성"에 기대어 죽었다. DB로 바꾸는 순간 그 삭제 근거가 소멸해 edit/검색/export가 모두 부활한다 — 저장소 선택 하나가 기능 수를 지배한다.
- **기존 도구(ledger/hledger) 채택:** 진지한 대안. 기각 사유는 목적 문장 — 그 도구들은 복식부기라는 더 큰 목적을 위해 학습 비용을 요구한다. 우리의 목적 문장에는 복식부기가 없다. 단, 포맷을 hledger 호환으로 갈 수 있는지 검토했고, 호환 포맷의 문법 복잡도가 add의 5초 목표를 해쳐 기각.

## 리스크 & 완화

- **R1: 기능 요청 압력("edit만이라도...").** → README에 심문 테이블 자체를 게시. 부활 조건을 명시: "동일 요청이 실사용자에게서 반복되고, $EDITOR 우회가 실측으로 비싸다고 확인될 때만 재심."
- **R2: 포맷 파손(구분자 깨짐, 인코딩).** → add가 쓰기 전 검증. report는 파싱 불가 행을 계산에서 빼되 행 번호와 함께 경고(조용히 삼키지 않음).

## "완료"의 판정 기준

- 사용자 대면 명령이 **정확히 2개**.
- 삭제된 기능 각각에 **대체 경로가 명시**되어 있다(에디터/grep/cat/git). 대체 없는 삭제는 0건.
- `add` 입력이 5초 이내, `report`가 월 마감 회고에 필요한 질문("어디에 얼마")에 추가 도구 없이 답한다.
- 저장 파일을 도구 없이 열어 읽고·고치고·옮길 수 있다 — **도구가 사라져도 데이터는 완전하다**는 것이 최종 시험.

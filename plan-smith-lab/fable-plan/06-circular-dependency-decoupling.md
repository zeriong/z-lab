# 순환 의존이 얽힌 백엔드 모듈 분리
- 추론 프레임: Middle-out
- 한 줄 요약: "어떤 순환이 실제로 아픈가"를 모르는 것이 최대 불확실성이다. import 그래프와 git 변경 이력을 교차하는 spike를 중심에 놓고, 결과에 따라 세 갈래로 갈라지는 조건부 플랜을 짠다.

## 프레임이 요구하는 시작점 — 확실한 것과 불확실한 것의 분리

**확실한 것:** 순환이 존재하고, 개발이 느리다는 불만이 있다.
**불확실한 것(중심 미지수):** ① 어느 순환이 실제 고통의 원인인가, ② 그 둘 사이에 인과가 있기는 한가.

②를 강조해 둔다 — "순환이 많다"와 "개발이 느리다"가 관찰됐다고 순환이 원인인 것은 아니다. Middle-out의 시작점은 이 인과 자체를 검증 대상으로 놓는 것이다. 목표 아키텍처를 먼저 그리는 것(outside-in)도, 전체 그래프 분석부터 시작하는 것(bottom-up 전수조사)도 이 미지수를 건너뛴다.

## 문제 정의 / 목표

순환 의존으로 인한 실질 고통(빌드 결합, 변경 파급, 테스트 불가)을 최소 비용으로 제거. 목표는 "순환 0"이 아니라 **"아픈 순환 0"** — 안 아픈 순환은 부채가 아니라 그냥 구조다.

## 중심 Spike — 두 데이터의 교차

**설계 원리:** 정적 그래프(어디에 순환이 *있나*)만으로는 고통을 못 찾는다. 고통은 동적 사실이다. 그래서 두 소스를 교차한다.

1. **정적:** import 그래프에서 강연결요소(SCC) 추출 → 순환 클러스터 목록과 크기.
2. **동적 (이 spike의 차별점):** git 이력 6~12개월에서 **co-change 분석** — "PR 하나가 SCC 내부의 몇 개 모듈을 동시에 건드리는가". 순환의 고통은 대부분 "하나를 고치면 같이 고쳐야 하는" 변경 결합으로 나타나므로, co-change율이 고통의 직접 대리 지표다. 추가로 CI 로그에서 SCC 단위의 빌드/테스트 시간 점유율.
3. **판정 매트릭스:** 각 SCC에 (co-change율 × 변경 빈도 × CI 시간 점유)를 부여. 상위 1~2개에 대해서만 코드를 읽어 순환의 성격(아래 분기)을 판정.

**Spike 예산: 2주 고정.** 2주에 판정이 안 나오면 그 자체가 분기 신호다(관측 인프라 부재 → 선행 과제 재정의).

## 분기 구조 (spike 결과가 앞뒤를 결정)

- **분기 A — 상위 SCC의 순환이 "방향 하나만 억울한" 표면형** (한쪽 의존이 타입/유틸 몇 개 때문):
  → 인터페이스 역전 또는 공용 타입의 제3 모듈 추출로 국소 수술. 주 단위 비용. 이후 남은 SCC는 점수 재계산 후 반복.
- **분기 B — 상위 SCC가 "개념이 실제로 한 몸인" 구조형** (도메인 개념이 두 모듈에 걸쳐 있음):
  → 모듈 경계 자체가 틀렸다는 뜻. 분리가 아니라 **병합 후 재분할**(두 모듈을 합치고, 실제 변경 축을 따라 다시 자름)을 검토. co-change 데이터가 새 경계의 근거가 된다 — 같이 바뀌는 것끼리 묶는 경계가 올바른 경계다.
- **분기 C — 고통의 원인이 순환이 아님** (co-change는 SCC와 무관하게 분산, CI 병목은 픽스처/인프라):
  → **이 프로젝트를 중단하고 문제를 재정의한다.** "모듈 분리"는 답이 정해진 질문이었을 뿐이다. 조사 비용 2주로 몇 달의 헛수고를 산 것 — spike의 최고 수익 시나리오.

## 명시적 가정 목록 (틀렸을 경우의 영향 포함)

- **가정 1:** git 이력이 충분히 길고 커밋 단위가 의미 있다(스쿼시 문화 등으로 co-change 신호가 뭉개지지 않음). → 틀리면 동적 신호 상실, 정적+인터뷰로 후퇴(신뢰도 하락 명시).
- **가정 2:** 고통은 소수 SCC에 집중돼 있다(파레토). → 틀리면(균등 분산) 국소 수술 전략 무효 → 분기 B의 전면 재편 또는 분기 C 재검토.
- **가정 3:** 팀이 spike 2주의 "코드 안 고치는 기간"을 수용한다. → 틀리면 spike를 진행 중 업무에 편승(신규 PR마다 SCC 접촉 기록)하는 저강도 버전으로.

## 접근 및 단계

1. **[중심] Spike 2주:** SCC × co-change × CI 교차 → 분기 판정. 판정 기준을 spike 시작 전에 수치로 고정(예: "상위 SCC의 co-change율이 전체 평균의 3배 이상이면 인과 인정") — 데이터를 보고 기준을 정하면 원하는 결론을 고르게 된다.
2. **분기 실행:** A(국소 수술) / B(병합 후 재분할) / C(중단·재정의).
3. **수술 규약(분기 A·B 공통):** 한 번에 한 순환만. 각 수술 후 SCC 재계산 — 순환 제거가 새 순환을 만들 수 있다. 수술 전후로 "그 모듈 쌍의 독립 빌드·독립 테스트 가능"을 회귀 검증.
4. **재발 방지:** 아픈 순환을 끊은 뒤, CI에 "SCC 크기 상한" 체크를 추가해 후퇴를 기계적으로 차단. 단, 상한은 현재 상태 기준으로 설정(기존 순환의 즉시 전면 금지는 빅뱅 유혹의 변종).

## 대안과 기각 사유

- **전수 순환 제거 캠페인:** 기각. 가정 2가 참이면 대부분의 순환은 무해하며, 무해한 순환의 제거는 diff와 리스크만 산다. "순환 = 악"이라는 도그마가 이 캠페인의 유일한 근거인데, 그 도그마 검증이 바로 spike의 일이다.
- **목표 아키텍처 선행 설계(클린 아키텍처 다이어그램부터):** 기각. 중심 미지수(무엇이 아픈가)를 모른 채 그린 경계는 미학이지 공학이 아니다. 특히 분기 B가 참일 경우, 올바른 경계는 다이어그램이 아니라 co-change 데이터에서 나온다.
- **분기 없이 . A로 단정하고 착수:** 기각. A가 가장 흔하긴 하나, B·C일 확률이 합쳐서 무시 못 할 수준이고 그 경우 A 접근은 매몰 비용이 된다.

## 리스크 & 완화

- **R1: spike가 늘어져 분석 마비.** → 2주 하드 데드라인 + 사전 고정된 판정 수치. 기한 초과 = "판정 불가"라는 유효한 결과로 처리.
- **R2: 분기 B의 병합 단계에 대한 조직 저항("합치는 게 후퇴 아니냐").** → co-change 데이터를 근거로 제시 — 이미 한 몸으로 변경되고 있음을 보이면 병합은 현실 인정이지 후퇴가 아니다.
- **R3: 수술 중 신규 개발과의 충돌.** → 수술 단위가 "한 순환"이라 짧음. 화면·기능 동결 없이 진행 가능한 크기를 유지(05의 최소 diff 감각).

## "완료"의 판정 기준

- 분기 판정이 **사전 고정된 수치 기준**으로 내려졌고, 그 근거 데이터가 보존돼 있다.
- 상위 고통 SCC가 해소되어 해당 모듈들이 독립 빌드·테스트되고, **co-change율이 실측으로 하락**했다(구조가 아니라 고통이 준 것을 확인).
- 남은 순환들에 "점수 미달로 의도적 존치" 표식이 있고, CI 가드가 후퇴를 차단한다.
- 분기 C였다면: 중단 결정과 재정의된 문제가 문서화된 것 자체가 완료다.

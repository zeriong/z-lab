# 팀 공용 디자인 시스템 컴포넌트 라이브러리
- 추론 프레임: api-first
- 한 줄 요약: 라이브러리의 원본은 컴포넌트 코드가 아니라 소비자의 10줄이다. 그 10줄을 CI에서 실행되는 헌법으로 만들고, "파괴적 변경"의 판정까지 이 코퍼스에 기계적으로 위임한다.

## 프레임이 요구하는 시작점 — 헌법 10줄

api-first의 실패 모드는 둘이다: ① 대충 만든 happy-path 예시를 걸어두고 곧장 내부 설계로 도망가는 것, ② 예시를 작성자의 취향에서 지어내는 것(소비자의 기존 세계 반영 실패). 이 플랜은 둘을 규칙으로 봉쇄한다 — 헌법의 모든 prop은 제품 코드베이스에서 발굴한 실제 호출부에 출처를 가져야 하고(취향 금지), 예시에는 팀에서 실제로 분쟁이 나는 모호한 케이스를 일부러 박는다(happy-path 금지). 예시는 분쟁 예방 계약이지 쇼케이스가 아니다.

```tsx
// 헌법 v1 — 이 코드가 컴파일되고 렌더·상호작용 테스트를 통과하는 한, 라이브러리는 깨지지 않은 것이다
<Dialog open={open} onOpenChange={setOpen}>
  <Dialog.Title>구독 해지</Dialog.Title>
  <Dialog.Body>
    <TextField label="사유" error={errors.reason} />
  </Dialog.Body>
  <Dialog.Footer>
    <Button variant="ghost" onClick={close}>취소</Button>
    <Button variant="danger" loading={isPending}
            className="w-full md:w-auto" onClick={submit}>해지</Button>
  </Dialog.Footer>
</Dialog>
```

각 줄이 판결하는 분쟁:

- **합성:** Dialog는 prop 덩어리가 아니라 부분 컴포넌트의 합성이다. 그리고 두 버튼 사이 간격은 누구 소유인가 — Footer다. **모든 컴포넌트는 외부 마진을 갖지 않는다.** 디자인 시스템은 컴포넌트 안이 아니라 컴포넌트 *사이*에서 죽는다(간격, 겹침 순서, 포커스 이동). "사이"의 소유권을 컨테이너에 명시 배정하는 것이 이 줄의 계약이다.
- **variant와 상태의 직교:** `variant="danger"`와 `loading`이 동시에 걸린다. variant는 의미(위계·톤), loading은 상태다. 상태를 variant로 찍는 설계(`variant="dangerLoading"`)는 조합 폭발로 가므로 이 줄이 그것을 헌법 위반으로 만든다. loading 중 클릭은 무시된다 — 문장이 아니라 상호작용 테스트로.
- **스타일 오버라이드의 에지:** `className="w-full md:w-auto"` — 소비자의 유틸리티 클래스 세계를 그대로 계약으로 승격. 충돌 시 승자 규칙: **레이아웃 속성(폭·정렬·마진)은 소비자가 항상 이기고, 색·타이포·상태 스타일은 라이브러리가 이긴다**(위반은 lint 경고, 차단은 않음). "className 받아요"라는 애매한 문장 대신 이기는 쪽을 예시에 못 박는다.
- **기존 세계의 승격:** `error={errors.reason}`은 팀 폼 훅이 이미 반환하는 `string | undefined` 그 모양이다. `invalid: boolean` + `errorMessage` 이중 prop은 작성자 취향이므로 금지 — 소비자가 이미 손에 쥔 형태가 계약이 된다. `error={undefined}`일 때 레이아웃 시프트가 없어야 한다는 것도 이 줄의 계약(렌더 테스트로 실측).

## 층위 이동 — 버전의 단위를 코드에서 코퍼스로

헌법은 문서가 아니라 리포지토리의 테스트 코퍼스다. 그러면 semver의 정의가 바뀐다: **파괴적 변경 = 코퍼스의 어떤 예시가 컴파일 또는 테스트에 실패하는 변경. 코퍼스 전부 통과 = 마이너 이하.** 릴리스마다 벌어지던 "이거 breaking인가요?" 논쟁이 CI의 기계 판정으로 대체된다. 역명제도 선언한다: **코퍼스 밖의 사용법이 깨지는 것은 파괴적 변경이 아니다**(Hyrum의 법칙 방어). 대신 제품 코드를 분기마다 스캔해 코퍼스 밖 패턴을 승격 후보로 접수한다 — 계약이 넓어지는 유일한 경로는 승격이다. 프레임의 재귀 적용: 새 컴포넌트·새 prop이 들어올 때마다 같은 절차(실제 호출부 발굴 → 에지 박기 → 코퍼스 편입)를 다시 돈다.

## 문제 정의 / 목표

3개 제품 팀이 같은 UI를 각자 포크해 만들고 있다. 목표는 "예쁜 컴포넌트 모음"이 아니라 **분쟁이 코드로 판결되는 공용 계약** — 신규 화면이 로컬 포크 없이 조립되고, 릴리스 판정에 사람의 직관이 개입하지 않는 상태.

## 명시적 가정 목록

- **가정 1: 제품 코드베이스에 이미 반복되는 UI 호출 패턴이 존재하고 접근 가능하다.** → 틀리면 헌법을 발굴이 아니라 발명해야 하고, 그 순간 프레임은 실패 모드 ②(작성자 취향)로 퇴화한다. 플랜 전체가 이 가정에 걸려 있다.
- 가정 2: 소비자 전원이 React + TypeScript 단일 스택. → 틀리면(타 프레임워크 소비자 존재) 헌법을 JSX로 쓸 수 없어, 시작점을 프레임워크 중립 계층(웹컴포넌트 또는 토큰 계약)으로 재설계해야 한다.
- 가정 3: 유틸리티 클래스(Tailwind류)가 소비 코드에 공존한다. → 틀리면 오버라이드 승자 규칙을 CSS 우선순위 설계로 대체(계약의 취지는 유지, 기제만 교체).
- 가정 4: CI에서 타입체크·렌더·상호작용 테스트를 돌릴 인프라가 있다. → 틀리면 "실행되는 명세"가 읽히지 않는 문서로 퇴화 — 프레임의 존재 이유 상실. 이 경우 인프라 구축이 0단계로 승격된다.
- 가정 5: 유지보수 인력은 1~2인 겸업. → 틀리면(전담팀 존재) 아래의 코퍼스 상한·분기 스캔 같은 운영부담 완화 장치를 느슨하게 재조정해도 된다.

## 접근 및 단계

1. **발굴(1주):** 3개 제품 리포를 grep해 버튼·다이얼로그·입력 필드의 실제 호출부를 수집하고 빈도·형태를 표로 만든다. → 검증: 헌법의 모든 prop 옆에 출처 파일 경로가 붙어 있는가. 출처 없는 prop은 반려.
2. **헌법 비준:** 위 10줄 + 컴포넌트별 예시(상한: 컴포넌트당 5개, 핵심 1 + 에지 위주)를 각 팀 대표와 확정. 분쟁 케이스(오버라이드 승자, 마진 소유, loading 중 클릭) 포함 의무. → 검증: 리뷰에서 "이 줄은 어떤 분쟁을 판결하나"에 전부 답할 수 있는가.
3. **코퍼스 실행화:** 예시를 컴파일(타입) + 렌더 스냅샷 + 상호작용 테스트(loading 클릭 무시, Dialog 키보드 포커스 트랩)로 CI에 편입. → 검증: className 병합 순서를 일부러 뒤집는 mutation PR이 CI에서 빨간불.
4. **내부 구현:** 헌법을 참으로 만드는 방향으로만. headless 프리미티브 채택 여부는 내부 자유 — 계약 표면만 우리 것이면 된다. → 검증: 코퍼스 green + 헌법에 없는 public prop 0개.
5. **운영 규칙 가동:** prop 추가 게이트 — 신청자는 자기 제품 코드의 실제 호출부를 첨부해야 하며, **기본값은 반려다**(기본값 있는 게이트만이 작동하는 게이트다). 승격되면 그 호출부가 코퍼스에 들어가고, 상한 초과 시 기존 예시 1개를 은퇴시킨다(사유 기록). → 검증: 첫 분기 게이트 로그(신청/승격/반려 건수)의 존재.

## 대안과 기각 사유

- **Storybook 문서 주도(쇼케이스 먼저):** 기각. 쇼케이스는 구조적으로 happy-path를 향한다 — 분쟁 예방이라는 예시의 계약 기능이 사라지고 프레임 실패 모드 ①로 직행. 부활 조건: 코퍼스 안정 후 온보딩·탐색용 뷰어로는 부활 가능(단, 계약의 원본은 여전히 코퍼스).
- **headless 라이브러리 직접 노출 + 스타일 가이드:** 기각. 소비자의 10줄이 합성 보일러플레이트로 30줄이 된다 — 헌법이 요구하는 밀도를 못 맞춘다. 내부 구현 채택은 허용(4단계). 부활 조건: 유지보수가 1인 미만으로 줄면 자체 표면 유지비가 계약 가치를 초과 — 직접 노출로 후퇴하고 헌법은 "권장 합성 레시피"로 강등한다.
- **불리언 prop 증식(isDanger, isLarge, …):** 기각. 헌법의 variant+loading 직교 줄이 이미 죽였다. 부활 조건 없음 — 취향 문제가 아니라 조합 폭발의 산수다.

## 리스크 & 완화

- **R1: 코퍼스 비대화 — CI가 느려지고 아무도 예시를 안 읽음.** → 컴포넌트당 5개 상한 + 승격 시 은퇴 규칙. 운영부담을 선의가 아니라 상한으로 통제한다.
- **R2: className 탈출구 남용으로 시스템 무력화.** → 승자 규칙이 색·타이포를 라이브러리 승리로 이미 계약. lint는 경고만 — 차단하면 소비자는 포크로 도망간다. 분기 스캔에서 같은 오버라이드가 3회 이상 반복되면 남용이 아니라 **누락된 variant의 신호**로 재분류하고 승격 후보로 접수.
- **R3: 코퍼스가 못 잡는 파손(계약 밖 사용법).** → 층위 이동의 역명제로 "파괴 아님"을 이미 선언. 잔여 리스크는 "선언은 했지만 소비자는 화난다" — 릴리스 노트에 코퍼스 diff를 자동 첨부해 예측 가능성으로 상쇄.
- **R4: 채택 실패 — 팀들이 포크를 계속 만듦.** → 포크를 위반이 아니라 접수 경로로 취급: 발견된 포크는 코퍼스 승격 후보로 심사한다. 인센티브를 감시가 아니라 승격에 건다.

## "완료"의 판정 기준

- **헌법 10줄이 CI에서 컴파일·렌더·상호작용 테스트로 실행되고 있으며, className 병합 순서를 뒤집는 1줄짜리 사보타주 PR이 CI에서 실패한다**(실행되는 명세의 실측).
- 릴리스 2회 연속, major/minor 판정이 코퍼스 통과/실패만으로 결정되고 판정 회의 0회.
- 신규 화면 3개가 로컬 포크 0개로 출시(분기 발굴 스캔으로 실측).
- prop 게이트 로그가 존재하고, 반려 건 중 최소 1건의 사유가 "실제 호출부 미첨부"로 기록되어 있다(게이트가 장식이 아니라는 증거).

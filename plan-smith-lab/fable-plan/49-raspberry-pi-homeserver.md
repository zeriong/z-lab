# 라즈베리파이 1대로 돌리는 홈서버 스택
- 추론 프레임: constraint-first
- 한 줄 요약: 제약 5개의 모순 판정이 두 가지 전제 파괴를 강제한다 — 상태를 기계에서 분리해 SD를 소모품으로 만들고, 서비스를 기능이 아니라 "죽었을 때 누가 아는가"로 심사한다. 구멍(클라우드 보조)은 몰래 내지 않고 가정으로 선언한다.

## 프레임이 요구하는 시작점 — 벽 5개, 그리고 모순 판정

- **C1 (RAM):** Pi 4·4GB 1대. 스왑으로 연명하는 스택 금지(스왑은 C2와 정면 충돌).
- **C2 (SD 수명):** 부팅 매체는 쓰기가 수명을 깎는 소모품. 상시 쓰기(로그·DB·컨테이너 레이어)를 SD에 두는 구조 금지.
- **C3 (전기):** 상시 가동 총 10W 이하. 이 벽의 실제 역할은 파이 자체가 아니라 "파이에 붙이고 싶어지는 것들"(HDD 어레이, 팬, 중고 데스크톱으로의 상향)을 죽이는 것.
- **C4 (소음):** 팬 0개. 무팬 = 지속 부하의 발열 상한 = 사실상 CPU 예산.
- **C5 (가족 눈치):** 거실의 기계는 존재가 느껴지는 순간 철거 대상이다. 조작적 정의를 먼저 고정한다: **"서버가 3일 죽어 있어도 운영자 외 가족 누구도 알아차리지 못한다."** 이 정의가 이 플랜에서 가장 많은 것을 죽인다.

**모순 판정 1 — C2 vs "서버는 상시 쓴다":** 서버의 본질은 상태를 쌓는 것이고, SD는 쓰기로 죽는 매체다. "SD 위의 서버"는 구조적 모순. 요구(서버)를 깎지 않고 전제를 깬다 — **기계와 상태를 분리한다.** SD는 저장소가 아니라 소모품 부팅 매체(브레이크 패드), 상태는 USB SSD, 구성은 git의 선언 1벌, 백업은 클라우드. SD의 죽음은 "장애"에서 "소모품 교체"로 재분류된다.

**모순 판정 2 — C5 vs 이중화 없는 1대 vs "가족에게 유용한 서비스":** 홈서버 정석 1번(Pi-hole을 가족 DNS로)은 가족의 인터넷 경로에 무이중화 단일 장애점을 꽂는 행위다. 최악의 순간은 서비스 가동 중이 아니라 그 사이 — 정전 복구 때 공유기와 파이가 동시에 켜지는 틈 — 에 온다. C5의 조작적 정의와 양립 불가 → **가족의 경로 위에 있는 서비스는 전부 입주 금지.** 서비스 심사 축이 "무엇을 해주는가"에서 **"죽으면 누가 아는가"**로 바뀐다.

**모순 판정 3 — C1+C4 vs 표준 홈랩 스택:** 실시간 트랜스코딩·상시 대시보드·컨테이너 수십 개는 모순이 아니라 그냥 벽 밖이다(전부를 모순이라 부르면 판정이 죽는다). 단 미디어 요구는 완화가 아니라 정밀화로 살린다: 요구는 "시청"이지 "임의 포맷의 실시간 변환"이 아니다 → 사전 인코딩 + direct play.

판정 결과: 교집합은 공집합이 아니다. 다만 살아남는 것은 "홈서버"라는 단어가 연상시키는 스택이 아니라, 위 두 전제 파괴를 통과한 스택뿐이다.

## 문제 정의 / 목표
파이 1대로, 가족 공용 공간에서, 겸업 1인 운영자가 잊고 살아도 유지되는 개인 서비스 묶음. 목표는 서비스 개수가 아니라 **"철거당하지 않는 구조"**(C5)와 **"기계가 죽어도 상태가 죽지 않는 구조"**(C2)의 동시 충족.

## 명시적 가정 목록
- **가정 1 (선언된 구멍):** 클라우드 보조 2개를 쓴다 — 암호화 백업용 오브젝트 스토리지(월 소액), 원격 접근용 Tailscale 중계. "1대로 돌린다"의 순수성에 난 구멍이며, 몰래 내는 대신 여기서 선언한다. 틀리면(순수 셀프호스팅이 요구라면) 백업이 두 번째 기계를 요구해 "1대"라는 전제 자체가 붕괴. **이 플랜 전체가 이 가정 위에 서 있다.**
- 가정 2: 시청 기기들의 코덱을 사전 인코딩으로 통일할 수 있다. 틀리면 트랜스코딩이 부활 → C4 위반 → 미디어 서비스만 퇴거(플랜은 유지).
- 가정 3: USB SSD 1개가 전력 예산 10W 안에 든다. 틀리면 전원 부족으로 SSD가 불안정해짐 → 유전원 허브 추가 후 전력 재실측.
- 가정 4: 게이트 숫자(RAM 2.5GB, 70°C)는 실측 전 추정치다. 가동 2주 실측 후 재조정하는 마일스톤으로 수명을 제한한다. 틀려도 플랜은 유지되고 숫자만 이동.

## 접근 및 단계
1. **기계/상태 분리 기반 공사:** 로그는 RAM(log2ram), 컨테이너와 상태 디렉토리는 SSD, 전체 구성은 git 저장소의 compose 선언 1벌. → 검증: **재구축 리허설 — SD를 뽑고 예비 SD에 처음부터 재구축, 30분 내 전 서비스 복귀.** 이 리허설을 통과하기 전에는 어떤 서비스도 "운영 중"이 아니다.
2. **입주 게이트 3개 문서화(판단의 규칙화 — 입주 시 1회 심사):**
   - RAM 게이트: 입주 후 idle RSS 합계 ≤ 2.5GB. 초과 시 새 입주 금지, 퇴거가 선행.
   - 쓰기 게이트: SD에 상시 쓰기를 만드는 프로세스는 입주 금지.
   - 가족 게이트: "3일 죽으면 운영자 외 누가 아는가?" 답이 존재하면 입주 금지.
   → 검증: 서비스마다 게이트 통과 기록 1줄.
3. **스택 입주(게이트 통과자):** Home Assistant(단, 기존 수동 경로 — 조명 스위치 등 — 를 대체하는 자동화는 금지, 편의 추가만. 가족 게이트를 서비스 내부 기능에까지 재귀 적용), Syncthing(개인 파일 동기화), Jellyfin(direct play 한정), Vaultwarden(운영자 개인용). → 검증: idle RAM 합계·온도 1주 실측 후 가정 4의 숫자 확정.
4. **백업 회로:** 상태 디렉토리 → 암호화 → 클라우드, 일 1회. 복원 리허설은 분기 1회 — 월 1회는 겸업 운영자의 인지 여유를 초과한다(운영부담 감사 결과). → 검증: 리허설에서 클라우드 백업만으로 상태 복원 성공.
5. **관측 최소화:** 알림 2개만 — 백업 실패, 온도/디스크 임계 초과. Prometheus+Grafana류 상시 대시보드는 그 자체가 RAM 게이트 탈락이다(제약을 감시하는 장치에도 제약을 재귀 적용). → 검증: 월 운영 개입 횟수 기록 ≤ 1회.

**발열 퇴거 규칙(기본값 있는 게이트):** 5분 평균 70°C 초과가 주 1회 이상이면 사전 결정된 순서로 퇴거 — Jellyfin → Home Assistant 애드온 → 그래도 초과면 스택 규모 재협상. 여름 한복판에서 결정하지 않고 지금 결정해 둔다.

## 대안과 기각 사유 — 제약이 죽인 것들의 포기 목록
- **Pi-hole 가족 DNS:** C5×무이중화가 죽임. 정석 1번이라 가장 아쉽다. 부활 조건: 파이 2대째 확보, 또는 공유기에 보조 DNS 폴백을 걸 수 있고 "폴백 시 광고 차단 무효"를 가족이 아닌 운영자만 감수하는 구성이 가능할 때.
- **실시간 트랜스코딩:** C4가 죽임. 사전 인코딩으로 정밀화(요구는 시청이지 변환이 아니므로 완화가 아니다).
- **HDD 2개 RAID NAS:** C3+C4가 죽임(회전 소음 + 전력 초과). 대체: SSD 1개 + 클라우드 백업. 이 대체가 가정 1의 구멍을 만들었다 — 죽인 것의 생존자에게 제약을 다시 들이대자 구멍이 드러났고, 그래서 선언했다.
- **공개 인터넷 노출(포트포워딩 + 리버스 프록시):** 운영 표면적, 그리고 "보안 사고 = 가족 눈치의 최대치"가 죽임. 대체: Tailscale 사설망. 부활 조건: 없음 — 이 예산과 인지 여유의 세계가 아니다.
- **중고 미니PC로의 상향:** C3(10W vs 30~60W)과 C4가 죽임. 부활 조건: RAM 게이트가 퇴거로도 해소되지 않는 시점 — 그때는 "파이 1대"라는 이 플랜의 전제가 수명을 다한 것이므로 플랜 자체를 다시 쓴다.

## 리스크 & 완화
- **R1 — SD 급사:** 모순 판정 1로 이미 정상 운영으로 재분류됨. 예비 SD 상비 + 1단계 리허설이 복구 시간을 보증한다. 리허설 없는 분리 선언은 장식이다.
- **R2 — SSD 단일 장애(분리가 만든 새 단일 장애점):** 상태 유실은 클라우드 백업으로 복구(RPO 24시간). 가족 게이트 덕에 복구 중에도 가족 비가시.
- **R3 — 규칙이 스스로 짐이 됨:** 게이트는 입주 시 1회 심사이고 상시 점검이 아니다. 알림 2개·분기 리허설 1회로 운영부담의 상한을 고정.
- **R4 — 백업은 도는데 복원이 안 됨(단계 사이의 구멍):** 백업 성공 알림은 복원 가능성을 증명하지 않는다. 분기 리허설이 유일한 증명 수단이며, 실패 시 그 분기의 최우선 운영 과제로 승격.

## "완료"의 판정 기준
- **SD 카드를 뽑아 폐기하고, 예비 SD + git 선언 + 클라우드 백업만으로 30분 내 전 서비스 복구** (모순 판정 1의 실측).
- **서버를 의도적으로 3일 내려도 가족 누구도 언급하지 않음** (C5의 실측 — 이 문장이 통과되면 Pi-hole 기각이 옳았다는 사후 증거).
- 스마트플러그 실측 전력 ≤ 10W, 팬 0개, idle RAM 합계 ≤ 2.5GB(가정 4의 재조정 값 기준).
- 포기 목록의 모든 항목에 "어느 제약이 죽였는지"와 부활 조건이 표기되어 있음.

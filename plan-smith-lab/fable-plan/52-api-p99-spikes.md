# 간헐적 API p99 레이턴시 스파이크 조사
- 추론 프레임: hypothesis-loop
- 한 줄 요약: 수리가 아니라 재현 레시피가 목표다. p99라는 그림자 대신 그것을 만든 느린 요청들의 모집단을 조사 대상으로 삼고, 가설 6개의 기각 조건을 데이터를 보기 전에 등록한다.

## 프레임이 요구하는 시작점 — 사실과 추측의 분리, 그리고 프레임 자체 심문

이 프레임의 실패 모드는 두 가지다. ① 조기 수렴 — 선호 가설(레이턴시 조사에서는 거의 항상 "GC다")만 파는 것. ② 가설 없는 계측 살포 — APM을 켜고 4주 뒤 "흥미로운 데이터"만 남는 것. 이 플랜의 장치: 기각 조건은 데이터 열람 전에 원장에 확정(사후 재해석 봉쇄), 저비용 선별 통과 전 가설당 1인일 초과 투자 금지(수렴 차단기), 모든 신규 계측은 자기가 가르는 가설을 지명(살포 봉쇄).

**사실**(대시보드 수준에서 이미 확인된 것): 5분 집계 p99가 하루 1~수 회, 수 분간 기준선의 3배 이상으로 튄다. p50·p90은 거의 움직이지 않는다. 배포 시각과의 육안 상관 없음. UTC 기준 시간대 편중 안 보임.

**추측**(지문 대조 전): "GC다", "풀 고갈이다", "이웃 소음이다", "재시도 폭풍이다", "특정 페이로드다" — 전부 원장의 행일 뿐, 지위가 같다.

## 층위 이동 — 지표가 아니라 요청 모집단을 조사한다

p99는 요청의 속성이 아니라 **집계 창의 속성**이다. 창은 조사할 수 없다 — 조사 가능한 개체는 요청이다. 조사 단위를 지표에서 **임계 초과 요청 코퍼스**로 옮긴다: 스파이크 창 여부와 무관하게, 요청 단위 임계(기준선 p99의 2배)를 넘는 모든 요청을 차원 태그(시각, 엔드포인트, 인스턴스, 물리 호스트, 테넌트, 페이로드 크기, 재시도 표식, 커넥션 대기 시간, 배포 에포크)와 함께 수집한다.

이 이동이 필요한 이유: 대시보드 층위에서 다섯 가설의 그림자는 전부 "p99가 오른다"로 같고, 지문은 요청 층위에서만 갈라진다. 부수 효용: 스파이크 판정에 못 미친 준(準)스파이크 요청도 표본이 되어 재발 빈도 가정(가정 1)의 압박을 줄인다.

**숫자 전에 의미 고정:**
- **스파이크 판정** = 5분 창 p99가 직전 14일 동시간대 p99 중앙값의 3배 초과, **그리고** 그 창의 임계 초과 요청 30건 이상(저트래픽 창의 1건짜리 착시 배제).
- **기각** = 사전 등록된 기각 조건이 코퍼스 데이터로 충족됨. 조건 개정은 금지가 아니라 이력 의무 — 개정하면 해당 가설은 "보류"로 강등되고 사유가 원장에 남는다.
- **재현 레시피** = 통제 환경에서 절차 10회 실행 시 8회 이상 스파이크 판정을 유발하고, 핵심 변수 1개를 제거하면 1회 이하로 떨어지는 절차 문서.

## 문제 정의 / 목표

간헐적(하루 0~수 회), 수 분 지속되는 p99 스파이크의 원인 규명. 1차 목표는 수리가 아니라 **재현 레시피** — 재현 없는 수리는 "고쳐졌는가"를 판정할 수 없어 회귀를 부른다. 수리는 후속 작업으로 분리한다.

## 명시적 가정 목록 (틀렸을 경우의 영향 포함)

- **가정 1: 스파이크는 조사 기간(4주) 내 주 수 회 이상 재발한다.** 틀리면 루프가 표본 기아로 정지 — 플랜 전체가 걸린 가정. 완화는 층위 이동에 내장(준스파이크 수확). 그래도 월 1회 수준이면 수동 포집 대기 모드로 격하하고 시간 상자 재협상.
- 가정 2: 요청 단위 차원 태그를 기록할 수 있다(트레이싱 또는 구조화 로그). → 틀리면 지문 대조가 상관관계 수준으로 퇴화. 0단계에서 한정 계측 추가하되, 각 계측은 가르는 가설 지명 의무.
- 가정 3: 근원은 우리 서비스와 직접 의존성 안에 있다. → 틀리면(상류/클라이언트 측) 서버 내부 시간으로 분해되지 않는 초과 지연으로 코퍼스에 나타남 — 그 자체가 탐지 가능한 지문이라 H0가 흡수한다.
- 가정 4: 통제 환경(스테이징 또는 프로덕션 카나리아 1대)에서 프로덕션 부하 형상을 근사할 수 있다. → 틀리면 재현 무대를 "통제된 프로덕션 실험"으로 옮겨야 하며 리스크·승인 비용 상승.

## 가설 원장 — 기각 조건 사전 등록

| 가설 | 예측 지문(코퍼스에서) | 기각 조건(사전 등록) | 실험 · 비용 |
|---|---|---|---|
| H0 측정 인공물 | LB/클라이언트 측 레이턴시와 서버 측 레이턴시의 괴리, 집계 파이프라인 지연과 동기 | 스파이크 창의 임계 초과 요청 80% 이상에서 서버 측 측정도 동반 상승 | 기존 로그 대조 · **최저** |
| H1 GC | 한 시점에 단일 인스턴스로 군집, GC 정지 로그와 ±창 정합, 초과 지연 ≈ 정지 시간 | 스파이크의 70% 이상에서 느린 요청이 3개 이상 인스턴스에 동시 분산, 또는 정합하는 정지 기록 부재 | GC 로그 활성화 · 저 |
| H2 풀 고갈 | 초과 지연 대부분이 커넥션 대기 구간으로 분해, 풀 포화 메트릭과 동기 | 느린 요청 80% 이상에서 커넥션 대기가 초과 지연의 30% 미만 | 풀 대기 계측 · 중 |
| H3 이웃 소음 | 군집 축이 서비스 버전이 아니라 물리 호스트, CPU steal·IO wait 동반 | 전용/공유 노드 간 스파이크 빈도 동률, 또는 호스트 축 군집 부재 | 노드 메트릭 결합 · 중 |
| H4 재시도 폭풍 | 스파이크 직전·중 요청량 급증, 느린 요청 중 재시도 표식 과대표집(지연→타임아웃→재시도 증폭 고리) | 스파이크 80% 이상에서 요청량 평상 대역 이내이고 재시도 표식 비율이 기준선과 동률 | 재시도 헤더 대조 · 저 |
| H5 특정 페이로드 | 엔드포인트·테넌트·페이로드 크기·쿼리 형상 중 한 축에서 리프트 3배 이상 과대표집 | 어떤 축에서도 리프트 3배 이상 없음 | 코퍼스 분석만 · **최저** |

**좌표계 의심(원장 대조 전 의무 1회):** 코퍼스를 축을 바꿔 재집계한다 — UTC↔로컬 시간·요일, 인스턴스, 호스트, 엔드포인트, 테넌트, 배포 에포크, 그리고 **스파이크 간 간격 분포**. 간격이 정확히 60초·정시의 배수로 주기성을 보이면 스케줄 작업(cron, 토큰 갱신, 캐시 만료)이라는 잠재 가설 H6으로 승격한다. 서비스 전체×5분 격자에서 균일해 보이는 것이 인스턴스×1분 격자에서는 군집일 수 있다 — 집계 단위가 은폐한 군집부터 걷어낸다.

**전멸 규칙(프레임의 재귀 적용):** 전 가설이 기각되면, 원장 밖의 숨은 전제 — "스파이크는 단일 현상이다" — 를 심문한다. 코퍼스를 지문 유사도로 군집화하고 군집별로 원장을 다시 돌린다. 서로 다른 두 원인이 다중화되어 있으면 어느 가설도 단독으로 80% 조건을 못 넘겨 전멸이 나는 것이 정상이기 때문이다.

**원장 밖 범주 대기열(망라성 점검):** DNS/TLS 핸드셰이크, 디스크 지연, 의존 서비스 자체의 스파이크, OS 수준 이벤트. 코퍼스 지문이 원장 어디에도 안 맞으면 이 대기열에서 승격한다.

## 접근 및 단계

0. **코퍼스 구축**(1주, 병행 수집) → 검증: 스파이크 1회당 임계 초과 요청 30건 이상 포집, 차원 결손률 5% 미만.
1. **H0 대조 + 좌표 스윕** → 검증: 축별 리프트 표가 원장에 첨부되고 H0 판정이 기록됨.
2. **저비용 지문 대조를 병렬로**(H5·H4·H1 먼저, H2·H3 나중 — 실험 비용 오름차순) → 검증: 각 가설이 채택 후보/기각/보류로 판정되고 근거 데이터 링크가 붙음.
3. **심화 게이트**: 지문 대조를 통과한 가설만 개입 실험(풀 크기 2배 A/B, GC 파라미터 카나리아, 재시도 예산 강제 등) 진입. 통과 전 어떤 가설에도 1인일 초과 투자 금지 → 검증: 원장의 투입 시간 기록.
4. **재현 레시피 작성 + 탈재현**(핵심 변수 제거 시 소멸 확인) → 검증: 아래 "완료" 기준 1.
5. **시간 상자 게이트**: 영업일 10일 내 채택 후보 0개면 기본값 발동 — 무원인 완화(타임아웃 정리 + 헤지 요청)로 전환. 단, **헤징은 H4가 기각된 경우에만 허용**(재시도 폭풍 위에 헤징을 얹는 것은 증폭기를 하나 더 다는 일). H4 미기각 상태의 기본값은 재시도 예산 도입이다.

## 대안과 기각 사유

- **APM 전면 도입부터:** 기각 — 가설 없는 계측 살포는 이 프레임의 실패 모드 그 자체. 부활 조건: 가정 2가 무너져 코퍼스가 불가능해지면 계측이 0단계로 승격 — 단 모든 메트릭이 가르는 가설을 지명해야 한다.
- **완화 먼저(오토스케일·헤징), 조사 나중:** 기각 — 신호를 가려 코퍼스를 오염시키고, H4가 원인이면 증상을 증폭시킨다. 부활 조건: SLO 소진율이 주당 오류 예산의 30%를 넘으면 진단보다 지혈이 우선 — 증거 대 속도의 트레이드를 공개적으로 선택하고 전환한다.
- **"일단 GC 튜닝"(관성적 첫수):** 기각 — 조기 수렴의 화신. GC는 원장의 한 행일 뿐 특권이 없다. 부활 조건 없음 — H1이 지문 대조를 통과하면 그때 심화하는 것이지, 먼저 만지는 일은 없다.

## 리스크 & 완화

- **R1: 코퍼스 수집 자체가 부하·비용.** → 임계 초과 요청 100% + 정상 요청 1% 표본만 수집, 저장 상한 사전 설정. 관측이 관측 대상을 바꾸면 그것이 새 H0다.
- **R2: 조사 중에도 배포는 계속된다(기준선 동결 불가).** → 배포 에포크를 코퍼스 차원으로 기록하고 에포크 경계의 빈도 변화를 스윕 축에 포함.
- **R3: 기각 임계값(70%/80%/리프트 3배)의 자의성.** → 의미는 고정했으나 수치는 초깃값임을 인정. 개정은 이력 의무(원장 기록 + 해당 가설 보류 강등)로만 허용.
- **R4: 운영 부담 — 원장·게이트·스윕이 온콜 위의 제2 업무가 되는 것.** → 조사 소유자 1인, 원장은 표 1장, 갱신은 주 2회 30분 고정. 이 리듬이 깨지면 규칙을 줄이는 게 맞다 — 규칙이 실행자를 묻으면 프레임의 자기 배반이다.

## "완료"의 판정 기준

1. **실측 문장:** 통제 환경에서 재현 절차 10회 실행 중 8회 이상 스파이크 판정 조건을 유발하고, 핵심 변수 1개를 제거한 대조 10회에서는 1회 이하로 떨어진다.
2. **원장 마감:** 6개(+승격분) 가설 전부가 채택/기각/보류로 판정되고, 각 판정에 사전 등록 조건과 대조한 데이터 링크가 붙어 있다. 보류에는 재개 조건이 명시되어 있다.
3. **(후속 작업의 기준, 이 플랜의 기준 아님):** 수정 배포 후 4주간 스파이크 판정 빈도가 조사 전 4주 대비 80% 이상 감소. 재현 레시피가 있어야만 이 문장이 판정 가능해진다는 것이 1차 목표를 레시피로 둔 이유다.

# 새벽 3시 프로덕션 DB 커넥션 고갈 장애 런북
- 추론 프레임: 시간압박 우선순위
- 한 줄 요약: 3계층(5분/30분/사후)으로만 구성한다. 5분 계층의 첫 90초는 "생각"이 아니라 "두 갈래 분류"이고, 모든 조치에는 부작용과 발동 조건이 붙는다.

## 프레임이 요구하는 시작점 — 이 문서의 사용 조건

읽는 사람: 방금 깬 당직자, 인지 능력 평소의 절반. 문서 규칙: ① 위에서 아래로 실행 가능, ② 조치마다 **부작용** 병기(새벽의 최대 적은 조치가 만드는 2차 장애), ③ 원인 규명은 복구 뒤로 — 단, **90초 분류 1회**만 허용한다. 분류 없는 지혈은 절반의 확률로 헛손질이기 때문이다(누수엔 재시작이 듣고, 폭주엔 안 듣는다).

## 문제 정의 / 목표

커넥션 풀 고갈로 앱이 DB에 못 붙는 상태. 목표 우선순위: ① 사용자 트래픽 복구 → ② 재발 없이 아침까지 버티기 → ③ 근본 원인은 해 뜨고 나서.

## 계층 1 — 5분 안 (지혈)

**[90초 분류] DB에서 활성 세션 요약 1회 조회** (`pg_stat_activity`의 state별 카운트).
- 부작용: 없음(읽기). 단 90초 넘기지 말 것 — 분류는 완벽할 필요 없고 지배적 패턴만 보면 된다.
- **패턴 ⓐ: `idle in transaction` 다수** → 앱이 트랜잭션을 열고 안 닫음(누수형). → **1-A로.**
- **패턴 ⓑ: `active` 다수 + 같은 쿼리 반복** → 느린 쿼리/폭주형. → **1-B로.**
- 판단 안 서면 → 1-A부터(더 흔하고 부작용이 작다).

**[1-A] 누수형 지혈**
1. `idle in transaction` 세션 중 오래된 것 일괄 종료(kill). → 부작용: 해당 트랜잭션 롤백 — 열고 방치된 트랜잭션이므로 대부분 안전하나, 배치 작업이면 재실행 필요.
2. 효과 없으면 앱 인스턴스 롤링 재시작(풀 강제 회수). → 부작용: 진행 중 요청 유실, 재시작 순간 에러 스파이크. **롤링**으로(전체 동시 재시작 금지 — 복구 후 재연결 폭풍이 DB를 두 번 죽인다).
3. 최근 배포가 있었다면(6시간 내) 직전 버전 롤백. → 부작용: 신기능 소실, DB 마이그레이션 동반 배포면 롤백 불가할 수 있음 — 이 경우 시도 금지하고 2번 유지.

**[1-B] 폭주형 지혈**
1. 지배 쿼리를 식별해 해당 쿼리만 kill. → 부작용: 그 기능(리포트 등) 일시 실패.
2. 발원지가 배치/크론이면 중지. → 부작용: 배치 미완 — 아침 재실행 목록에 기록.
3. 외부 트래픽 폭주면 해당 엔드포인트 레이트리밋/차단. → 부작용: 일부 사용자 기능 제한 — 전면 장애보다 낫다.

**금지 조치(새벽의 유혹):** DB `max_connections` 상향. → DB 메모리는 커넥션당 소모된다. 새벽에 한도를 올리는 것은 고갈을 OOM으로 바꾸는 것 — 더 나쁜 장애로의 교환. 계층 3에서만 검토.

계층 1 탈출 조건: 에러율 정상 복귀 + 커넥션 사용률 80% 미만 10분 유지.

## 계층 2 — 30분 안 (아침까지 버티는 가드)

1. **타임아웃 이중 설치:** 앱 풀에 커넥션 획득 타임아웃, DB에 `idle_in_transaction_session_timeout` + `statement_timeout`(보수적으로, 예: 30s). → 부작용: 정상적으로 긴 작업(대형 리포트)이 잘림 — 잘리는 작업 목록을 기록해 사후 화이트리스트.
2. **재발 감시 강제화:** 커넥션 사용률 80% 알림을 지금 건다(이미 있다면 임계 하향). → 부작용: 새벽 오탐 가능 — 감수한다, 두 번째 고갈을 자면서 맞는 것보다 낫다.
3. **원인 후보 스냅샷 보존:** kill 하기 전의 `pg_stat_activity` 덤프, 시간대 로그, 배포 이력을 한 폴더에. → 부작용: 없음. 사후 분석의 원료 — 지혈 과정에서 증거가 소멸되는 것을 막는다.
4. 1-A/1-B 조치의 임시성 명시: 오늘 밤 롤백·차단한 것들의 목록을 인수인계 문서로. → 다음 근무자가 "왜 이게 꺼져 있지"로 2차 사고를 내는 것 방지.

## 계층 3 — 사후 (해 뜨고 나서)

1. **원인 확정:** 계층 2-3의 스냅샷으로 누수 지점(트랜잭션 열고 안 닫는 코드 경로) 또는 폭주 지점 특정. 누수면 해당 경로에 try-finally/컨텍스트 매니저 강제 + 정적 검사 도입.
2. **구조 개선 검토:** 커넥션 풀 프록시(pgbouncer 등) 도입 — 앱 수평 확장이 DB 커넥션 수를 직접 늘리지 않는 구조로. `max_connections` 조정은 이 검토 안에서만.
3. **타임아웃 화이트리스트:** 계층 2-1에서 잘린 정상 장기 작업을 별도 풀/세션 설정으로 분리.
4. **런북 회귀:** 오늘 밤 실제로 쓴 조치·안 쓴 조치·막힌 지점(권한? 대시보드 부재?)으로 이 문서를 수정. 특히 90초 분류가 실제로 90초에 됐는지 — 안 됐다면 그 조회 쿼리를 저장된 원클릭으로 만든다.
5. **게임데이:** 스테이징에서 커넥션 고갈을 의도적으로 재현하고 이 런북으로 복구 훈련(개정판 검증).

## 명시적 가정 목록 (틀렸을 경우의 영향 포함)

- **가정 1:** 당직자가 DB 세션 조회·kill, 앱 재시작, 레이트리밋 권한을 즉시 보유. → 틀리면 이 런북은 무용지물 — **권한 사전 점검이 이 문서의 배포 조건**이다(사후 4에서 검증).
- **가정 2:** `pg_stat_activity` 급 세션 가시성 존재(RDS 등 관리형도 동등 뷰 제공). → 틀리면 90초 분류 불가 → 1-A 기본 경로 단독 운용(효과 확률 하락 명시).
- **가정 3:** 시스템이 PostgreSQL 계열. → 다른 DB면 명령은 바뀌나 계층 구조와 누수/폭주 이분법은 유지.

## 대안과 기각 사유

- **분류 없이 무조건 전체 재시작:** 기각. 폭주형이면 재시작 직후 재고갈 + 재연결 폭풍으로 악화. 90초 분류의 기대 수익이 비용을 압도한다.
- **완전 원인 규명 후 조치:** 기각. 새벽의 각 분은 사용자 피해다. 규명은 증거 보존(2-3)으로 대체하고 사후로 미룬다.
- **max_connections 즉시 상향:** 기각(금지 조치로 명문화). 위 참조.

## 리스크 & 완화

- **R1: 90초 분류가 오분류.** → 두 경로 모두 안전측 설계(1-A 우선 규칙 + 각 조치 부작용 명시)라 오분류 비용이 제한적. 분류는 방향 설정이지 확정 진단이 아님을 문서에 명시.
- **R2: 당직자가 이 문서를 처음 봄.** → 게임데이(3-5)로 훈련. 런북은 읽는 문서가 아니라 연습한 문서여야 새벽에 작동한다.

## "완료"의 판정 기준 (런북 문서로서)

- 계층 1만 순서대로 실행해도 두 패턴(누수/폭주) 모두에서 복구에 도달한다(게임데이로 실증).
- 모든 조치에 부작용·발동 조건이 붙어 있고, 금지 조치가 명시돼 있다.
- 권한·가시성(가정 1·2)이 사전 점검되어 "런북은 있는데 권한이 없는" 상태가 아니다.
- 실제 장애 후 이 문서가 개정됐다는 이력이 남는다(살아 있는 문서).

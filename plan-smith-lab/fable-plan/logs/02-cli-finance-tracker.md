# 로그 — 02 CLI 재무 추적 (삭제 우선 / 미니멀리즘) [Fable]

## 오염 노트
Opus 버전이 4개 기능(add/report/edit + 포맷)으로 착지한 것을 안다. 나는 심문을 다시 돌렸고 **edit에서 갈라졌다**. 이 갈라짐이 anchoring의 반작용(일부러 더 깎기)인지 자문했다 — 판단 근거를 적어 검증 가능하게 남긴다: edit 삭제의 논거는 "저장소가 플레인 텍스트라면 편집기는 이미 존재한다($EDITOR)"이고, 이는 미니멀리즘 프레임의 원칙(있는 것을 다시 만들지 않기)에서 직접 나온다. 즉 프레임을 더 세게 적용하면 나오는 자연스러운 결론이라고 판단한다. 다만 "Opus보다 더 깎아야 한다"는 무의식적 경쟁 동기가 0이었다고 증명할 수 없다 — 이것이 동일 세션 순차 실험의 근본 한계다.

## 진입 시 사고
프레임을 읽고 첫 번째로 정한 것은 삭제의 기준점이었다. 삭제는 기준 없이는 취향 싸움이 된다. 그래서 근본 목적을 한 문장으로 고정하고("마찰 0 기록 + 월 1회 회고"), 입증 책임을 기능 쪽에 지웠다(유죄추정). 두 번째로 정한 것이 더 중요했다: **삭제의 지렛대를 먼저 설치**하는 것. "플레인 텍스트 1파일"을 선결 결정으로 놓으니, 이후 심문에서 기능들이 개별 논쟁 없이 "파일이 이미 한다"로 일괄 처형됐다. 삭제 우선 프레임의 실전 요령은 기능을 하나씩 깎는 게 아니라, 여러 기능을 동시에 불필요하게 만드는 구조적 결정을 먼저 찾는 것이라고 봤다.

## edit 삭제 — 이 플랜의 승부처
가장 오래 고민한 항목. "오타 정정 수단이 없으면 데이터 신뢰가 무너져 도구를 버린다"는 반론이 강력했다. 뒤집은 논리: 정정 수단이 없는 게 아니다 — **$EDITOR가 정정 수단이다.** 도구가 edit을 갖는 순간 행 지정 문법, 삭제 확인, 실수 복구 같은 자체 UI 표면적이 생기고, 그건 "에디터를 서투르게 재발명"하는 일이다. 심지어 재심까지 열어("방금 1건 취소만이라도?") 기각을 유지했다. 이 결정으로 최종 기능이 2개가 됐고, "제품은 명령이 아니라 파일 포맷"이라는 이 플랜의 정체성이 완성됐다.

## 심문에서 발견한 것 — 저장소 선택이 기능 수를 지배한다
SQLite 대안을 기각하며 명확해진 구조: 삭제된 기능들의 사망 근거가 전부 "텍스트 파일의 성질"에 걸려 있으므로, 저장소를 DB로 바꾸면 죽은 기능이 전부 부활한다. 즉 이 도구의 기능 수는 기능 목록에서 정해지는 게 아니라 저장소 한 결정에서 정해진다. 미니멀리즘의 실체는 절제력이 아니라 **의존 구조의 설계**라는 것 — 이 플랜에서 가장 일반화 가치가 큰 통찰.

## 프레임 이탈 / 관성 회귀 지점
- 기존 도구(hledger) 채택을 진지 대안으로 검토한 것은 프레임 밖의 내 습관(만들기 전에 있는 것 찾기)이다. 다만 이 습관은 미니멀리즘과 방향이 같아 충돌은 없었다.
- R2(포맷 파손 시 경고)에서 "조용히 삼키지 않음"을 명시한 것은 미니멀리즘보다 견고성 성향의 발로 — 프레임이 요구하지 않은 방어를 또 넣었다.
- 가정 1(개발자형 사용자)에 플랜 전체가 걸려 있음을 굵게 표시했다. edit 삭제가 이 가정 없이는 성립하지 않기 때문 — 공격적 삭제일수록 가정 명시가 무거워진다는 균형 감각.

## 방향성
"도구를 최소화하는 최선의 방법은 데이터를 도구로부터 독립시키는 것." 완료 기준의 마지막 줄 — 도구가 사라져도 데이터는 완전하다 — 이 이 플랜의 방향을 한 문장으로 요약한다.

## 자가 평가
프레임 충실도: 상. 심문이 실제로 결정을 만들었고(edit 처형), 재심까지 열어 기각을 유지했으며, 최종 2개로 정량 목표를 크게 상회. 흠: 견고성 습관의 혼입(R2), 그리고 오염 조건에서 "더 깎기" 동기의 완전 배제 불가.

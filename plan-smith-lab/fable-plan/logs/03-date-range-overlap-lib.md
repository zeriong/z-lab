# 로그 — 03 날짜 범위 유틸 (API-first) [Fable]

## 오염 노트
Opus 버전은 클래스 API(`new DateRange`)를 예시로 썼고 반열림 규약에 도달했다. 나는 **함수형 API + POJO 입력**으로 갈라졌다. 갈라짐의 근거를 명시한다: API-first의 첫 질문은 "사용자 코드가 어떻게 생겼는가"인데, 이 라이브러리의 사용자는 범위 데이터를 이미 `{start, end}` 형태로 갖고 있다(DB, API 응답). 그 데이터를 클래스로 감싸게 하는 것은 사용자에게 세금이다. 이 논리는 프레임의 1번 원칙에서 직접 나온다. 반열림 규약의 수렴은 오염이라기보다 수학적 강제로 본다 — subtract 기대값을 정직하게 쓰면 어떤 작성자든 같은 곳에 도달한다(로그의 갈라짐 비교 재료: 프레임이 강제하는 수렴 vs 작성자 재량의 발산).

## 진입 시 사고
예시 10줄을 쓰기 전에 스스로 물은 것: "이 라이브러리 사용자의 손에 이미 무엇이 들려 있는가?" API-first를 "예시를 먼저 쓴다"는 절차로만 이해하면 저자 취향의 예시를 쓰게 된다. 예시는 사용자의 기존 세계(데이터 모양, 호출 맥락)를 반영해야 명세 자격이 있다. 그래서 예시의 변수를 `vacation`, `oncall`처럼 실사용 냄새가 나는 데이터로 쓰고, 입력을 POJO로 못 박았다. 첫 줄(`import { overlaps, ... }`)이 이 플랜의 최대 결정이었고, 나머지는 그 파생.

## 예시가 결정을 만든 순간들
- subtract 기대값을 쓰는 순간 반열림이 강제됐다(경계일 7/10이 어느 쪽에도 중복되지 않으려면). 규약을 고민해서 고른 게 아니라 기대값이 규약을 연역했다.
- 빈 구간(`start === end`)을 예시 마지막 줄에 **일부러** 넣었다. 에지 케이스를 계약에 미리 박제하면 이슈 트래커의 미래 논쟁 하나가 죽는다. API-first의 예시는 해피 패스 전시가 아니라 분쟁 예방 계약서라는 관점.
- ISO 문자열의 사전순 == 시간순 성질을 깨달은 순간 내부 설계가 통째로 단순해졌다(파싱 0, Date 객체 0). 이건 예시를 "문자열 입력"으로 쓴 것의 후행 보상이었다 — 사용자 편의로 고른 입력 타입이 구현 단순성까지 끌고 왔다. 운이 아니라, 사용자에게 자연스러운 표현이 대체로 도메인의 본질적 표현이기 때문이라고 해석한다.

## 속성 테스트 — 프레임 밖에서 가져온 것
"subtract + intersect를 merge하면 원본"이라는 대수 불변식 검증은 API-first가 요구한 게 아니라 내 정확성 성향이다. 다만 이것이 규약의 일관성(이 플랜의 목표 품질)을 지키는 가장 강한 도구라 프레임의 목적에 종속된다고 판단해 포함했다. 01(증명 부담), 02(fail-fast)에 이어 정확성/견고성 습관이 세 플랜 연속 혼입 — 내 지문일 가능성이 높아지고 있다. 메타리뷰에서 추적.

## 프레임 이탈 / 관성 회귀 지점
- R3(혼합 정밀도)은 예시에 없는 시나리오를 방어한 것 — 계약 밖 방어이므로 엄격한 API-first에선 과잉일 수 있으나, 문자열 비교 전략의 유일한 "조용한 오답" 구멍이라 막았다. 계약주의와 견고성의 타협점.
- "접근 및 단계"의 선형 서술은 여전하나, 1단계를 "규약 문서를 코드보다 먼저 커밋"으로 두어 프레임의 우선순위를 순서에 반영했다.

## 방향성
"사용자가 이미 가진 것을 계약으로 승격하라." 01이 증명 부담의 위임, 02가 데이터의 도구 독립이었다면, 03은 **인터페이스의 사용자 세계 종속**이다. 셋을 관통하는 축: 내가 소유하는 표면적을 최소화하고 나머지를 사용자/생태계/라이브러리에 맡긴다.

## 자가 평가
프레임 충실도: 상. 예시가 실제로 최상위 결정(함수형/POJO)과 규약(반열림, 빈 구간)을 연역했고, 예시 자체를 CI 명세로 승격했다. 흠: 계약 밖 방어(R3)와 속성 테스트의 혼입은 프레임 순수성보다 내 성향.

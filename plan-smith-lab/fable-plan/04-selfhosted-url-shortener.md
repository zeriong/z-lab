# 셀프호스팅 URL 단축 서비스
- 추론 프레임: Constraint-first
- 한 줄 요약: 제약 3개를 벽으로 세우면 모순이 드러난다. 모순의 해소는 "리다이렉트를 서버가 아니라 산출물(artifact)로 만드는 것" — 읽기 경로에서 서버를 제거한다.

## 프레임이 요구하는 시작점 — 제약을 먼저, 해 공간은 그 안에서만

- **C1 (돈):** 월 $5 서버 1대. RAM ~1GB, 관리형 DB·LB·복제본 금지.
- **C2 (사람):** 운영자 1인 겸업. 새벽 호출을 유발하는 모든 구조 금지. 운영 개입 빈도가 월 1회를 넘으면 설계 실패로 간주.
- **C3 (가용성):** 다운타임 허용 불가.

**모순 판정부터:** C1은 이중화를 금지하고, C3은 단일 장애점을 금지한다. 서버 1대는 언젠가 반드시 죽으므로, **"서버가 살아 있어야 서비스가 산다"는 구조를 유지하는 한 C1∧C3은 충족 불가능**하다. Constraint-first의 결론: 서비스를 살리려면 요구를 깎는 게 아니라, **서비스의 핵심 경로에서 서버라는 존재를 제거**해야 한다.

## 모순의 해소 — 리다이렉트는 계산이 아니라 데이터다

관찰: 단축 URL의 읽기 경로(`/abc` → 301)는 요청 시점에 계산할 것이 없다. 매핑은 쓰기 시점에 이미 확정된 **정적 데이터**다. 그렇다면 읽기를 서버 응답이 아니라 **빌드 산출물**로 만들 수 있다:

> 쓰기(링크 생성) 시마다, 전체 매핑을 정적 리다이렉트 산출물로 컴파일해 **엣지(무료 티어 정적 호스팅/엣지 KV)에 배포**한다. 읽기 트래픽은 우리 서버에 도달하지 않는다.

이 구조에서 C3의 의미가 정확해진다:
- **읽기(기존 링크):** 엣지가 응답. 우리 서버가 꺼져 있어도, 심지어 며칠 죽어 있어도 기존 링크는 전부 동작. → C3 충족(사용자가 체감하는 다운타임 = 기존 링크 실패이므로).
- **쓰기(새 링크):** 서버가 죽으면 몇 분간 생성 실패. → 이것은 "다운타임"이 아니라 "쓰기 지연"으로 재분류. 운영자 1인 서비스에서 쓰기는 사실상 운영자 본인·소수 사용자의 저빈도 행위다.

캐시로 때우는 접근(오리진 + CDN 캐시)과의 차이: 캐시는 오리진이 "원본"이라 miss·만료·신규 링크에서 오리진 생존을 요구한다. 산출물 배포는 엣지가 원본의 완전 사본이라 **오리진의 생존이 읽기 경로의 전제조건에서 완전히 빠진다.** C3을 근사가 아니라 구조로 충족.

## 문제 정의 / 목표

개인/소규모 URL 단축기를 위 제약 안에서. 목표는 기능이 아니라 **"운영자가 잊고 살아도 링크가 죽지 않는 구조"**(C2와 C3의 교집합).

## 명시적 가정 목록 (틀렸을 경우의 영향 포함)

- **가정 1:** 링크 수 수만 건 이하, 신규 생성 빈도 낮음(일 수십 건 이하). → 틀리면(대량·고빈도) 전체 재컴파일 배포가 느려져 증분 배포 필요. 수백만 건이면 이 아키텍처 자체 재검토.
- **가정 2:** 엣지 무료 티어(정적 호스팅/KV)의 지속 가용. → 틀리면 아키텍처의 기둥이 빠짐. 완화: 산출물이 표준 형식(정적 파일)이라 타 엣지로 이식이 한나절.
- **가정 3:** 링크 생성 지연(쓰기 경로 다운) 수 분~수 시간이 허용된다. → 틀리면(즉시 생성이 SLA) C1∧C3 모순이 재등장, 예산 재협상 필요.
- **가정 4:** 클릭 통계는 요구가 아니다(엣지 로그 수준이면 충분). → 틀리면 읽기 경로에 동적 계산이 부활해 구조 붕괴. **이 가정이 아키텍처의 숨은 기둥.**

## 접근 및 단계

1. **진실의 원본:** $5 서버의 SQLite 1파일(`code → url`). 쓰기 API + 관리 화면만 이 서버가 담당.
2. **컴파일러:** 쓰기 발생 시 매핑 전체를 엣지 배포 형식(정적 `_redirects` 파일 또는 엣지 KV 벌크 업로드)으로 변환·푸시. 배포 실패 시 재시도 큐(로컬 파일 큐면 충분 — C1).
3. **읽기 경로:** 도메인을 엣지에 연결. 우리 서버는 읽기 트래픽을 아예 받지 않는다.
4. **백업:** SQLite 파일 + 최신 산출물을 오브젝트 스토리지 무료 티어에 일 1회 스냅샷. 복구 = 새 서버에 파일 2개 복원(C2: 복구 절차가 1페이지).
5. **관측(C2 수준으로 최소):** 알림은 2개만 — "엣지 배포 실패 연속 3회", "디스크 80%". 둘 다 새벽 대응 불필요(읽기는 계속 살아 있으므로).

## 대안과 기각 사유 (제약이 죽인 것들)

- **오리진 서버 + CDN 캐시:** 기각. 신규 링크·캐시 미스가 오리진 생존을 요구 → C3이 오리진 가동률에 다시 종속. 산출물 방식이 상위 호환. **(C3이 죽임)**
- **Postgres + Redis + 이중화:** 기각. C1이 죽임. 정면 HA는 예산의 10배 세계.
- **서버리스 전면 이전(엣지 함수 + 유료 KV):** 절반 기각. 읽기는 이미 엣지로 갔다. 쓰기까지 서버리스화하면 $5 서버를 없앨 수 있으나, 관리 화면·백업 제어의 자유도가 떨어지고 무료 티어 약정에 전면 종속(가정 2 리스크 증폭). "셀프호스팅"이라는 문제 정의도 위반.
- **클릭 통계·커스텀 도메인·만료 링크:** 기각. C2(운영 표면적)와 가정 4를 지키기 위해. **제약 때문에 포기한 것 중 가장 아쉬운 것은 클릭 통계** — 엣지 제공 로그로 대체하고, 그 이상은 이 예산의 세계가 아님을 명시.

## 리스크 & 완화

- **R1: 배포 파이프라인 자체가 새 단일 장애점(쓰기 경로).** → 실패해도 기존 링크 무영향(읽기와 격리됨). 재시도 큐 + 실패 알림. "쓰기의 장애 ≠ 서비스의 장애"가 이 설계의 안전 특성.
- **R2: SQLite와 엣지 산출물의 불일치(배포 중단 시).** → 산출물에 버전 스탬프, 배포는 원자적 스왑(전체 파일 교체). 불일치는 "최신 링크 몇 개 지연"으로만 나타남 — 가정 3 안.
- **R3: 서버 완전 소실.** → 4단계 백업에서 복원. 복원하는 동안에도 모든 기존 링크 정상 — 이 문장이 이 아키텍처의 존재 이유다.

## "완료"의 판정 기준

- **오리진 서버를 내리고 24시간 방치해도 기존 링크 전부 정상 응답** (C3의 구조적 충족을 실측).
- 월 청구서 $5 이하 + 엣지 무료 티어 한도 내(C1 실측).
- 월간 운영 개입 0~1회, 장애 복구 절차 1페이지·30분 이내(C2 실측).
- 포기 목록(통계·즉시 쓰기 SLA·커스텀 도메인)이 "제약 때문"으로 문서화되어 있고, 각각 어느 제약이 죽였는지 표기됨.

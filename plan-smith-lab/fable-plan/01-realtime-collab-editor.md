# 실시간 협업 텍스트 에디터 (웹)
- 추론 프레임: Backward-chaining
- 한 줄 요약: 인수 조건 5개를 먼저 못 박고 역산한다. 조건에 없는 것(프레즌스 등)은 아키텍처를 결정할 자격도 없다.

## 프레임이 요구하는 시작점 — 최종 인수 조건 5개

이 시스템이 완성됐다고 선언하려면 아래 다섯 문장이 검증 가능하게 참이어야 한다. 모든 기술 선택은 이 다섯 문장에서만 역산된다. 여기 없는 요구는 v1의 아키텍처를 움직일 수 없다.

- **AC1 (수렴):** 임의의 편집 시퀀스 후 네트워크가 안정되면, 모든 클라이언트의 문서가 **바이트 단위로 동일**하다.
- **AC2 (무손실 동시성):** 두 사용자가 같은 단어 안에 동시에 타이핑해도 **두 입력이 모두 보존**된다. 어느 쪽도 덮어써지지 않는다.
- **AC3 (오프라인 왕복):** 15분 오프라인으로 편집한 클라이언트가 재접속하면, **수동 충돌 해소 UI 없이** 자동 병합된다.
- **AC4 (로컬 즉시성):** 키 입력의 화면 반영은 네트워크 상태와 무관하게 **다음 프레임 안에** 일어난다.
- **AC5 (협업 undo):** Ctrl+Z는 **내가 한 마지막 편집만** 되돌린다. 그 사이 도착한 상대방의 편집은 건드리지 않는다.

의도적으로 **제외한 것**: 커서 프레즌스, 권한, 리치텍스트. 이것들은 인수 조건이 아니므로 아키텍처 결정권이 없다 — backward-chaining의 규율은 "조건에 없는 것이 설계를 흔들지 못하게 하는 것"까지 포함한다.

## 문제 정의 / 목표

여러 사용자가 한 문서를 동시에 편집하는 웹 에디터. 진짜 문제는 에디터 UI가 아니라 **분산 복제본의 병합 의미론**이다. 목표를 "AC1~AC5를 동시에 만족하는 최소 시스템"으로 고정한다.

## 명시적 가정 목록 (틀렸을 경우의 영향 포함)

- **가정 1:** 문서당 동시 편집자는 수십 명 이하. → 틀리면(수백~수천) 연산 메타데이터 누적이 병목이 되어, 서버가 순서를 강제하는 설계의 상대 이점이 커짐.
- **가정 2:** 대상은 플레인 텍스트/마크다운. 블록·표 같은 트리 구조는 v1 제외. → 틀리면 시퀀스 병합만으로 부족, 트리 수준 병합 의미론이 추가로 필요.
- **가정 3:** 중앙 릴레이 서버는 허용되지만, 서버가 죽어도 편집 자체는 계속돼야 한다(AC4의 정신). → 틀리면(서버 권위 허용) OT 계열의 부활 여지.

## 접근 및 단계 — 각 조건이 무엇을 죽이고 무엇을 강제하는가

**AC2에서 역산:** "같은 단어에 동시 타이핑, 둘 다 보존"은 병합이 위치(오프셋) 기반이면 성립하지 않는다 — 상대 편집이 끼어들면 내 오프셋이 무효화된다. 따라서 문자마다 **위치와 독립적인 안정 식별자**가 필요하다. 이 한 줄이 이미 자료구조의 방향(식별자 기반 시퀀스)을 강제한다.

**AC1 + AC3에서 역산:** 수렴이 "서버가 전역 순서를 정해줄 때만" 성립하는 설계(OT)는 AC3에서 무너진다. 15분치 발산한 편집 로그를 서버 기준으로 변환(transform)하는 것은 이론상 가능하나 변환 함수의 조합 검증이 지옥이고, 실제 구현 사고의 대부분이 여기서 난다. 반면 병합 연산 자체가 교환·결합·멱등이면(CRDT), 오프라인 로그는 재접속 후 아무 순서로 합쳐도 같은 결과다. → **AC3이 OT를 기각하는 결정타.** AC1은 그 기각을 "정확성 증명 부담" 관점에서 재확인한다: OT의 수렴은 서버 권위 + 변환 함수 정확성에, CRDT의 수렴은 자료구조 성질에 기댄다. 검증해야 할 표면적이 다르다.

**AC4에서 역산:** 로컬 적용이 먼저, 전파는 나중(local-first). CRDT는 이 순서가 기본값이다. 추가로 AC4는 성능 예산도 강제한다 — 문서가 커져도 로컬 삽입이 O(문서 길이)가 되면 안 되므로, 트리 기반 인덱스를 가진 구현이 필요.

**AC5에서 역산:** 협업 undo는 자주 잊히는 결정 인자다. "내 편집만 되돌리기"는 undo 스택이 전역 히스토리가 아니라 **origin(편집 주체)별로 스코프**되어야 함을 뜻한다. 이것을 자체 구현하는 것은 CRDT 본체만큼 어렵다 — 삭제된 범위 위에서의 undo, 상대 편집과 얽힌 undo의 의미론이 필요하다. → 이 요구가 "검증된 라이브러리 채택"을 사실상 강제하는 두 번째 근거다(첫 번째는 AC1의 증명 부담).

**결론 (역산의 합류점):** 안정 식별자 시퀀스 CRDT + origin 스코프 undo + local-first 전파를 모두 프로덕션 수준으로 제공하는 것 — **Yjs** (`Y.Text` + `UndoManager(trackedOrigins)`). 직접 구현은 AC1·AC5 양쪽의 증명 부담 때문에 기각.

**단계 (각 단계는 담당 AC를 명시):**
1. `Y.Text` + 에디터 바인딩(CodeMirror 6) — AC2·AC4의 기반. 이 단계만으로 "혼자 쓰는 에디터"로 동작해야 함.
2. `UndoManager`를 로컬 origin으로 스코프 — AC5. **일부러 2번째에 둔다**: undo 의미론은 전송층보다 먼저 검증 가능하고, 뒤로 미루면 재작업 위험이 가장 큰 부분.
3. 전송층: WebSocket 릴레이로 update(바이너리 델타) 브로드캐스트 — AC1. 릴레이는 순서 권위가 아니라 파이프.
4. 영속화: IndexedDB 로컬 저장 + 상태 벡터 기반 재동기화 — AC3.
5. 인수 조건 5종 자동화 테스트(아래 판정 기준) — 완료 게이트.

## 대안과 기각 사유 (최소 1개)

- **OT + 중앙 서버(구글 독스 계보):** 기각. AC3(장기 오프라인)에서 변환 복잡도 폭발, 가정 3(서버 사망 시 편집 지속) 위반. OT가 이기는 조건은 "가정 1이 깨지고(대규모 동시), 오프라인 요구가 없을 때"뿐 — 본 요구와 반대.
- **주기적 스냅샷 + 3-way merge(git 방식):** 기각. AC2 위반 — 실시간 문자 단위 동시성에서 3-way merge는 충돌 마커를 낳고, 이는 AC3의 "수동 해소 없음"과도 충돌.
- **CRDT 자체 구현:** 기각. AC1·AC5의 정확성 증명 부담을 스스로 지는 선택. 학습 목적이 아닌 한 정당화 불가.

## 리스크 & 완화

- **R1: CRDT 메타데이터(tombstone) 누적으로 문서 비대화.** → 주기적 스냅샷 컴팩션. 가정 1 범위에서는 실측상 문제가 되기 전에 시간을 벎.
- **R2: AC5의 에지 — 상대가 삭제한 범위에 대한 내 undo.** → UndoManager의 기본 의미론을 그대로 수용하고, 이 에지를 테스트로 명문화(자체 정의하려 들지 않음).
- **R3: 릴레이 서버 장애.** → 진실은 클라이언트에 있으므로(local-first) 서버는 무상태 재기동. AC4는 서버와 무관하게 유지됨을 테스트로 보장.

## "완료"의 판정 기준

각 AC를 1:1로 기계 검증한다:
- AC1: 무작위 동시 편집 퍼저(2~4 클라이언트, 10³ 연산) 후 전 복제본 해시 일치.
- AC2: 같은 오프셋 동시 삽입 시나리오에서 두 문자열 모두 존재.
- AC3: 오프라인 15분 시뮬레이션 → 재접속 → 해시 일치 + 충돌 UI 미노출.
- AC4: 10⁴자 문서에서 키 입력→렌더 계측 16ms 이내.
- AC5: A·B 교차 편집 후 A의 undo가 B의 편집을 보존하는 시나리오 테스트.

5개 전부 통과가 유일한 완료 정의다. "기능이 다 있음"은 판정에 쓰지 않는다.

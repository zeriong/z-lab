# 1인 SaaS의 4주 무인 운영
- 추론 프레임: safety-net
- 한 줄 요약: 목표는 "4주간 무사고"가 아니라 "어떤 장애에서도 바닥이 깨지지 않는 것"이다. 폴백 상태(금고 모드) 1개를 먼저 짓고, 모든 이상은 등급별로 자동 강등되며, 4주간 아무것도 안 울렸다면 성공이 아니라 감지 실패부터 의심한다.

## 프레임이 요구하는 시작점 — 바닥의 정의와 금고 모드

이 프레임의 실패 모드부터 심문한다. safety-net은 두 방식으로 장식이 된다. ① 안전판의 유지비 자체가 실행자를 잡아먹는 것 — 이번 실행자는 여행 중이라 인지 여력이 거의 0이므로, **운영자 정기 일과는 주 1회 확인 탭 1번, 그 외 0**을 설계 제약으로 못 박는다. 이 예산을 넘는 장치(일일 대시보드 점검, 상세 리포트)는 짓지 않는다. ② 안전 장치가 한 번도 안 울린 것을 성공으로 착각하는 것 — 그래서 "무발동 시 감지 검증 의무"를 완료 기준에 문장으로 박는다.

바닥은 숫자보다 먼저 의미로 고정한다. 이 서비스의 바닥은 가동률이 아니다:

- **B1 데이터:** 고객 데이터가 유실·유출되지 않고, 백업이 계속 돌며 검증된다.
- **B2 정직:** 상태 페이지가 실제 상태와 일치한다 — 죽었으면 죽었다고 공지되어 있다.
- **B3 과금:** 작동하지 않는 서비스에 과금되지 않는다.

신규 가입, 부가 기능, 응답 속도는 바닥이 아니다 — 조정 대상이지 사수 대상이 아니다. 성공 = "바닥 무붕괴"이며, "목표 도달(무장애·매출 유지)"이 아니다.

**폴백 상태 S2("금고 모드") 1개:** 어떤 상태에서든 한 번의 전이로 도달 가능한 단일 안전 상태. 앱은 유지보수 화면(앱과 다른 호스팅의 정적 페이지), 과금은 일시 정지, 백업과 상태 페이지만 살아 있다. 핵심 성질: **S2는 남은 여행 기간 내내 지속돼도 사업이 죽지 않아야 한다.** S2 4주의 비용(이탈·환불·평판)을 출발 전에 계산해 감당 가능함을 확인한다. 감당 불가면 바닥을 잘못 잡은 것이므로 여기서 설계를 다시 한다 — 바닥은 조정하는 것이지 위반하는 것이 아니다.

## 구조 재정의 — 감시 체계가 아니라 상태 기계

반사적 답("모니터링 + 알림 + 여행지에서 대응")은 문제의 표현을 잘못 잡은 것이다. 대응 주체가 없는 4주라면 감시의 산출물이 '알림'이어선 안 되고 '전이'여야 한다. 그래서 서비스를 운전 모드의 상태 기계로 재정의한다.

- **S0 정상 → S1 저하 운전**(신규 가입 차단, 무거운 기능 차단, 발송 큐잉) **→ S2 금고 모드.**
- **전이 비대칭(프레임의 핵심 부품):** 하강은 즉시·자동 — S0→S1→S2는 감지 즉시 무인 전이. 상승은 느리게·조건부 — S1→S0만 자동 허용(전 지표 연속 6시간 정상 시), **S2에서의 상승은 수동 전용**. 금고에서 나오는 열쇠는 사람만 쥔다.
- **미분류는 전부 하강:** 등급표에 없는 이상은 판단하지 않고 S2로 보낸다. 런타임 판단을 요구하는 규칙은 무인 체계에서 규칙이 아니다.

등급의 의미를 먼저 고정한다(구체 숫자는 조용한 주에 실측으로 채운다):

- **등급0** = 자동 조치로 5분 내 원복 가능한 이상(프로세스 재시작, 디스크 정리). 조용히 수행, 일지 기록만.
- **등급1** = 핵심 기능(기존 고객의 로그인·데이터 접근) 밖의 저하. → S1 전이.
- **등급2** = 핵심 기능 손상, 또는 미분류 이상. → S2 전이 + 사전 작성된 고객 공지 자동 발송.
- **등급3** = 바닥 자체의 위협(유출 의심 신호, 백업 연속 2회 실패). → S2 전이 + 휴가 불문 운영자 호출. 여행을 깰 가치가 있는 유일한 범주를 지금 정의해 둔다.

## 문제 정의 / 목표

1인 운영 SaaS를 운영자 부재 4주 동안, 바닥(B1~B3)이 한 번도 깨지지 않도록 유지한다. 성공의 정의는 "장애 0건"이 아니라 "바닥 무붕괴 + 감지 체계의 생존 입증"이다.

## 명시적 가정 목록

- **가정 1: 출발 1주 전부터 5주간 변경 동결(배포 0회)이 사업적으로 허용된다.** 1인 SaaS의 최대 장애 원인은 변경이다 — 동결로 잔여 장애 원인을 외부(인프라·결제사·인증서·악용)로 좁히는 것이 전체 설계의 전제. → 틀리면(여행 중 배포 약속이 존재) 무인 전제가 무너지고 이 플랜 전체가 무효.
- 가정 2: 앱에 유지보수/과금 정지 스위치(S2)를 출발 전에 구현·시험할 시간이 있다. → 틀리면 폴백 상태가 없으므로 프레임 미적용과 같다 — 출발 연기가 옳다.
- 가정 3: 주 1회 이상은 인터넷 접속이 가능하다. → 틀리면(완전 오지) 사망자 스위치가 "운영자 도달 불가"로 판정하고 하강 민감도를 올린 최보수 모드로 동작 — 플랜은 유지되나 S2 체류 확률이 오른다.
- 가정 4: 지원·환불·분쟁 규모가 자동 규칙으로 처리 가능한 수준(월 수십 건 이하)이다. → 틀리면 지원 부채가 폭증 — 자동 환불 한도 상향 또는 임시 외주로 완화.
- 가정 5: 결제사·이메일 등 외부 벤더 자체의 장애는 범위 밖. → 틀려도 과금 정지는 B3 안이므로 바닥은 유지된다.

## 접근 및 단계

1. **금고 모드(S2) 구축** — 정적 유지보수 화면(별도 호스팅), 과금 일시정지 훅, 사전 작성 고객 공지. 검증: 스테이징에서 S2 전이·복귀 왕복 1회.
2. **상태 기계·등급 배선** — 외형 감시(실제 로그인 경로의 합성 트랜잭션, 5분 주기) → 등급 판정 → 자동 전이. 검증: 등급별 결함 주입 1회씩, 전이 로그 확인.
3. **감시자의 감시(프레임의 재귀 적용)** — 내부 감시자가 죽으면? 외부 독립 하트비트 서비스와 내부 감시자가 서로를 교차 확인하고, 알림 경로는 이중화(푸시+이메일). 검증: 감시자 프로세스를 죽여 외부발 경보 도달 확인.
4. **사망자 스위치 양방향화** — 매주 월요일, 알림 파이프라인 전체를 관통하는 시험 경보가 운영자 폰에 발송된다. 운영자의 유일한 정기 의무는 수신 확인 탭 1회. 24시간 미확인이면 시스템은 "운영자 도달 불가"로 간주해 하강 민감도를 올린다. 반대로 시험 경보가 안 오면 감지 파이프라인 사망이므로, 운영자는 맹인 상태를 가정하고 예방적 점검(원격 상태 확인, 필요시 선제 S2)을 한다. 검증: 조용한 주에 1사이클 왕복.
5. **저속 장애 소탕(단위 사이 탐색)** — 즉시형 헬스체크 사이로 새는 것들: TLS 인증서·도메인 만료, 호스팅·벤더 결제 카드 만료, 디스크 증가 추세, 메일 발송 쿼터, 결제 분쟁(증빙 제출 7일 시한), 삭제 요청류의 법정 기한. 각각 출발 전 선결(갱신·선결제) 또는 규칙화: 분쟁 $X 이하 자동 승복, 환불 $Y 이하 자동 승인, 자동 지출 총한도 $Z — 초과 시 등급2. 검증: 만료일 전수 조사표에서 5주 내 만료 0건.
6. **인간 폴백 1명, 버튼 2개** — 지인 1인에게 딱 두 권한(S2 전환, 서버 재부팅)과 1페이지 지침. 데이터 접근 권한 없음. 호스팅 계정 잠금 같은 계정 층위 사고 전담. 검증: 리허설 1회.
7. **조용한 주(출발 전 7일)** — 무인 구성을 켠 채로 집에서 생활한다. 손대지 않고 7일을 버텨야 출발 자격. 오탐으로 울리는 등급 임계는 이 주에 실측으로 조정. 검증: 수동 개입 0회.

## 대안과 기각 사유

- **대리 운영자 전권 위임:** 기각. 1인 SaaS의 운영 지식 이전은 몇 주로 되지 않고, 전권 계정 공유는 그 자체가 B1(데이터) 위협이다. 부활 조건: MRR이 커져 "4주 금고 모드의 리스크 비용 > 임시 고용 비용"이 되는 시점.
- **모니터링 + 여행지 대응(낙관 운영):** 기각. 이 프레임이 금지하는 반사적 기본값. 시차·연결성·인지 여력을 전제하며, 폰에서의 압박 수리는 더 큰 장애를 만든다. 부활 조건: 3일 이하의 여행.
- **전면 유지보수 모드 4주(S2 상시):** 기각. 최악 비용을 확정 지불하는 안이다. 단, 이 대안의 계산("S2로 4주를 사업이 버티는가")은 기각 후에도 폐기하지 않고 시작점의 S2 생존성 검증으로 재사용한다.

## 리스크 & 완화

- **R1 감시자 자체의 사망** → 3단계 교차 감시 + 4단계 주간 시험 경보가 파이프라인 전체를 매주 실사한다.
- **R2 S2의 미검증 결함(진짜 장애에서 처음 작동)** → 완료 기준의 프로덕션 실사(아래)로 강제한다.
- **R3 악용·보안(스팸 가입 폭주, 크리덴셜 스터핑)** → 가입 서킷브레이커(시간당 임계 초과 시 가입 자동 폐쇄 = S1). 유출 의심 신호는 등급3으로 직결.
- **R4 오탐 연발로 휴가 잠식** → 등급2 이상만 호출, 0~1은 일지로만. 임계는 조용한 주 실측으로 조정.
- **R5 사소한 원인의 금고행으로 S2 체류 장기화** → 수용한다. 가용성 몇 시간을 내주고 바닥을 지키는 교환이며, 전이 비대칭의 의도된 비용이다.

## "완료"의 판정 기준

- **출발 72시간 전, 프로덕션 앱 프로세스를 예고 없이 죽인다 — 사람 손 없이 10분 내 자동 전이(복구 또는 S1/S2)가 일어나고, 상태 페이지가 갱신되고, 알림이 내 폰에 도달하면 통과.** 실패 시 수정 후 재시험, 재실패 시 출발 연기.
- 조용한 주 7일간 수동 개입 0회, 주간 시험 경보 1사이클 수신 확인 완료.
- 여행 중: 시험 경보 4/4주 수신 확인.
- **귀국 후 판정: 4주간 등급1 이상 발동 0건이면 성공 선언 금지 — 외부 관측 기록(엣지 로그·외형 감시 이력)과 내부 일지를 대조해 감지 누락 0건이 확인될 때만 성공으로 판정한다.** 발동이 있었다면: 모든 전이가 규칙대로 일어났는지, 그리고 바닥 3항목(백업 복원 테스트 1회 통과, 상태 페이지-실상태 불일치 0건, 오과금 0건)이 유지됐는지로 판정한다.

# 백엔드 개발자의 React 8주 학습 계획
- 추론 프레임: misconception-first
- 한 줄 요약: 백엔드 멘탈모델이 React에서 오작동하는 지점 5개를 먼저 지도화하고, 진도를 "주차"가 아니라 "오개념의 사망"으로 센다. 주차는 예산, 관문은 사망 확인서다.

## 프레임이 요구하는 시작점 — 오개념 지도

이 프레임의 실패 모드는 두 가지다. ① 오개념 목록을 만들어 놓고 결국 난이도순으로 가르치는 회귀, ② 간섭할 선행 지식이 없는 백지에 적용하는 헛발질. 이 플랜은 ①을 막기 위해 정렬 기준에서 "난이도"라는 단어를 추방하고 **오염 반경**(그 오개념이 잠복한 채 하류 개념 몇 개를 조용히 오염시키는가)으로 고정하며, ②를 막기 위해 전제조건 검사를 서문이 아니라 **0주차 관문**으로 만든다.

**전제조건 검사(0주차, 통과 못 하면 이 플랜 무효):** 학습자에게 요청-응답·동기 흐름·"진실은 DB에" 습관이 실제로 굳어 있는가(백엔드 3년+ 수준), 그리고 프론트 프레임워크 경험이 없는가를 문진한다. Vue/Angular 경험자는 다른 오개념 지도가 필요하고, 백지 신입에게는 이 프레임 자체가 무력하다 — 그 경우 대안 1(정석 커리큘럼)로 전환.

**오개념 지도 (오염 반경 내림차순):**

- **M1 "렌더 = 요청당 1회 응답":** 서버 템플릿(JSP/Thymeleaf) 모델. 실제로는 컴포넌트 함수 본문이 상태 변화마다 통째로 재실행된다. 잠복 증상: 지역 변수에 상태 저장, stale closure를 버그가 아니라 괴담으로 경험. 오염 반경 최대 — 상태·클로저·이펙트·성능 이해 전부가 이 위에 선다.
- **M2 "대입 = 즉시 반영":** 동기 흐름 모델. `setCount(c+1)` 직후 `count`는 여전히 옛 값 — setState는 대입이 아니라 다음 렌더의 예약이다.
- **M3 "상태는 제자리에서 고친다(UPDATE문 본능)":** 객체를 직접 변이하고 그대로 set → 참조가 같아 재렌더가 안 된다. React의 변경 감지 수단이 참조 동일성임을 모르는 상태.
- **M4 "이펙트 = 트리거(@EventListener, DB trigger)":** "X 바뀌면 Y 실행"의 만능 도구로 오용 → 파생값을 이펙트로 계산, 이펙트 체인, 무한 루프. 교정 모델: 이펙트는 React 바깥 시스템과의 동기화 전용, 파생값은 렌더 중 계산.
- **M5 "상태의 원본은 서버니까 클라이언트는 사본을 잠깐 든다":** fetch 결과를 일반 상태처럼 다룸 → 로딩·에러·경쟁 조건(늦게 도착한 이전 요청)·낡은 캐시가 전부 미스터리가 된다. **주의: 이 오개념은 useState를 배울 때 잠복하고 데이터 페칭에서야 발화한다** — 교정 시점을 배우는 시점이 아니라 발화하는 시점에 놓는다.

양성 전이(충돌 불필요, 오히려 지렛대): HTTP·캐싱 헤더 지식은 M5 교정의 자산이고, PK 개념은 리스트 key를, 순수 함수 개념은 렌더 순수성을 거저 준다. 프레임은 간섭만 지도화하지만, 안 간섭하는 자산을 오해로 착각해 충돌을 낭비하지 않도록 명시해 둔다.

## 문제 정의 / 목표

백엔드 개발자 1인이 8주 안에, 사내 React 코드베이스에 리뷰 통과 수준으로 기여할 수 있는 상태에 도달한다. 목표는 "React 문법을 안다"가 아니라 **"백엔드 모델이 내놓는 오답을 스스로 예측하고 기각할 수 있다"**이다.

## 명시적 가정 목록

- **가정 1 (전체가 걸림): 학습자는 백엔드 습관이 깊게 굳었고 프론트 프레임워크 경험이 없다.** 틀리면 오개념 지도 자체를 교체해야 한다 — 이 플랜의 뼈대가 무효. 그래서 0주차 관문으로 검증한다.
- 가정 2: 주 7시간(평일 저녁 3회 + 주말 1회) 확보 가능, 총 약 56시간. 틀리면 주차 수가 늘어날 뿐 관문 순서는 불변.
- 가정 3: JS 기본기(함수·배열 메서드·async/await)는 Node 접촉 등으로 보유. 틀리면 선행 1~2주 추가.
- 가정 4: 목적은 실무 투입. 틀리면(면접 대비 등) 봉인 해제 조건이 달라진다 — 면접은 봉인 항목의 어휘를 요구하므로.

## 접근 및 단계

**층위 선언: 이 플랜의 진도 단위는 주차가 아니라 오개념의 사망이다.** 주차는 용량 예산일 뿐, 다음 단계로 가는 관문은 사망 확인서다. **사망의 조작적 정의:** 변형 충돌 과제에서 (a) 코드 실행 전 예측이 정답이고 (b) 구모델이 내놓을 오답과 그 추론을 재현 설명할 수 있음 — 둘 다 충족 시 3줄 확인서(오개념/오답 재현/교정 문장) 작성.

모든 교정 주차의 과제 형식은 **예측 → 충돌 → 재구성**으로 통일한다: 실행 전 2~3문장 예측을 쓰고, 구모델이 오답을 내도록 설계된 코드를 실행하고, 교정 모델을 자기 언어로 재작성한다. 설명식 강의는 오개념을 못 죽인다 — 충돌만이 죽인다.

- **0주(관문):** 전제조건 검사. → 검증: 문진 통과 여부로 플랜 유효성 자체를 판정.
- **1주:** 환경(Vite)·JSX·컴포넌트·props(비충돌 화물) + **M1 충돌**(지역 변수 카운터 vs useState — 클릭 시 값 예측). → 검증: M1 변형 과제 사망 판정.
- **2주:** 이벤트·폼(화물) + **M2 충돌**(`setCount(count+1)` 3연속 호출의 결과 예측). → 검증: M2 사망 확인서.
- **3주:** 리스트·key(화물) + **M3 충돌**(배열 push 후 set — 화면이 안 바뀌는 이유를 사전 예측). → 검증: M3 사망 확인서.
- **4주:** **M4 충돌**(이펙트 체인 코드의 렌더 횟수 예측, 무한 루프 재현) + 파생값을 렌더 중 계산으로 옮기는 재구성. → 검증: M4 사망 확인서.
- **5주:** **M5 충돌** — 서버 상태 라이브러리 없이 fetch를 손코딩하며 로딩·에러·경쟁 조건을 직접 맞는다(빠른 탭 전환 시 어느 응답이 화면에 남는지 예측). → 검증: M5 사망 확인서.
- **6주(재귀 적용 — 과교정 회수):** 교정은 2차 오개념을 낳는다. 예정된 2건에 2차 충돌: M2′ "파생값까지 전부 state로"(카운트에서 파생되는 짝수 여부를 state로 두면 무엇이 어긋나는가), M4′ "이펙트 공포/만능의 잔재". + 상태 끌어올리기·Context(봉인 심사의 재료). → 검증: 2차 확인서 2건.
- **7~8주(사이 공간 투입):** 오개념은 연습 맥락(교정 모델이 점화된 상태)이 아니라 자기 코드 맥락에서 재발한다. 외부 API 연동 CRUD SPA를 봉인 위반 0으로 완성하고, M1~M5를 체크리스트로 한 코드 리뷰를 받는다. → 검증: 리뷰에서 M1~M5 관련 지적 건수.

**예산 규칙(판단의 사전 룰화):** 오개념 1건당 예산 1주. 안 죽으면 1회만 이월하고, 이월 2회 발생 시 꼬리(7~8주 프로젝트의 범위)를 절삭한다 — 충돌 과제를 압축하지 않는다(압축은 설명식 회귀다). **스킵 게이트(기본값 있는 관문):** 첫 충돌에서 예측이 처음부터 맞으면 그 오개념은 이 학습자에게 없을 수 있다 — 기본 동작: 변형 1개로 확인 후 조기 통과, 남은 예산은 다음 관문으로 이월.

## 봉인 목록 — 의도적으로 아직 안 만지는 것들

봉인의 근거는 난이도가 아니라 **은폐성**(오개념이 죽기 전에 도입하면 그 오개념을 가려버림)이다.

- **상태관리 라이브러리(Redux/Zustand):** 해제 조건 — 자기 프로젝트에서 3단계+ prop drilling을 겪고, Context로 먼저 풀어본 뒤 "Context가 못 푸는 문제"를 한 문단으로 설명 가능할 때.
- **서버 상태 라이브러리(React Query/SWR):** 해제 조건 — 5주차 손코딩에서 경쟁 조건·낡은 데이터를 직접 맞은 뒤. 조기 도입은 M5를 은폐한다.
- **메모화 3종(useMemo/useCallback/React.memo):** 해제 조건 — Profiler로 실측된 렌더 병목 존재. 그 전엔 M3(참조) 이해를 카고컬트로 대체할 뿐.
- **Next.js/SSR/서버 컴포넌트:** 해제 조건 — 8주 완주 + 클라이언트 렌더 모델을 남에게 설명 가능. M1이 죽기 전의 SSR은 렌더 모델을 이중으로 오염시킨다.

**봉인 파기 유혹 룰:** 튜토리얼·AI가 봉인 항목을 권할 때마다 "왜 필요해 보였는지" 1줄 기록하고 무시한다. 같은 항목 3회 누적 시 해제 심사 — 유혹의 빈도 자체가 해제 조건의 데이터다.

## 대안과 기각 사유

- **정석 커리큘럼(공식 튜토리얼 난이도순 완주):** 기각 — 이 프레임의 실패 모드 그 자체. 오개념은 난이도 축이 아니라 발화 시점 축에 산다. 부활 조건: 0주차 관문 실패(선행 습관이 없다고 판명).
- **프로젝트 우선(일단 만들며 배우기):** 기각 — 충돌이 설계되지 않으면 오개념은 "돌아가는 코드" 밑에 잠복한 채 실무까지 따라온다. 부활 조건: 8주 완주 후의 다음 단계로는 최선.
- **강의 완주형(인강 40시간):** 기각 — 설명은 예측을 강제하지 않으므로 구모델과 충돌하지 않는다. 부활 조건: 없음(보조 자료로만).

## 리스크 & 완화

- **R1 오개념이 예산 내에 안 죽음:** 예산 규칙이 처리(이월 1회 → 꼬리 절삭). 정체가 반복되면 가정 3(JS 기본기) 위반을 의심하고 0주차로 회귀.
- **R2 과교정(2차 오개념):** 6주차 회수 패스로 예정 처리. 예정에 없는 과교정 발견 시 확인서에 추가 등록.
- **R3 기록 부담이 학습을 잠식:** 기록 총량 주 15분 상한 — 초과 시 기록 항목을 줄이지, 충돌 과제를 줄이지 않는다.
- **R4 비충돌 화물의 누락:** 오개념 뼈대는 JSX·폼·리스트·라우팅·TS 같은 무간섭 지식을 다룰 언어가 없다 — 화물 목록을 주차에 명시 부착했고, 7~8주 프로젝트가 누락 검출기 역할을 한다.
- **R5 검증자 부재(혼자 학습):** 사망 판정의 변형 과제를 스스로 고르면 판정이 물러진다 — 변형 과제는 사수 또는 AI에게 "이 오개념을 시험할 처음 보는 문제"로 생성시켜 출제자와 응시자를 분리한다.

## "완료"의 판정 기준

- **실측 문장: 8주 종료 시, 제3자(사수 또는 AI)가 선별한 처음 보는 충돌 문제 5개(오개념당 1개)를 코드 실행 전에 예측해 4개 이상 적중하고, 각 문제에서 구모델의 오답을 재현 설명한다.**
- 사망 확인서 7장(M1~M5 + 2차 2건)이 전부 존재한다.
- 외부 API 연동 CRUD SPA가 봉인 위반 0으로 완성되고, M1~M5 체크리스트 리뷰에서 지적 0건.
- 봉인 목록 4건 각각에 대해 "지금 해제 조건이 충족됐는가"를 예/아니오로 답하고 근거를 댈 수 있다 — 해제가 아니라 해제 판단 능력이 완료 조건이다.

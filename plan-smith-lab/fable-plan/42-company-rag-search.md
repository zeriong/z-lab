# 전사 문서 검색(RAG) 시스템 도입
- 추론 프레임: envelope
- 한 줄 요약: 자릿수 계산이 아키텍처 등급을 정한다. 현 숫자에서 자체 호스팅은 손익분기의 1/20~1/200이고, 벡터 총량은 "서비스"가 아니라 "파일"의 크기다 — 색인은 야간에 컴파일하는 산출물로, 등급 전환은 청구서 숫자에 걸린 게이트로 만든다.

## 프레임이 요구하는 시작점 — 자릿수 계산 (이 절에서 아키텍처 명사 금지 — 벡터·임베딩·청크 같은 저장·처리 단위 명사는 불가피하여 허용)

### 0. 숫자보다 먼저, 숫자의 의미
- **문서량** = 검색 대상에 포함할 문서 최종본의 총 토큰 수. 버전 이력·중복 첨부는 제외.
- **질의 1건** = 사용자 1인의 질문 1회 + 그에 대한 생성 응답 1회.
- **갱신 1건** = 색인 반영이 필요한 사건: 문서의 생성·수정·삭제, 그리고 **권한 변경**(내용이 안 바뀌어도 "누가 볼 수 있는가"가 바뀌면 색인 관점에선 갱신이다).
- **자체 호스팅의 비용** = 장비비 + **운영 인력의 시간**. 인건비를 빼면 장부가 안 맞는다 — 이 계산의 숨은 통화.

### 1. 네 개의 자릿수
- **문서량:** 임직원 ~300명, 5년 축적, 문서 ~5×10⁴건 × 평균 2×10³토큰 ≈ **총 10⁸ 토큰**. 500토큰 청크로 ~2×10⁵개, 벡터 총량 **~1.5GB**.
- **임베딩 비용:** 10⁸ 토큰 × 위탁 처리 단가(10⁻¹ $/10⁶토큰 자릿수) ≈ **최초 1회 10¹ $**. 전체를 다시 돌려도 커피값.
- **질의량:** 300명 중 일 사용 20% × 5질의 ≈ **3×10² 질의/일**. 질의당 생성 비용 ~10⁻² $ → **월 10² $**.
- **갱신량:** 주간 변경률 1~2% ≈ **10² 문서/일** = 토큰 2×10⁵/일 → 재임베딩 비용 일 10⁻² $.

### 2. 읽기 단위의 선택 — 같은 숫자를 무엇으로 읽는가
- 질의량을 **율(rate)**로 읽으면 0.004 qps — 동시성·확장 설계가 성립할 이유가 없다. 옳은 독법은 **월 청구서**: 10² $/월.
- 문서량은 율로 읽을 이유가 없다. **볼륨**으로 읽으면 1.5GB — 상시 가동 서비스가 아니라 **파일 하나의 크기**다.
- 갱신을 율로 읽으면 상시 증분 처리 설비가 소환되지만, **일일 볼륨**(10²건)으로 읽으면 하루 1회 배치, 수 분이면 끝난다.

### 3. 등급 판정 — 나눗셈이 결정한다
자체 호스팅의 고정비 하한 = 추론 장비 10³ $/월 + 운영 0.5인 이상(10³~10⁴ $/월) ≈ **월 10³~10⁴ $**. 사용량 과금의 변동비 = 질의량 × 10⁻² $/질의. 손익분기 질의량 = (10³~10⁴ $) ÷ (10⁻² $/질의) ≈ **10⁵~10⁶ 질의/월 ≈ 5×10³~5×10⁴ 질의/일**. 현 예상 3×10²/일은 손익분기의 **1/20~1/200**. 판정은 계산이 내린다: **상용 사용량 과금 등급**. 취향도 보안 정서도 아닌 나눗셈의 결과다.

## 층위 이동 — 색인은 서비스가 아니라 빌드 산출물
볼륨 독법(벡터 1.5GB, 갱신 10²건/일)이 허락하는 재정의: 벡터 색인을 상시 가동 데이터베이스로 세우지 않는다. **콘텐츠 해시로 임베딩을 캐시하고, 매일 밤 코퍼스 전체를 색인 파일 하나로 컴파일해, 질의 서버는 그 파일을 읽기 전용으로 적재**한다. 색인은 상태를 가진 서비스에서 재현 가능한 산출물로 강등된다 — 장애 복구는 "재컴파일", 롤백은 "어제 파일".

산출물 방식 자체에 봉투를 재적용(프레임의 재귀): 야간 전체 재컴파일 비용 = 캐시 미스분 재임베딩(일 10⁻² $) + 2×10⁵ 벡터 색인 빌드(분 단위). 매일 전체를 다시 짓는 사치가 이 자릿수에선 공짜다. 모델 교체 시 전량 재임베딩도 10¹ $ — 갈아타기 비용까지 봉투 안이다. 단, 이 재정의는 자릿수가 2개 커지면 무효(민감도 표 1행).

## 문제 정의 / 목표
임직원이 사내 문서(위키·정책·회의록·규정)를 자연어로 질문하고 근거 문서와 함께 답을 받는 검색 시스템. 목표는 최신 아키텍처가 아니라 **"현 자릿수에 맞는 최소 등급 + 자릿수가 바뀔 때의 전환 규칙"**.

## 명시적 가정 목록
- **가정 1 (전체가 걸림): 코퍼스의 절대 다수(≥90%)는 외부 처리 위탁 계약 하에 상용 API로 전송 가능하며, 전송 불가 문서를 도입 전에 식별·분리할 수 있다.** → 틀리면(분류 불가이거나 과반이 전송 금지) 비용 계산과 무관하게 상용 등급 자체가 무효 — 이 플랜의 등급 판정 전체가 뒤집힌다.
- 가정 2: 문서량 총 10⁸ 토큰 자릿수. → 틀리면(문서량이 10¹⁰ 자릿수면) 단일 노드 산출물 방식 붕괴, 색인 계층 분리 등급으로.
- 가정 3: 질의량 초기 10²/일 자릿수. → 틀리면(도입 직후 10⁴/일이면) 손익분기 재계산을 도입 시점으로 앞당긴다.
- 가정 4: 신선도 요구는 24시간(야간 배치 1회로 충족). → 틀리면(시간 단위 요구면) 컴파일 주기 단축(한계 ~1시간), 분 단위면 증분 파이프라인으로 등급 상승.

## 접근 및 단계
1. **코퍼스 실측:** 문서 소스 전수 크롤링, 문서 수·토큰 수·전송 불가 문서 비율 측정. → 검증: 실측 자릿수가 가정 1·2와 대조되어 민감도 표에 기입됨.
2. **봉투 재계산 + 등급 확정:** 실측값으로 3절 나눗셈 재실행. → 검증: 등급 판정문 1줄이 실측 숫자에서 직접 유도됨.
3. **파일럿(1개 부서, 문서 10³~10⁴):** 야간 컴파일 산출물 + 질의 경로 가동. → 검증: 부서가 만든 30문항 정답 세트에서 근거 문서 top-5 포함률 ≥70%, 질의당 실측 비용이 봉투의 ±1 자릿수 내.
4. **권한 필터(단위 사이의 틈):** 임베딩은 권한을 모른다 — 검색과 생성 사이가 유출 지점이다. 질의 시점에 질의자의 열람 권한으로 후보를 사후 필터한다. 10⁵ 청크 자릿수에선 사후 필터가 공짜다. → 검증: 권한 없는 계정으로 민감 주제 질문 20건, 근거·본문 유출 0건.
5. **전사 확장 + 게이트 계기판:** 민감도 표의 각 행을 대시보드 숫자로 연결. → 검증: 문서 수정 후 24시간 내 검색 반영, 첫 달 청구서가 10² $ 자릿수.

## 재설계 트리거 (지금 약속하는 숫자)
- 월 청구서 **$2,000 초과 2개월 연속** → 자체 호스팅 손익분기 재계산 실행. 기본값: 재계산 없이는 이전 금지(패닉 이전 차단).
- 벡터 총량이 **단일 노드 RAM의 50% 초과** → 색인 계층 분리 검토 개시.
- 신선도 SLO(24h) 위반 **주 3회** → 컴파일 주기 단축, 1시간 벽에 닿으면 증분화 설계 착수.
- 질의량 **10¹/일 미만 3개월 연속** → 철수 판정 회의. 안 쓰는 시스템도 트리거의 대상이다.

## 민감도 표 — 어느 가정이, 어느 지점에서 결론을 뒤집는가

| 축 | 현 자릿수 | 뒤집히는 지점 | 뒤집히면 |
|---|---|---|---|
| 문서량(토큰) | 10⁸ | 10¹⁰ (100배) | 단일 노드 산출물 → 색인 계층 분리 |
| 질의량 | 3×10²/일 | 5×10³~5×10⁴/일 | 자체 호스팅 손익분기 재계산 |
| 갱신량/신선도 | 10²건/일 · 24h | 10⁴건/일 또는 <1h | 야간 배치 → 증분 파이프라인 |
| 질의당 단가 | 10⁻² $ | 5배 인상 | 손익분기 1/5 접근 — 재계산 |
| 전송 금지 문서 비율 | <10% (가정 1) | 과반 (숫자 아닌 규정) | 비용 무관, 해당 파티션 자체 호스팅 강제 |

마지막 행은 비수치 축이다 — 봉투 계산은 자신이 계산할 수 없는 강제 조항을 표에 명시해야 정직하다.

## 대안과 기각 사유
- **자체 호스팅 전면(오픈 가중치 생성 모델 + 자체 임베딩 + 벡터 DB 클러스터):** 기각 — 고정비 10³~10⁴ $/월 vs 변동비 10² $/월, 자릿수 1~2개 차이. 부활 조건: 질의량이 손익분기 구간 진입, 또는 민감도 표 5행 발동.
- **상시 가동 벡터 DB(관리형 포함):** 기각 — 1.5GB는 서비스가 아니라 파일의 크기이고, 상태 있는 서비스는 운영 통화(백업·버전 관리·장애 대응)를 따로 청구한다. 부활 조건: RAM 50% 트리거 또는 시간 단위 증분 갱신 요구.
- **RAG 없이 롱컨텍스트 전체 주입:** 기각 — 질의당 10⁸ 토큰은 컨텍스트 창의 10²~10³배, 성립한다 해도 질의당 비용이 10² $ 자릿수. 나눗셈이 즉사 판정.

## 리스크 & 완화
- **R1: 봉투 계산의 사각 — 품질.** 자릿수는 비용을 판정하지만 "답이 맞는가"는 못 판정한다. 도입 실패의 최빈 원인은 비용이 아니라 쓸모없는 답이다. → 3단계 품질 게이트(top-5 ≥70%)를 확장의 선행 조건으로. 이 70%는 계산값이 아니라 잠정 관행값 — 파일럿 후 재조정 대상임을 명시.
- **R2: 권한 유출.** → 4단계 사후 필터 + 유출 0건 게이트. "권한 변경 = 갱신 이벤트"를 색인 파이프라인의 계약으로 명시.
- **R3: 질의량 예측 오류(급증/무사용).** → 양방향 모두 트리거로 규칙화됨 — 급증은 재계산, 무사용은 철수 판정.
- **R4: 단가 변동·모델 단종.** → 민감도 표 4행이 감시. 임베딩 캐시가 콘텐츠 해시 기반이라 모델 교체 = 전량 재임베딩 10¹ $ — 이주 비용도 봉투 안.

## "완료"의 판정 기준
- **전사 오픈 후 첫 달: 무작위 표본 50문항에서 근거 문서 top-5 포함률 ≥70%, 권한 밖 문서 노출 0건, 월 청구서 10² $ 자릿수 — 세 숫자를 실측으로 확인한다.**
- 민감도 표의 모든 행에 실측치가 기입되고, 재설계 트리거 4개가 대시보드 숫자에 연결되어 있다.
- 임의 문서 10건을 수정해 24시간 내 검색 반영을 실측.

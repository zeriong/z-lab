# 오픈소스 라이브러리 — 날짜 범위 겹침 계산 유틸
- 추론 프레임: API-first
- 한 줄 요약: 사용자가 칠 코드 10줄을 헌법으로 먼저 쓰고, 그 코드가 참이 되도록만 내부를 만든다. 클래스가 아니라 순수 함수를 계약으로 택했다.

## 프레임이 요구하는 시작점 — 사용자 코드 10줄 (이 문서의 헌법)

```js
import { overlaps, intersect, subtract, merge, gapBetween } from 'date-range-kit'

const vacation = { start: '2026-07-01', end: '2026-07-15' }
const oncall   = { start: '2026-07-10', end: '2026-07-20' }

overlaps(vacation, oncall)                 // => true
intersect(vacation, oncall)                // => { start: '2026-07-10', end: '2026-07-15' }
subtract(vacation, oncall)                 // => [{ start: '2026-07-01', end: '2026-07-10' }]
merge([vacation, oncall])                  // => [{ start: '2026-07-01', end: '2026-07-20' }]
gapBetween(vacation, oncall)               // => null            (겹치면 간격 없음)
overlaps({ start: '2026-07-01', end: '2026-07-01' }, vacation)  // => false  (빈 구간)
```

이 10줄이 명세다. 아래 전부는 이 줄들을 참으로 만들기 위한 종속 결정이다.

## 예시가 강제한 결정들 (한 줄씩 심문)

- **`import { overlaps, ... }` — 클래스가 아니라 순수 함수.** 예시 첫 줄이 이미 가장 큰 설계를 결정했다. 사용자는 `new DateRange(...)`로 자기 데이터를 라이브러리 타입으로 감싸고 싶어하지 않는다 — 그들의 범위 데이터는 이미 DB 행, API 응답 안에 `{start, end}` POJO로 존재한다. 함수형 API는 (1) 사용자 데이터를 무변환으로 받고, (2) 트리 셰이킹되며, (3) 반환값도 POJO라 그대로 직렬화된다. **"사용자가 이미 가진 데이터 모양을 계약으로 승격"**이 이 API의 정체성.
- **`intersect(...)` 반환값이 `{start: '07-10', end: '07-15'}`** → 경계 규약이 **반열림 [start, end)** 로 강제된다. 닫힌 구간이면 `subtract`의 기대값(`07-01~07-10`)에서 7월 10일이 양쪽에 속해 모순. 인접 구간(`end === next.start`)이 "안 겹침"으로 떨어지는 것도 이 규약의 파생 — 캘린더·예약 도메인의 실사용 직관과 일치.
- **`subtract`가 배열 반환** → 결과가 0·1·2조각일 수 있음을 타입이 선언. 가운데를 빼면 2조각이 나오는 현실을 API가 숨기지 않는다.
- **빈 구간(`start === end`)은 무엇과도 안 겹침** → 마지막 줄이 에지 케이스의 답을 계약에 박제. 반열림 규약의 논리적 귀결([x, x) = ∅)이기도 하다 — 규약이 일관되면 에지 케이스가 "결정"이 아니라 "도출"이 된다.
- **`gapBetween`이 `null` 반환** → 겹침/인접 시 간격 없음을 예외가 아니라 값으로. 호출부의 `if (gap)` 한 줄을 위한 결정.
- **입력이 ISO 문자열** → 문자열 사전순 == 시간순이라는 ISO-8601의 성질 덕에, 내부에서 Date 객체·타임존 해석 없이 **문자열 비교만으로** 전 연산이 성립한다. 이 관찰이 구현 전체를 지배한다(아래).

## 문제 정의 / 목표

날짜 범위의 겹침·교차·차집합·병합·간격 계산. 목표는 **문서를 안 읽은 사용자가 반환값을 정확히 예측**하는 것. 기능 수가 아니라 규약의 일관성이 제품.

## 명시적 가정 목록 (틀렸을 경우의 영향 포함)

- **가정 1:** 입력은 ISO-8601 문자열(`YYYY-MM-DD` 또는 datetime)이고 두 입력의 정밀도가 같다. → 틀리면(혼합 정밀도, 타 포맷) 문자열 비교 전략 붕괴 → 파싱 층 필요, "의존성 0·해석 0" 정체성 훼손.
- **가정 2:** 타임존 해석은 호출자 책임. 라이브러리는 순서만 본다. → 틀리면 범위 폭증. 경계는 여기 긋는다.
- **가정 3:** 반열림 규약이 대상 도메인(예약·일정)의 다수 직관과 일치. → 틀리면 계약 자체 재협상(마이너 버전으로 못 바꿈 — 이 규약은 영구 계약).

## 접근 및 단계

1. **규약 문서를 코드보다 먼저 커밋:** 반열림, 빈 구간, null 의미론을 README 최상단에. 예시 10줄을 doctest로 등록 — **예시가 CI에서 실행되는 명세**가 된다.
2. **원자 연산 1개:** `overlaps(a, b) = a.start < b.end && b.start < a.end` (문자열 비교). 이 부등식이 라이브러리의 전부다.
3. **파생 연산:** `intersect` = max/min 조합, 빈 결과는 null. `subtract` = 좌/우 잔여 조각 수집. `gapBetween` = 비겹침 시 사이 구간.
4. **`merge`:** start 정렬 후 단일 패스 스윕. 인접(`end === start`)도 병합 — 반열림 규약의 파생임을 테스트 이름에 명시.
5. **속성 기반 테스트:** "subtract 결과들과 intersect를 다시 merge하면 원본" 같은 대수적 불변식을 무작위 입력으로 검증. 규약의 일관성을 수학으로 지키는 층.

## 대안과 기각 사유

- **클래스 API (`new DateRange(...)`):** 기각. 사용자 데이터가 이미 POJO인 세계에서 감싸기/풀기 세금을 물린다. 메서드 체이닝의 미학은 라이브러리 저자의 것이지 사용자의 것이 아니다.
- **닫힌 구간 [start, end]:** 기각. 예시의 subtract 기대값과 모순, 인접 판정에 특수 규칙 필요.
- **epoch ms 정수 입력:** 기각. 예시를 그렇게 쓰면 사용자가 매 호출마다 변환해야 한다 — API-first의 관점에서 "내부가 편한 타입"은 근거가 아니다. (내부 변환조차 불필요함을 ISO 문자열 비교가 보장.)
- **타임존/캘린더 연산 내장:** 기각. 가정 2. 겹침 계산과 시간대 해석은 다른 문제다.

## 리스크 & 완화

- **R1: 반열림 규약을 모르는 사용자의 off-by-one.** → README 첫 문장 + `gapBetween` 등 이름에 의미를 실음 + 빈 구간 예시를 10줄 안에 포함(이미 함).
- **R2: 잘못된 입력(start > end, 비 ISO 문자열).** → 각 함수 진입 시 검증하고 `RangeError`. 조용한 오답이 최악이므로 fail-fast.
- **R3: 혼합 정밀도 입력(`'2026-07-01'` vs `'2026-07-01T09:00'`).** → 문자열 비교가 조용히 틀리는 유일한 구멍. 정밀도 불일치 감지 시 throw(가정 1의 방어선).

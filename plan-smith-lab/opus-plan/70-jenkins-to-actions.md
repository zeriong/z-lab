# 수년 된 Jenkins에서 GitHub Actions로의 점진 이전
- 추론 프레임: min-diff
- 한 줄 요약: 파이프라인 1개를 원자 단위로 삼아 병행 운영→검증→전환하고, 각 단계는 한 문장 롤백이 가능해야 하며, 공유 축(시크릿·러너)과 개선은 전부 최후로 미룬다.

## 프레임이 요구하는 시작점 — 한 문장 롤백이 되는 원자 단위

min-diff의 시작점: 모든 단계는 독립 배포·독립 롤백이 되어야 한다. 롤백을 한 문장으로 못 쓰면 그 단계는 너무 크다 → 쪼갠 뒤 돌아온다. 그래서 이 플랜은 "무엇을 옮길까"가 아니라 "가장 작은, 되돌릴 수 있는 단위가 무엇인가"에서 시작한다.

- **원자 단위 = 파이프라인 1개**(Jenkins 잡 1개 / 레포 1개의 빌드). 전체 CI를 한 번에 옮기지 않는다.
- 각 파이프라인 이전은 세 문턱을 통과한다: 병행(shadow) → 검증 → 전환(cutover). 문턱마다 **롤백 문장을 먼저** 쓴다. 이게 승인 게이트다(선언이 아니라 게이트).
- 게이트 예시: "이 파이프라인의 Actions 트리거를 끄고 Jenkins 잡을 다시 켠다." 이 한 문장이 안 써지면 그 단계는 크기 초과다. 예컨대 파이프라인이 되돌릴 수 없는 부수효과(프로덕션 배포·레지스트리 퍼블리시)를 포함하면 shadow가 불가능해 롤백 문장이 성립하지 않는다 → 부수효과를 dry-run/스크래치로 분리해 롤백 가능한 크기로 쪼갠 뒤 복귀한다.

## 문제 정의 / 목표

수년간 누적된 Jenkins 파이프라인 다수를 GitHub Actions로 옮긴다. 목표는 "더 좋은 CI를 새로 짓기"가 아니라 **동작을 보존한 채 실행 주체만 안전하게 바꾸기**다. 개선(속도·구조·시크릿 현대화)은 이전의 목표가 아니며, 이전이 끝난 뒤 별도 과제로 분리한다.

## 명시적 가정 (틀렸을 경우의 영향)

- **가정 A (전체가 걸림): 각 파이프라인을 부수효과 없이 shadow로 병행 실행할 수 있다.** Actions가 실제 배포·퍼블리시를 하지 않고도(dry-run/스크래치 네임스페이스) Jenkins와의 동등성을 검증할 수 있다는 전제다. 틀리면(모든 빌드가 공유 레지스트리 퍼블리시·프로덕션 배포 같은 비가역 부수효과를 강제) "병행→검증→전환" 척추가 통째로 붕괴한다 → 이 경우 min-diff가 아니라 다른 프레임(예: safety-net)이 필요하다.
- 가정 B: Jenkins 잡의 동작을 config/스크립트에서 충분히 재현할 수 있다. 틀리면(암묵지·수동 설정 다수) 이식이 리버스 엔지니어링이 되어 느려지지만 min-diff 골격 자체는 유지된다.
- 가정 C: 병행 기간 동안 러너가 두 시스템을 동시에 감당할 용량이 있다. 틀리면 shadow 실행이 경합한다 → shadow를 스태거/스로틀하거나 임시 러너를 증설한다.
- 가정 D: 조직이 수개월간 두 시스템 공존을 견딜 수 있다. 틀리면(빨리 끝내라는 압력) big-bang 유혹이 복귀한다 → 유혹 로그로 방어하되, 압력이 실재하면 대안 섹션의 부활 조건을 발동한다.
- 가정 E: 시크릿을 열거·복사할 수 있다. 틀리면 공유 축 이전(6단계)이 어려워진다.

## 접근 및 단계 (파이프라인 단위, 단계별 검증)

0. **인벤토리 & 순서 정하기.** 모든 Jenkins 잡을 목록화하고, 각 잡을 (부수효과 유무 × 블라스트 반경)으로 분류한다. 첫 이전 대상은 부수효과 없고 대표성 있는 잡(리프 라이브러리 빌드), 마지막은 프로덕션 배포 잡. 검증: 목록 개수가 실제 Jenkins 잡 개수와 일치.

각 파이프라인마다 1~4를 독립 반복한다:

1. **그대로 이식(port, 개선 금지).** Jenkins 잡의 동작을 Actions 워크플로로 1:1 재현한다. 못생긴 스텝도 그대로. 공유 축은 건드리지 않고 기존 것을 재사용한다(기존 self-hosted 러너에 Actions 러너를 추가 등록, 시크릿은 수동 복사). 검증: 브랜치에서 워크플로가 Jenkins와 같은 산출물을 낸다.
2. **병행(shadow).** 같은 트리거(같은 push/PR)에 Actions도 실행하되 authoritative는 여전히 Jenkins. Actions는 배포·퍼블리시를 하지 않는다(dry-run/스크래치). 검증(게이트): **연속 20회** 실행에서 산출물 바이트 동등 또는 스테이징 diff 없음(중간에 1회라도 불일치 시 카운터를 0으로 리셋하고 원인 교정 후 재시작). 롤백 문장: "Actions 워크플로 파일을 삭제한다"(shadow라 부수효과 0).
3. **전환(cutover).** Actions를 authoritative로 승격(실제 배포·퍼블리시), Jenkins 잡은 트리거만 끄고 잡 정의는 보존한다. 검증(게이트): **실제 커밋 10건**이 연속 green + 성공률 ≥ Jenkins 베이스라인(직전 30일 성공률) + 소요시간 ≤ 베이스라인 중앙값 ×1.2. 롤백 문장: "Actions 트리거를 끄고 Jenkins 잡 트리거를 켠다."
4. **소킹(soak).** **영업일 10일** 관찰. 이 기간 동안 롤백 경로를 유지한다. 이 잡의 Jenkins config는 shadow 시작 시점에 동결(드리프트 방지).

공유 축은 모든 파이프라인이 4까지 통과한 뒤에만 건드린다:

5. **러너 이전.** self-hosted → GitHub-hosted 또는 새 러너 풀로 교체. 블라스트 반경이 전 파이프라인이라 독립 롤백이 불가 → 그래서 최후. 검증: 러너 교체 후 전 파이프라인 green.
6. **시크릿 정리.** Jenkins/Actions 이중 보관 상태를 Actions 단일로 정리(OIDC 등은 여기서 별도 개선 항목으로). 검증: Jenkins 크리덴셜 스토어를 비운 뒤 전 파이프라인 green.
7. **폐기(비가역).** 전 파이프라인이 소킹 통과 + Jenkins 서버 24h 정지 실험 통과 후에만 Jenkins 잡/서버를 삭제.

## 이전과 개선의 분리 (min-diff 필수)

이식 커밋은 동작 보존만 한다. 개선(캐시, 매트릭스 빌드, 병렬 테스트, OIDC, 빠른 러너, 모노리스 빌드 분리)은 이식 커밋에 절대 섞지 않는다. 개선은 해당 파이프라인이 cutover+soak를 통과한 뒤 별도 커밋/별도 PR로만 한다. 이유는 진단이다: shadow에서 산출물이 어긋날 때 원인이 "이전 실수"인지 "개선"인지 구분돼야 한다. 섞으면 이 구분이 불가능해진다.

## Rule of three (공유 워크플로 추출)

같은 스텝이 두 파이프라인에 중복돼도 아직 reusable workflow/composite action으로 추출하지 않는다. 세 번째 중복에서만 추출한다. 이전 초기에 성급히 공유 워크플로를 만들면 그게 또 하나의 공유 축(블라스트 반경)이 되어, "공유는 최후로"라는 원칙과 충돌한다.

## 유혹 로그 (기록하고 기각)

| 유혹 | 기각 사유 | 부활 조건 |
|---|---|---|
| self-hosted 러너를 GitHub-hosted로 갈아타며 이전 | 러너는 공유 축, 이전+개선 동시 = 진단 불가 | 6단계 |
| 이 참에 OIDC로 시크릿을 없애자 | 개선(동작 변경) | 6단계 이후 별도 백로그 |
| 빌드 20분인데 캐시/매트릭스로 줄이자 | 개선 | cutover 이후 별도 PR |
| 느린 모노리스 빌드를 이참에 쪼개자 | 동작 변경 + 개선 | 이전 완료 후 |
| 잡 50개니 스크립트로 한 번에 변환하자 | big-bang, 롤백 한 문장 불가 | 대안 섹션 참조 |

## 대안과 기각 사유 (부활 조건)

- **빅뱅 주말 전환**(전체 동결→일괄 변환→플립): 기각. 롤백이 한 문장이 안 됨("Jenkins 전체 복원"), 블라스트 반경=전체. 부활: 파이프라인이 1~2개로 사소하거나, 라이선스 만료 같은 강제 마감으로 병행 시간이 없을 때 — 단 Jenkins 전체 스냅샷 백업을 전제로.
- **이전+개선 동시**: 기각. 이식/개선 분리 위반, 버그 진단 불가. 부활: 없음(개선은 폐기 후 별도 백로그).
- **공유 축(러너/시크릿) 먼저 이전 후 그 위로 파이프라인 이주**: 기각. 공유 축 먼저 = 블라스트 반경 최대, 독립 롤백 불가, 파이프라인 하나 검증도 전에 타깃 인프라부터 지음. 부활: 기존 러너 하드웨어가 EOL로 강제 교체될 때 — 단 새 러너가 Jenkins도 계속 서빙하게 해서 파이프라인 이전 원자성을 유지.

## 리스크 & 완화

- R1 shadow 이중 부수효과(양쪽 다 배포): shadow의 부수효과 0을 보장(dry-run 플래그/스크래치 네임스페이스), 첫 shadow 전에 이 가드를 먼저 검증.
- R2 병행 무한 지속(두 시스템 영원히): 파이프라인별 cutover 마감일(= 해당 파이프라인의 shadow 게이트 통과일 + 영업일 10일; 초과 시 롤백 또는 에스컬레이션) + 동시 진행 WIP 상한(**shadow~soak 진행 중 파이프라인 최대 2개**) + 유혹 로그.
- R3 Jenkins 잡 드리프트(이전 중 원본이 바뀜): shadow 시작 시 해당 잡 config 동결, 불가피한 변경은 Actions에도 재적용.
- R4 러너 경합: shadow 스태거/스로틀(가정 C).
- R5 시크릿 이중 보관 표류: 임시 허용하되 6단계에서 일괄 정리, 유혹 로그에 기록해 잊지 않음.
- R6 비가역 폐기 조기 실행: 소킹 + 24h 정지 실험 통과 전 삭제 금지(게이트).

## "완료"의 판정 기준

- (파이프라인별, 실측) 소킹 중인 임의의 이전 파이프라인에서 롤백 문장을 실제로 실행 — Actions 트리거를 끄고 Jenkins 잡을 켠다 — 했을 때 실제 커밋이 여전히 같은 산출물/배포를 낸다. 롤백 경로가 살아있음을 실측.
- (전체, 실측) 모든 파이프라인이 Actions authoritative로 영업일 10일 이상 성공률 ≥ Jenkins 베이스라인(직전 30일 성공률). 그 뒤 **Jenkins 서버를 24시간 정지시켜도 이전 완료된 어떤 파이프라인도 실패하지 않는다** — 통과 시에만 폐기.
- (분리 검증) git 로그에 "이식+개선"을 동시에 한 커밋이 하나도 없다(로그 검사로 확인 가능).
- (프레임 실재) 유혹 로그에 기록·기각된 "이 참에" 항목이 1개 이상.

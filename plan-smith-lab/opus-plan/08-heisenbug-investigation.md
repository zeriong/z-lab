# "가끔, 특정 사용자에게만, 재현 불가" 버그 조사 플랜
- 추론 프레임: 가설-실험 사이클
- 한 줄 요약: 가설 3개 이상을 세우고, 각각을 기각/채택할 최소 실험을 설계한 뒤, 관찰 → 가설 갱신의 루프로 좁혀간다.

## 프레임이 요구하는 시작점 — 관찰된 사실만 먼저

가설을 세우기 전에, **추측이 섞이지 않은 관찰 사실**만 분리한다. 재현 불가 버그의 가장 흔한 실패는 "재현 안 됨"에 좌절해 추측으로 코드를 고치는 것이므로, 사실/추측의 경계를 먼저 긋는다.

관찰된 사실(가정된 것 아님):
- (F1) "가끔" — 상시 아님 → 상태·타이밍·데이터 의존 가능성.
- (F2) "특정 사용자에게만" → 그 사용자 집합의 공통 속성이 원인 후보.
- (F3) "재현 불가" → 우리 환경에 그 사용자의 무언가(데이터/설정/클라이언트)가 없다.

이 세 사실이 아래 가설의 씨앗이다.

## 문제 정의 / 목표

간헐적·사용자 한정·재현 불가 버그의 원인을 규명한다. 목표는 "고친다"가 아니라 먼저 **"원인 가설을 관찰로 기각/채택 가능한 것으로 만든다"**. 재현 경로 확보가 1차 성공.

## 가설 (3개 이상) 과 각 기각/채택 실험

각 가설은 "이게 참이면 무엇이 관찰돼야 하는가"와 짝을 이룬다. 실험은 **최소 비용으로 참/거짓을 가르는** 것으로 설계한다.

- **H1 — 데이터 의존 (특정 사용자의 데이터 형태가 트리거).**
  근거: F2 + F3(우리 환경엔 그 데이터가 없음).
  기각/채택 실험: 문제 사용자군의 데이터 특성을 쿼리로 프로파일링(널값, 비정상 길이, 특수문자, 오래된 스키마 잔재). 해당 특성을 스테이징에 복제 → 재현되면 H1 채택.
  값싼 우선순위 근거: 실험 비용 낮고, 참이면 재현 경로가 바로 열림.

- **H2 — 경합 조건 (동시성/타이밍).**
  근거: F1("가끔") + 특정 사용자가 우연히 특정 사용 패턴(동시 요청, 빠른 연타).
  기각/채택 실험: 의심 경로에 상세 타이밍 로그·트레이스 추가 후, 발생 시점의 동시 요청 여부 확인. 부하 재현 시도. 로그에 경합 흔적 없으면 H2 기각.

- **H3 — 환경/클라이언트 차이 (특정 브라우저·앱버전·지역·타임존).**
  근거: F2("특정 사용자") — 사용자 자체가 아니라 그들의 환경이 공통일 수 있음.
  기각/채택 실험: 발생 사용자의 User-Agent·앱버전·타임존·지역 분포를 로그에서 집계. 한 축으로 쏠리면 H3 채택 후 그 환경 재현.

- **H4 — 상태 누적 (오래 켜둔 세션/캐시 오염).**
  근거: F1 + F3(짧은 테스트 세션에선 안 쌓임).
  기각/채택 실험: 장시간 세션 시뮬레이션, 캐시를 인위적으로 오염시켜 재현 시도.

## 명시적 가정 목록 (틀렸을 경우의 영향 포함)

- **가정 1:** 발생 시점을 사후에라도 로그/APM으로 특정할 수 있다. → 틀리면(관측성 부재) 첫 과제가 조사가 아니라 "관측성 심기"로 바뀜.
- **가정 2:** 문제 사용자를 식별해 그들의 속성을 조회할 수 있다(개인정보 범위 내). → 틀리면 H1·H3의 프로파일링이 막힘.
- **가정 3:** 버그가 단일 원인이다. → 틀리면(복합 원인) 단일 가설 채택 후에도 재현이 불안정할 수 있어 루프를 더 돌아야 함.

## 접근 및 단계 (관찰 → 가설 갱신 루프)

1. **관측성 확보:** 가정 1을 먼저 검증/보강. 로그·트레이스·에러 그룹핑이 없으면 여기부터. (실험의 전제조건)
2. **가설 병렬 프로파일링:** H1·H3는 기존 로그 집계만으로 값싸게 신호를 얻을 수 있으니 먼저. 신호가 한쪽으로 쏠리면 해당 가설로 집중.
3. **재현 경로 구축:** 신호가 가장 강한 가설의 조건을 스테이징에 복제. 재현 성공 = 1차 목표 달성.
4. **루프:** 실험 결과로 가설을 갱신. 기각된 가설은 버리고, 관찰이 새 가설을 낳으면 추가. 재현될 때까지 반복하되 **매 루프에서 "무엇을 새로 알았나"를 기록**.
5. **수정·검증:** 재현 경로가 생기면 수정은 그 경로를 빨간→초록으로 바꾸는 회귀 테스트로 잠금.

## 대안과 기각 사유

- **로그만 보고 바로 추정 수정(가설 없이 패치):** 기각. 재현 경로 없이 고치면 "고친 것 같음"만 남고 검증 불가. 재현 불가 버그에서 가장 흔한 실패.
- **가설 하나에 올인:** 기각. F1~F3가 여러 원인 유형을 동시에 지지하므로, 조기 수렴은 확증편향 위험. 최소 3가설 병렬이 프레임의 요구.

## 리스크 & 완화

- **R1: 어떤 가설도 재현을 못 만듦.** → 관측성을 더 심고(가정 1 강화) 발생 사례를 더 모아 새 사실을 확보. 루프를 닫지 말 것.
- **R2: 확증편향(선호 가설만 파고듦).** → 각 가설에 "기각 조건"을 실험 설계 시점에 미리 못 박아, 기각을 회피하지 못하게.

## "완료"의 판정 기준

- **재현 경로 확보**(제어된 환경에서 버그를 의도적으로 일으킬 수 있음) — 이것이 1차 완료.
- 채택된 가설이 실험 관찰로 뒷받침되고, 기각된 가설들은 "왜 기각됐는지"가 기록됨.
- 수정 후 재현 경로가 회귀 테스트로 잠겨, 같은 원인 재발이 CI에서 잡힘.

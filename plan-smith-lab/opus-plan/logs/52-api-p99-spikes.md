# 로그 — 52 API p99 스파이크 (hypothesis-loop) [Opus]

## 진입 시 사고
이 과제를 보자마자 튀어나온 관성은 둘이었다. 하나는 "p99 스파이크 = GC" 반사 — 편애 가설 하나를 붙잡고 GC 튜닝부터 파는 조기 수렴. 다른 하나는 원인을 잡기도 전에 "고치기"로 직행하는 습관 — 타임아웃을 낮추고 재시도를 끄며 손을 대는 것. hypothesis-loop는 둘 다 정면으로 막았다. 3개 이상 가설을 기각 실험과 짝지어 병렬로 세우라는 요구가 GC 편애를 금지했고, "1차 목표는 fix가 아니라 재현 레시피"라는 규정이 조기 완화 충동을 금지했다. 그래서 문서의 무게중심을 "원인 후보 5개 + 사전 기각조건 + 좌표 전환"에 두고, 완화는 대안 섹션으로 밀어냈다. 관성대로 갔으면 GC 로그만 들여다보다 "겹치는 것 같다"로 성급히 결론냈을 것이다.

## 프레임이 만든 결정적 순간
프레임 없이는 절대 안 나왔을 결정은 좌표계 의심에서 나온 "글로벌 집계만으로 기각 금지" 규칙이다. 관성대로였다면 "전 구간 p99가 매끈하니 마이크로버스트도 아니고 특정 테넌트 문제도 아니다"라고 여러 가설을 한 번에 조기 기각했을 것이다. 좌표 전환을 강제하니, 같은 데이터가 1초 버킷·호스트별·테넌트별로 쪼개질 때에야 군집이 드러날 수 있다는 게 보였고, 그래서 "기각 조건은 그 가설에 맞는 좌표에서 평가한다"는 별도 규칙이 필요해졌다. 두 번째 결정은 실험 비용을 원장의 1급 필드로 둔 것이다. 이 덕분에 H1·H2(기존 로그·메트릭만 보면 되는 싼 것)를 병렬 1순위로, H3·H4·H5(재집계·태깅이 필요한 중간 비용)를 뒤로 자연스럽게 정렬했다. 비용 필드가 없었으면 그냥 GC부터 순서 없이 팠을 것이다.

## 프레임 이탈 / 관성 회귀 지점
정직하게 셋을 자백한다.
- "접근 및 단계"가 결국 1→6 선형 절차 리스트로 흘렀다. hypothesis-loop는 관측→갱신의 루프인데, 6단계에 "루프"라고 적어두긴 했으나 문서의 겉모습은 여전히 상승하는 선형 체크리스트다. 이건 04 로그에서 지적한 것과 똑같은 관성 회귀다.
- 재현 레시피를 done의 핵심으로 못 박으면서 "재현 가능하다"를 가정 1로 깔았는데, 간헐적·비결정적 스파이크는 원리적으로 재현이 안 될 수 있다. 가정 1을 굵게 표시해 방어는 했지만, 재현 실패 시의 분기(통계적 SLO 관리로의 전환)를 한 줄로만 처리하고 깊게 파지 않았다. 무난한 낙관 서사(재현→수정)로 수렴한 혐의가 있다.
- 주어진 5개 가설을 그대로 받아썼다. 커버리지 스윕에서 락 경합·캐시 스탬피드·DNS/TLS·스레드풀 포화를 "미등록 후보"로 이름만 얹고 원장에 정식 편입하지 않았다. 망라 우선을 표방하면서 5개 틀에 안주한 것이라, 코버리지가 반쪽이다.

## 방향성
증상을 고치기 전에, 좌표를 바꿔가며 원인을 "재현 가능한 레시피"로 굳힌다.

## 자가 평가
프레임 충실도: 상. 세 필수 부품(사전 등록 기각조건, 1급 비용 원장, 좌표계 의심)이 장식이 아니라 실제로 판정 순서와 기각 규칙을 바꿨다. 흠은 둘: 접근 섹션이 루프를 선언만 하고 형태는 선형으로 회귀한 점, 그리고 재현 불가 분기(가정 1이 깨지는 경우)를 얕게 처리해 낙관 서사에 살짝 기댄 점. 리뷰어라면 나는 "5개 가설이 다 보류로 끝났을 때 이 플랜이 무엇을 하는가"를 가장 먼저 공격하겠다.

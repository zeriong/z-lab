# 사내 데이터 플랫폼 셀프서브 쿼리 인터페이스
- 추론 프레임: api-first
- 한 줄 요약: 분석가가 이미 SQL로 치던 질의 8종을 계약으로 승격하고, 애매한 케이스(도메인 간 조인 권한·PII 컬럼)를 그 예시 안에 박제한 뒤, 내부는 그 예시가 참이 되는 방향으로만 짓는다.

이 과제를 보자마자 내 반사적 습관은 "데이터 카탈로그 → 권한 모델 → 거버넌스 콘솔 → 쿼리 UI" 순으로 플랫폼 아키텍처부터 그리는 것이다. api-first는 그걸 금지한다. 분석가가 실제로 손가락으로 칠 질의부터 쓰고, 내부 구조는 그 질의가 참이 되게 만드는 종속물로만 존재한다.

## 프레임이 요구하는 시작점 — 분석가가 실제로 칠 8종 질의

전제: 분석가는 이미 SQL과 스프레드시트라는 세계를 갖고 있다. 새 DSL이나 드래그드롭 BI를 발명하지 않는다. **그들이 이미 쥔 것(SQL 방언 + "시트 새로고침" 습관)을 계약으로 승격**한다. 아래 8종은 UI 목업이 아니라 계약이다. 각 질의 옆 주석 = 플랫폼이 돌려줘야 하는 판정.

```sql
-- 1) 평범한 집계: 그냥 통과 (권한 있는 도메인, PII 없음)
SELECT country, count(*) FROM orders WHERE created_at >= '2026-07-01' GROUP BY 1;   --> RESULT

-- 2) PII 컬럼 무심코 선택: 거부가 아니라 자동 마스킹 + 감사기록 (분석가는 흐름이 안 끊긴다)
SELECT user_id, email FROM users LIMIT 100;   --> RESULT, email = "h***@***.com", audit: masked

-- 3) PII 원본이 진짜 필요: 사유 태그가 있어야만 실행. 없으면 거부(2번과 갈리는 지점)
SELECT email FROM users /*+ pii_reason:'CS 환불 대사, 티켓#4821' */ WHERE user_id = 55;   --> RESULT raw

-- 4) 권한 있는 도메인(orders) + 권한 없는 도메인(payments) 조인: 쿼리 전체를 죽이지 않고
--    막힌 컬럼만 지목 + 신청 경로를 반환 (조인 권한의 애매함을 계약에 박제)
SELECT o.id, p.card_bin FROM orders o JOIN payments p ON p.order_id=o.id;
--> DENIED: column payments.card_bin needs grant 'payments.read'. orders.* OK. 신청: /grants?col=payments.card_bin

-- 5) 조인 자체는 되지만 결과가 PII로 재식별되는 경우(orders는 익명, users.email과 조인 시 재식별)
SELECT u.email, o.total FROM users u JOIN orders o ON o.user_id=u.id;   --> RESULT, email 마스킹 유지(조인해도 못 벗긴다)

-- 6) 스프레드시트 습관 승격: 큰 결과 내보내기. 10만 행 초과 시 자동 페이지네이션 + 만료되는 서명 URL
SELECT * FROM orders WHERE created_at >= '2026-01-01';   --> EXPORT: 3 pages, signed CSV url (24h 만료)

-- 7) 실수로 무한 스캔: 비용 가드가 견적 후 차단, 파티션 힌트를 제안
SELECT * FROM events;   --> DENIED: 추정 스캔 2.1TB > 한도. 힌트: WHERE dt = '2026-07-18'

-- 8) 저장·파라미터화된 질의(시트에서 매주 새로고침하던 것): 이름으로 재실행
SELECT * FROM saved('weekly_gmv', region => 'APAC');   --> RESULT (정의된 파라미터만 허용)
```

이 8종이 계약의 전부다(가짓수보다 분기 커버가 목적). 여기서 파생되는 강제 사항: (a) 인터페이스 문법은 표준 SQL, 확장은 주석 힌트로만 — 분석가의 기존 근육 기억을 깨지 않는다. (b) 인가·마스킹·비용 가드는 쿼리 파싱 단계에서 결정되어 위 주석대로 응답해야 한다. **이 8종은 픽스처 웨어하우스에 대해 CI가 매번 재생하는 골든 테스트로 저장한다 — 예시가 곧 실행 가능한 명세(doctest)다.** 4번이 언젠가 card_bin을 흘리면 빌드가 깨진다.

## 문제 정의 / 목표

분석가가 데이터팀에 티켓을 넣지 않고 직접 안전하게 질의하게 한다. 목표는 "새 도구 도입"이 아니라 **기존 SQL/시트 습관을 그대로 두면서 권한·PII·비용 사고만 구조적으로 못 나게** 만드는 것.

## 명시적 가정 목록 (틀렸을 경우의 영향)

- **가정 1 (전체가 걸림): 대상 분석가의 다수가 이미 SQL을 읽고 쓴다.** → 틀리면(비-SQL 사용자 지배) 계약의 전제인 "SQL 승격"이 무너지고 드래그드롭 BI가 정답이 된다. 이 플랜 전체가 이 한 줄에 매달려 있다.
- 가정 2: 권한이 컬럼/도메인 단위로 표현 가능하다. → 틀리면(행 단위·목적 기반이 지배) 4번·5번의 계약을 다시 써야 한다.
- 가정 3: 어떤 컬럼이 PII인지 카탈로그에 태깅돼 있거나 태깅할 수 있다. → 틀리면 2·3·5번의 자동 마스킹이 근거를 잃고 오탐/누락이 난다.
- 가정 4: 쿼리 비용을 실행 전 견적할 수 있는 엔진(파티션 메타데이터)이 있다. → 틀리면 7번의 사전 차단이 사후 청구서로 바뀐다.

## 접근 및 단계 (각 단계의 검증 방법)

1. **계약 동결**: 위 8종을 리포지토리에 `contract.sql`로 커밋, 각 줄에 기대 판정을 주석으로 고정. — 검증: 분석가 5명에게 "네가 지난주 친 질의가 이 8종 중 어디에 해당하나" 인터뷰, 빠진 패턴 있으면 줄 추가.
2. **파싱·인가 레이어**: SQL을 파싱해 참조 컬럼/도메인을 추출, 권한·PII 태그·비용 견적을 붙여 판정. — 검증: `contract.sql`을 픽스처 웨어하우스에 재생, 8종 판정이 주석과 일치해야 통과(doctest).
3. **마스킹·재식별 방어**: 2·5번처럼 조인으로도 PII가 안 벗겨지는지. — 검증: 5번을 적대적으로 변형한 케이스(서브쿼리·CTE 우회) 10개를 골든 테스트에 추가, 전부 마스킹 유지.
4. **내보내기·저장 질의**: 6·8번. — 검증: 10만 행 초과 export가 페이지네이션되고 URL이 24h 후 401을 주는지 실측.
5. **파일럿 롤아웃**: 한 팀(예: 그로스 분석가 5~8명)에만 열고 2주. — 검증: 그 기간 데이터팀 티켓 수 감소 + 인가 사고 0건.

## 대안과 기각 사유 (부활 조건 포함)

- **드래그드롭 BI(새 GUI DSL)**: 기각. 분석가가 이미 쥔 SQL 세계를 버리게 하므로 api-first의 "기존 세계 승격" 원칙 위반, 학습 비용 + 표현력 한계. — **부활 조건**: 가정 1이 깨져 비-SQL 사용자가 다수로 판명되면 즉시 재검토.
- **모든 PII를 무조건 거부(마스킹 없음)**: 기각. 2번 흐름이 끊겨 분석가가 우회로(로컬 CSV 복사)를 만들고 거버넌스가 오히려 무너진다. — 부활 조건: 규제가 마스킹조차 불허하는 데이터 등급이 생기면 그 등급에만 적용.
- **티켓 기반 데이터팀 대행 유지(현행)**: 기각. 자체가 병목. — 부활 조건: 파일럿에서 인가 사고가 발생해 셀프서브 위험이 대행 비용을 초과하면 롤백.

## 리스크 & 완화

- R1: PII 태깅 누락 → 마스킹 안 된 채 유출(가정 3 붕괴). → 신규 컬럼은 기본 "PII 의심"으로 두고 태깅 전까지 마스킹, 카탈로그 커버리지를 주간 지표화.
- R2: 파서가 못 읽는 SQL 방언 → 인가 우회. → 파싱 실패 시 실행 거부(fail-closed), 실패 쿼리를 계약 확장 후보로 수집.
- R3: 비용 견적 과소 → 웨어하우스 청구 폭발(가정 4). → 견적 + 실행 중 실시간 스캔량 상한 이중 가드.
- R4: 분석가 채택 실패(습관 안 바꿈). → 계약을 그들의 실제 지난주 쿼리에서 뽑았으므로 진입장벽 최소화, 파일럿에서 채택률 측정.

## "완료"의 판정 기준

- `contract.sql`의 8종 질의를 픽스처 웨어하우스에 재생하면 **8종 판정이 전부 주석과 일치**(특히 4번은 payments.card_bin을 거부하고 orders는 통과, 5번은 조인 후에도 email 마스킹 유지) — CI에서 실측.
- 권한 없는 분석가 계정으로 4번을 실행하면 **쿼리 전체 실패가 아니라 막힌 컬럼만 지목 + 신청 링크 반환** — 실측.
- 파일럿 2주간 **데이터팀 대행 티켓이 감소하고 인가 사고 0건** — 실측.
- 기각한 대안(드래그드롭 BI, 무조건 거부)이 각각 부활 조건과 함께 문서에 명시돼 있음.

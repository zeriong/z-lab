# 오픈소스 라이브러리 — 날짜 범위 겹침 계산 유틸
- 추론 프레임: API-first
- 한 줄 요약: 사용자가 쓸 코드 예시를 문서 맨 위에 먼저 확정하고, 그 예시가 참이 되도록만 내부를 설계한다.

## 프레임이 요구하는 시작점 — 사용자 코드 예시 10줄 (이것이 명세다)

내부 자료구조나 알고리즘을 생각하기 전에, 이 라이브러리를 쓰는 사람이 **실제로 타이핑할 코드**를 먼저 쓴다. 이 10줄이 곧 계약이며, 아래 모든 설계는 이 줄들을 참으로 만드는 작업일 뿐이다.

```js
import { DateRange } from 'range-overlap'

const a = new DateRange('2026-01-01', '2026-01-10')
const b = new DateRange('2026-01-05', '2026-01-20')

a.overlaps(b)             // => true
a.intersection(b)         // => DateRange('2026-01-05', '2026-01-10')
a.subtract(b)             // => [ DateRange('2026-01-01', '2026-01-05') ]
DateRange.merge([a, b])   // => [ DateRange('2026-01-01', '2026-01-20') ]
a.gap(b)                  // => null   (겹치므로 간격 없음)
a.contains('2026-01-03')  // => true
new DateRange('2026-02-01', '2026-01-01') // => throws RangeError
```

이 예시에서 이미 설계 결정 대부분이 **강제**된다. 아래 "가정"과 "접근"은 전부 이 10줄을 읽고 역으로 뽑아낸 것이다.

## 예시에서 강제로 도출된 결정들

- `intersection(b)`가 `'2026-01-05'~'2026-01-10'`을 반환한다는 것 → **경계 규약을 확정해야 함.** 반열림 구간 `[start, end)`을 채택. 이유: 그래야 `subtract`가 `[a.start, b.start)` = `01-01~01-05`로 딱 떨어지고, 인접 구간(`end == 다음 start`)이 겹치지 않는 것으로 자연스럽게 처리됨. 만약 닫힌 구간 `[start, end]`이면 `01-10`이 양쪽에 중복되어 subtract/merge 경계가 지저분해진다. → **API 예시가 경계 규약을 결정했다.**
- `subtract`, `merge`가 **배열**을 반환 → 결과가 0개 또는 2개로 쪼개질 수 있음을 타입이 인정해야 함. 단일 객체 반환은 예시와 모순되므로 기각.
- `gap`이 `null` 반환 → "겹치면 간격 없음"의 표현을 예외가 아니라 null로. 예시가 반환 타입을 지정.
- 생성자가 역순 입력에 `throw` → 불변식(start < end)을 생성 시점에 강제. 이후 모든 메서드는 유효 범위를 가정할 수 있어 내부가 단순해짐.

## 문제 정의 / 목표

날짜/시간 구간의 겹침·교집합·차집합·병합·간격을 계산하는 작고 의존성 없는 유틸. 목표는 알고리즘의 화려함이 아니라 **API의 예측 가능성** — 사용자가 문서를 안 봐도 결과를 맞힐 수 있어야 한다.

## 명시적 가정 목록 (틀렸을 경우의 영향 포함)

- **가정 1:** 시간대(timezone)는 사용자가 넘기는 문자열/Date에 이미 반영돼 있고, 라이브러리는 tz를 해석하지 않는다(순수 순서 비교만). → 틀리면(tz 계산 요구) 범위가 폭증하며 별도 설계 필요. **경계를 여기서 긋는 것이 핵심 가정.**
- **가정 2:** 반열림 `[start, end)`이 사용자 직관과 맞다. → 틀리면 예시의 `intersection` 기대값이 바뀌므로 계약 재협상.
- **가정 3:** 값은 불변(immutable). 메서드는 새 객체를 반환하고 자신을 변경하지 않는다. → 틀리면 예시의 체이닝 안전성이 깨짐.

## 접근 및 단계 (예시를 참으로 만드는 순서)

1. **불변식 확정:** 생성자에서 `start`, `end`를 비교 가능한 정수(예: epoch ms)로 정규화하고 `start < end` 아니면 `RangeError`. 이후 내부는 두 정수 쌍일 뿐.
2. **원자 술어 하나:** 모든 것을 `overlaps`(= `a.start < b.end && b.start < a.end`, 반열림 기준) 위에 세운다. intersection/subtract/gap이 전부 이 한 부등식에서 파생.
3. **파생 메서드:** intersection = `[max(starts), min(ends))`, 비면 null. subtract = 남는 좌/우 조각 0~2개. gap = 안 겹칠 때 `[min.end, max.start)`.
4. **집합 연산 `merge`:** 시작값 정렬 후 1-pass 스윕으로 인접/겹침 병합 — 예시의 `[a,b] → [01-01~01-20]`을 실현.
5. **타입/문서:** 예시 10줄을 그대로 README 최상단 + 실행 가능한 doctest로 고정. 예시가 곧 회귀 테스트.

## 대안과 기각 사유

- **닫힌 구간 `[start, end]` 채택:** 기각. 예시의 subtract/merge 경계 기대값과 충돌하고, 인접 구간 처리가 특수 케이스투성이가 됨.
- **가변(mutable) API + 메서드 체이닝으로 in-place 수정:** 기각. 가정 3 위반, 예시의 `a`가 여러 번 재사용되는데 변형되면 결과가 오염됨.
- **tz·달력 연산 내장:** 기각(가정 1). 범위 폭증 대비 이득 적음. 필요하면 사용자가 넘기기 전에 처리.

## 리스크 & 완화

- **R1: 반열림 규약이 일부 사용자 직관과 어긋남.** → README 최상단에서 규약을 첫 문장으로 명시, 예시로 즉시 보여줌.
- **R2: 부동/에지 케이스(빈 구간, 같은 경계 인접).** → 원자 술어 하나로 환원했으므로 에지 케이스를 그 부등식의 등호 처리로 일괄 검증.

## "완료"의 판정 기준

- 문서 최상단 예시 10줄이 **그대로 실행돼 주석의 기대값과 일치**한다(예시 = 테스트).
- 모든 공개 메서드가 원자 술어 `overlaps` 또는 그 파생으로 설명 가능(내부 개념 수 최소화).
- 외부 런타임 의존성 0개.

# 대형 JS 코드베이스의 TypeScript 점진 전환
- 추론 프레임: min-diff
- 한 줄 요약: 파일 하나를 최소 단위로, "동작을 그대로 옮기는 전환"과 "타입을 좋게 만드는 개선"을 커밋 수준에서 분리해, 어떤 단계든 한 문장으로 되돌릴 수 있을 때만 진행한다.

## 프레임이 요구하는 시작점 — 단계의 최소 단위와 한 문장 롤백

이 과제를 보자마자 내 반사적 습관은 "codemod 하나 돌려서 레포 전체를 .ts로 바꾸고, tsconfig에 strict 켜고, 이왕 파일 여는 김에 눈에 띄는 버그도 고치자"였다. 그런데 이게 정확히 min-diff가 금지하는 것이다. min-diff의 시작점은 "모든 단계는 독립 배포·독립 롤백 가능해야 하고, 롤백을 한 문장으로 못 쓰면 그 단계는 너무 크다"이다. 그래서 설계의 첫 결정은 아키텍처가 아니라 **단계의 크기**다.

- **단위 = 파일 1개**(또는 강하게 묶인 소모듈 1개). allowJs를 켠 공존 빌드를 먼저 만들어, 파일 하나를 .ts로 바꿔도 나머지 .js가 그대로 굴러가게 한다.
- 각 단계의 롤백은 한 문장이어야 한다: "그 파일의 rename+타입 주석 커밋을 revert하면 .js 원본이 돌아온다." 이 문장이 안 써지는 단계(예: 20개 파일 동시 변환)는 쪼개서 다시 온다. ← 이것이 진입 관문(gate)이지 선언이 아니다.

## 문제 정의 / 목표

수십만 라인 규모의 JS 코드베이스를, 서비스 중단·기능 회귀 없이 TypeScript로 옮긴다. 목표는 "전부 엄격 타입"이 아니라 **언제든 되돌릴 수 있는 상태를 유지하며 컴파일 그린을 확장**하는 것. 타입 품질 향상은 전환이 끝난 뒤 별도 트랙으로 민다.

## 명시적 가정 목록 (틀렸을 경우의 영향)

- **가정 1 (전체가 걸림): 빌드/번들 파이프라인이 JS와 TS를 동시에 처리할 수 있다(allowJs, 점진 컴파일).** → 틀리면 파일 단위 공존 자체가 불가능해져 min-diff의 전제(독립 배포)가 무너지고 빅뱅 외 선택지가 없어진다. 그래서 0단계에서 이것부터 실측한다.
- 가정 2: 기존 코드에 동작을 고정하는 테스트가 어느 정도 있다. → 틀리면 "동작 보존"을 검증할 수단이 없어 전환 커밋의 무해성을 증명하지 못한다(리스크 R1로 이어짐).
- 가정 3: 런타임 산출물(emit된 JS)이 전환 전후로 동일하게 유지된다(타입은 지워질 뿐). → 틀리면(enum·데코레이터·target 상향으로 코드 생성) "전환=무동작변경"이 거짓이 되어 분리 원칙이 깨진다.
- 가정 4: 의존 그래프의 리프(말단) 모듈부터 위로 올라갈 수 있다. → 틀리면 순환 의존 탓에 파일 하나만 바꾸는 게 불가능해 단위가 커진다.

## 접근 및 단계 (각 단계: 검증 + 한 문장 롤백)

0. **공존 빌드 세우기.** tsconfig(allowJs: true, strict: false, noEmitOnError: false), 번들러가 .ts를 통과시키는지 확인. → 검증: 아무 .js 하나를 내용 그대로 .ts로 확장자만 바꿔 빌드·테스트 그린. → 롤백: tsconfig·확장자 변경 커밋 revert.
1. **리프 모듈부터 파일 단위 전환(전환 커밋).** .js→.ts, 명백한 타입만 주석, 애매한 곳은 any/unknown. 로직은 한 줄도 안 바꾼다. → 검증: emit된 JS diff가 실질적으로 비어 있음 + 기존 테스트 그린. → 롤백: 해당 파일 커밋 하나 revert.
2. **경계에 타입 세우기(전환 유지).** 외부에서 들어오는 값은 unknown으로 받아 좁히는 최소 가드만 추가. → 검증: 컴파일 그린 + 런타임 동작 동일. → 롤백: 그 파일 revert.
3. **개선 커밋(별도 트랙).** any 제거, strict 플래그를 하나씩 켜기(noImplicitAny → strictNullChecks 순). → 검증: 플래그별로 컴파일 그린 + 테스트. → 롤백: 플래그 한 개 되돌리기 = 한 문장.

핵심 규율: **전환 커밋과 개선 커밋은 절대 같은 커밋에 섞지 않는다.** 전환은 "동작 보존", 개선은 "타입 향상". 전환 후 버그가 나면 emit diff가 비어 있으므로 원인이 전환이 아님을 즉시 알 수 있고, 개선 후 버그면 타입 변경이 로직을 건드린 것이다. 이 분리가 없으면 회귀의 원인을 둘로 갈라낼 수 없다.

## any 허용 전략

전환 단계에서 any는 죄가 아니라 도구다. 목표가 "잘 타이핑됨"이 아니라 "TS로 컴파일됨"이므로 애매한 곳은 넉넉히 any/unknown으로 둔다. 단:

- 의도적 any는 검색 가능한 표식을 남긴다: `// TODO(ts-migrate): any` 또는 eslint-disable 한 줄. 이게 개선 트랙의 백로그가 된다.
- strict/noImplicitAny는 전환 중엔 끈다. 켜는 순간 전환이 로직 수정을 강요해 분리 원칙이 깨진다. strict는 전환이 끝난 파일에 개선 커밋으로 하나씩 켠다.
- 경계(외부 입력)만 예외적으로 any 대신 unknown — 여기서의 any는 런타임 안전에 직접 영향을 준다.

## 대안과 기각 사유 (부활 조건)

- **빅뱅 codemod(ts-migrate 등으로 레포 일괄 변환):** 기각. 수천 파일 동시 변경은 롤백이 한 문장이 안 되고("전부 되돌린다"는 곧 전환 포기), 회귀 원인 이분이 불가능. → 부활 조건: 레포가 충분히 작고(수백 파일) 테스트 커버리지가 높아 일괄 변환 1커밋을 24h 내 검증·되돌릴 수 있을 때.
- **strict 전면 선(先)적용 후 오류 소거:** 기각. 전환과 개선을 강제로 한 덩어리로 묶어 분리 원칙 위반. → 부활 조건: 신규 코드에 한해(그린필드 디렉터리)만 strict 기본값.
- **"기능 단위" 전환:** 기각. 기능은 여러 파일에 걸쳐 단위가 커지고 롤백 문장이 길어진다. → 부활 조건: 의존 그래프상 그 기능이 이미 단일 모듈로 응집돼 있을 때.

## 유혹 로그 (min-diff 필수 — 빅뱅·겸사겸사 기각 기록)

기록하고 명시적으로 기각한 것들:

- "파일 여는 김에 이 버그도 고치자" → 기각. 버그픽스는 전환 커밋에서 분리해 별도 커밋·별도 PR로. 전환 PR의 diff는 타입 주석과 확장자뿐이어야 리뷰어가 "무동작"을 눈으로 검증한다.
- "이왕 하는 거 폴더째 변환" → 기각(빅뱅). 단위는 파일이다.
- "여기 있는 var/require도 ESM으로 현대화하자" → 기각. 문법 현대화는 또 다른 동작-변경 위험, 별도 트랙으로.

## Rule of three (min-diff 필수)

전환 중 같은 객체 형태가 두 파일에서 반복돼도 공용 `types.ts`로 빼지 않는다. **세 번째** 중복이 나타날 때 비로소 공용 타입을 추출한다. 이유: 이른 추출은 (a) 여러 파일을 한 커밋에 묶어 단위를 키우고, (b) 아직 안정되지 않은 형태를 성급히 고정한다. 두 번까지는 인라인 중복을 허용하는 게 min-diff답다.

## 리스크 & 완화

- R1: 동작 보존을 검증할 테스트 부족(가정 2 실패). → 전환 대상 파일에 최소 특성화(characterization) 테스트를 "전환 선행 작업"으로 먼저 추가. 그래도 없으면 emit JS diff를 유일 방어선으로 삼고 그 사실을 PR에 명시.
- R2: emit 차이 유발 기능(enum, namespace, target 상향)이 "전환=무동작"을 깬다(가정 3 실패). → 전환 커밋에서 이런 기능 사용 금지, enum 대신 union 리터럴 도입은 개선 트랙으로.
- R3: 파일 공존 기간이 길어져 .js/.ts 혼재가 상시화. → "완료" 기준에 잔여 .js 수 상한과 기한을 넣어 관성적 방치를 막음.
- R4: 순환 의존으로 파일 단위가 안 지켜짐(가정 4 실패). → 그 순환을 먼저 별도 전환-선행 커밋으로 끊고(동작 보존) 그다음 전환.

## "완료"의 판정 기준

- 임의의 전환 PR 하나를 골라 revert했을 때 **빌드·테스트가 그대로 그린이고 emit된 JS diff가 비어 있음**(독립 롤백·무동작 실측).
- 전환 커밋의 diff에 로직 변경이 0줄(확장자·타입 주석만)임을 리뷰에서 확인 — 전환/개선 분리의 실측.
- 남은 .js 파일 수가 사전 합의한 상한(예: 0, 혹은 지정된 레거시 디렉터리) 이하.
- 의도적 any가 전부 `TODO(ts-migrate)` 표식을 달고 있어 개선 백로그로 grep됨.

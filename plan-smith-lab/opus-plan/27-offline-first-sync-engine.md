# 오프라인 우선 모바일 메모 앱의 동기화 엔진 설계
- 추론 프레임: backward
- 한 줄 요약: "동기화가 완료됐다"의 인수조건 5개를 먼저 못 박고, 각 조건이 어떤 엔진·자료구조 대안을 죽이는지를 역산해 설계를 확정한다.

## 프레임이 요구하는 시작점 — 인수조건 5개와 그것이 죽이는 대안

기능이 아니라 "완료의 조건"을 먼저 확정한다. 아래 5개는 실패하면 재작성이 최대(merge 코어·자료 모델·서버 저장 모델까지 갈아엎어야 함)인 조건만 골랐다. 즉 앵커 자격 시험을 통과한 것들이다. 각 조건 옆에 "이 조건이 죽이는 대안"을 명시해, 미리 정해둔 결론(내 관성인 CRDT)으로 되돌아가는 것을 막는다.

- **A1 수렴(determinism):** 모든 기기가 온라인이 되어 변경을 교환하면, 메시지 도착 순서와 무관하게 바이트 동일한 상태로 수렴한다. → **죽이는 대안:** 벽시계 타임스탬프 기반 LWW(시계 오차 시 비결정적), "서버가 최종 진실이고 도착 순서로 덮어쓰기". 결정적 병합만 남는다.
- **A2 동시 편집(no lost update):** 두 기기가 같은 노트를 동시에 편집하면 양쪽 편집이 모두 보존된다. → **죽이는 대안:** 노트 단위 LWW(한쪽 통편집이 다른 쪽을 삭제), 굵은 락. 세밀한(문자/블록 단위) 병합만 남는다.
- **A3 장기 오프라인:** 수 주~수 개월 오프라인이던 기기가 복귀해도 "너무 오래 떨어져 있었으니 초기화하라"는 요구 없이 병합된다. → **죽이는 대안:** 서버 순차 버전번호·OT 리베이스(발산이 크면 변환 비용 폭발), 만료되는 세션 토큰 기반 동기화. 발산량에만 비례하는 교환식(commutative) 병합만 남는다.
- **A4 복원:** 로컬 저장소를 잃은 기기(재설치·기기 교체)가 서버/피어로부터 전체 상태를 되찾는다. → **죽이는 대안:** op 로그가 잃어버린 기기에만 있는 클라이언트 전용 설계. 내구성 있는 서버 로그+스냅샷이 강제된다.
- **A5 충돌 UX 없음:** 사용자는 "충돌 — 버전을 고르시오" 대화를 절대 보지 않는다. → **죽이는 대안:** OT/Git식 3-way 병합의 충돌 마커. 모든 충돌 부류에 미리 정해진 자동 해소가 있는 전역 병합 함수가 강제된다.

**앵커 밖 요구의 자격 박탈:** 실시간 커서/프레즌스, 서식(rich text), 종단간 암호화(E2EE), 버전 타임머신 UI, 다중 사용자 공유·권한 — 이들은 5개 조건에 없다. 따라서 아키텍처를 움직일 권리가 없다. 미래 모듈로 들어올 수는 있어도, 이 문서의 merge 코어·저장 모델을 재설계하게 만들 수 없다(E2EE는 서버측 압축과 충돌하나, 앵커가 아니므로 이번엔 코어를 바꾸지 못한다 — R 절에 지뢰로만 기록).

## 문제 정의 / 목표

주 사용 시나리오는 단일 사용자의 다기기(폰·태블릿) 사용이며, 기기는 오래 오프라인일 수 있다. 목표는 위 5개 조건을 실측으로 만족하는 동기화 엔진의 자료구조·프로토콜·서버 저장 모델을 확정하는 것이다. 기능 확장이 아니라 "5개 조건의 충족"이 성공이다.

## 명시적 가정 목록 (틀렸을 경우의 영향)

- **가정 1(전체가 걸림): 노트 본문은 편집 가능한 텍스트이고, 같은 노트의 같은 지점을 두 기기가 실제로 동시 편집하는 경우가 실재한다.** → 틀리면(노트가 append-only거나 사실상 한 기기에서만 편집) 시퀀스 CRDT 전체가 과설계이며 노트 단위 LWW-레지스터로 충분하다. 이 가정이 A2·A5의 무게를 지탱한다.
- 가정 2: 기기당 노트는 수천~수만, 노트당 수 KB. → 틀리면(초대형 문서·수억 op) 문자당 ID의 CRDT 메타 오버헤드가 폭발 → 블록 단위 granularity로 하향.
- 가정 3: 신뢰 가능한 중앙 서버가 존재하고 op 로그를 내구성 있게 보관한다(E2EE는 초기 범위 밖). → 틀리면(E2EE 필수 또는 순수 P2P) A4 복원·압축 전략을 재설계.
- 가정 4: 최대 오프라인 지평(예: 90일)을 제품이 허용하고, 그 이상은 "복원 경로"로 처리한다. → 틀리면(무한 오프라인을 델타 병합으로 요구) 툼스톤 GC 불가 → 저장소 무한 증가.

## 접근 및 단계 (각 단계는 하나의 앵커를 검증한다)

역산 결과, A1+A2+A5는 CRDT 계열을 강제하고 A3는 교환식 병합+버전벡터를, A4는 서버 내구 로그를 강제한다. 단, A1만으로는 단일값 메타(제목·고정·보관 플래그)까지 시퀀스 CRDT로 만들 이유가 없다 — 여기서는 Lamport 순서의 LWW/MV-레지스터가 살아남는다(전부 CRDT화는 과설계).

1. **병합 계약(전역 병합 함수) 확정.** 문서 = 맵 CRDT, 노트 = {본문: 시퀀스 CRDT(RGA 계열), 메타: LWW/MV-레지스터, 존재: add-wins 셋}. → 검증: 무작위 op 집합을 N개 순서로 적용해 최종 상태가 동일함을 속성 테스트(A1 실측).
2. **자료구조·ID.** 원소 ID = (actorId, lamport counter). 벽시계 금지. → 검증: 두 액터가 같은 위치에 동시 삽입 → 두 문자 모두 존재+결정적 순서(A2 실측).
3. **버전벡터+델타 동기화.** 클라이언트가 자기 VV를 보내면 서버가 누락 op만 회신(양방향). → 검증: 피어를 K op만큼 오프라인시켰다 복귀 → 델타 크기가 발산량에 비례하며 수렴(A3 실측).
4. **서버 내구 op 로그+스냅샷/압축.** VV 기반 안전 GC를 오프라인 지평으로 경계 짓는다. → 검증: 클라이언트 상태를 지운 뒤 빈 VV로 부트스트랩 → 전체 상태 바이트 동일 복원(A4 실측). 지평 경과 후 툼스톤이 회수되어 저장소가 유한함도 확인.
5. **병합 정책 표.** 모든 충돌 부류(동시 텍스트 편집, 편집 vs 삭제, 동시 제목, 동시 체크박스)에 사용자 프롬프트 없는 자동 해소를 사전 지정. → 검증: 충돌 부류를 열거해 어느 코드 경로도 사용자 충돌 UI를 내지 않음을 단언(A5 실측). 적대적으로: 리뷰어가 프롬프트를 강제하는 사례 구성을 시도.
6. **채택 가능한 롤아웃.** 단일 기기 우선 출시 → 플래그 뒤 내부 기기에서 동기화 켜기 → 서버 병합 결과와 클라이언트 상태를 섀도우 비교 → 확대. (이는 앵커 밖 편의이며, 아래 자백에 그 성격을 기록한다.)

핵심 교차점: A3(장기 오프라인)와 A4(복원)는 GC에서 만난다. 오프라인 피어가 아직 못 본 op·툼스톤은 안전히 지울 수 없어 저장소가 무한 증가한다. 이를 지평(90일)으로 경계 짓고, 그 이상 떨어진 기기는 "신규 기기"로 취급해 델타가 아니라 복원(A4) 경로를 태운다. 즉 복원 메커니즘이 유한 GC를 안전하게 만든다 — 역산이 아니었으면 GC를 별개 문제로 미뤘을 것이다.

## 대안과 기각 사유 (부활 조건 포함)

- **OT(Operational Transformation):** 기각. 중앙 서버 변환 필요+장기 발산 시 변환 비용 폭발(A3 위반), 변환 함수 정확성이 악명 높음. → 부활: A3를 포기(항상 준-온라인, 짧은 발산)하고 클라이언트 메모리를 최소화하려면 경쟁력 회복.
- **전면 LWW(노트 단위 벽시계):** 기각. A2(유실)·A1(시계 의존 비결정) 위반. → 부활: 동시 편집이 의미 없는 단일값 필드(고정 여부 등)에는 Lamport 순서 LWW-레지스터로 **좁게 채택**됨. 전면 부활은 가정 1이 거짓일 때만.
- **Git식 3-way 병합(충돌 마커):** 기각. A5 정면 위반. → 부활: 소비자 메모엔 없음. 사용자가 병합 충돌을 수용하는 개발자용 코드-노트 제품이면 재검토.
- **순수 P2P(서버 없음):** 기각. A4 복원은 항상 가용한 내구 사본을 요구하나 순수 P2P는 복원 시점에 피어가 온라인임을 보장 못 함. → 부활: 상시 클라우드 피어를 두거나 "다른 기기가 온라인일 때만 복원"을 수용하면 재진입.

## 리스크 & 완화

- **R1 CRDT 메타 오버헤드 → 대형 노트/장기 이력에서 메모리 폭발.** → 블록 단위 granularity, 컬럼형/RLE 인코딩, 주기적 압축. 트리거: 노트당 오버헤드가 임계 초과 시 granularity 하향.
- **R2 동일 지점 동시 타이핑의 interleaving(글자 뒤섞임).** 자동 병합은 수렴하나 A5의 정신(깔끔한 병합)은 근사 충족에 그침. → interleaving 저항 시퀀스 CRDT(Fugue/RGA 변형) 사용. 병리적 동일 단어 동시 편집이 이상하게 읽힐 수 있음을 정직히 표기 — 다만 절대 프롬프트로 막지는 않음(A5의 문자 충족).
- **R3 툼스톤/로그 무한 증가.** → 지평 경계+복원 폴백(위 교차점).
- **R4 스키마 진화.** CRDT 문서는 출시 후 마이그레이션이 어려움. → 문서 스키마 버전화, 전방 호환 필드 처리, 마이그레이션 op.
- **R5 actorId 중복(재설치 시 재사용)으로 인과성 붕괴.** → 설치마다 새 actorId 발급, 재사용 금지.
- **R6 앵커 밖 지뢰(E2EE).** 서버가 op를 읽지 못하면 4단계 압축이 무력화됨. 이번엔 앵커가 아니라 코어를 바꾸지 않으나, E2EE가 로드맵에 오르면 서버 저장 모델 재설계가 강제됨을 미리 경고.

## "완료"의 판정 기준 (실측 문장)

- **A1:** 3개 시뮬레이션 액터의 op 순열 10,000회 무작위 시행 → 100% 바이트 동일 최종 상태.
- **A2:** 두 기기가 오프라인 상태로 같은 줄을 편집 → 동기화 후 양쪽 기기에 두 편집이 모두 존재, 결정적 순서, 데이터 유실 0.
- **A3:** 89일 오프라인 + 로컬 5,000 op 기기가 복귀 → 델타 전송으로 수렴, 초기화 요구 없음.
- **A4:** 로컬 데이터 없는 기기에서 앱 삭제 후 재설치 → 전체 노트가 피어와 바이트 동일하게 복원.
- **A5:** 열거된 충돌 부류 테스트 스위트 전반에서 사용자 충돌 프롬프트를 내는 코드 경로가 0개(단언/grep로 실측).

## 프레임 이탈 / 관성 회귀 자백

- **진입 관성:** 이 과제를 보자마자 내 반사는 "동기화 엔진 = CRDT = Yjs/Automerge, 그 구조를 서술하자"였다. backward는 해답에서 출발하는 걸 금지한다 — 5개 조건에서 대안을 죽여가며 CRDT를 역산하게 했고, 더 중요하게는 CRDT가 강제되지 *않는* 곳(단일값 메타는 LWW-레지스터로 충분)을 찾게 했다. 그러나 정직히: A1+A2+A5가 워낙 강하게 CRDT를 가리켜, 내가 사전 결론으로 back-chaining했을 가능성을 완전히 배제 못 한다. "어느 조건이 어느 대안을 죽이나"라는 검사는 실제로 돌렸지만 결론이 내 출발점과 일치한다 — 이것이 backward의 바로 그 실패 모드이고 나는 부분적으로만 벗어났다.
- **관례 회귀:** "접근 및 단계"가 결국 빌드 순서(1→6)라는 전방 구현 서사로 흘렀다. 04 로그가 지적한 것과 같은 회귀다.
- **안전한 기본값:** 6단계 파일럿/플래그 롤아웃을 넣었다. 5개 조건 어디도 롤아웃 계획을 요구하지 않으므로 이는 앵커 밖 편의다. 정직한 엔지니어링이라 남겼지만, 내가 "수렴적 안전판"으로 무른 지점임을 표시한다.
- **가장 약한 절:** R2(interleaving). A5는 문자상(프롬프트 없음) 충족되나 정신(깔끔한 병합)은 근사에 그쳤고, Fugue/RGA를 손짓만 했지 UX가 수용 가능함을 증명하지 못했다. 리뷰어는 여기를 쳐야 한다.

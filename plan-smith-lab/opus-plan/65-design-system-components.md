# 팀 공용 디자인 시스템 컴포넌트 라이브러리
- 추론 프레임: api-first
- 한 줄 요약: 소비 개발자가 실제로 타이핑할 10줄(합성·variant·스타일 오버라이드 충돌 포함)을 헌법으로 먼저 못 박고, 그 예시를 참으로 만드는 방향으로만 내부를 설계한다.

## 프레임이 요구하는 시작점 — 소비 코드 10줄(계약)

이 과제를 보자마자 내 반사적 습관은 "스택부터"다: React냐, 스타일링은 CSS-in-JS냐 Tailwind냐, 토큰 먼저 정의하고 Storybook·모노레포부터 세운다. api-first는 정확히 그걸 금지한다. 내부는 아래 예시를 참으로 만드는 데 필요한 만큼만 존재한다.

```tsx
import { Button, Dialog, Field } from "@team/ui";

<Dialog>
  <Dialog.Trigger asChild>
    <Button variant="primary" size="sm">저장</Button>        // 합성: 기존 Button을 트리거로 위임(asChild)
  </Dialog.Trigger>
  <Dialog.Content>
    <Field label="이메일" error={errors.email}>              // 합성: Field가 label/error/임의 입력을 감쌈
      <input {...register("email")} />                       // 팀이 이미 쓰는 react-hook-form input 그대로
    </Field>
    <Button variant="ghost" className="mt-4 w-full">취소</Button>  // 오버라이드 에지: className이 variant와 충돌
  </Dialog.Content>
</Dialog>
```

이 10줄이 헌법이다. 세 조항이 예시 안에 **의도적으로** 박혀 있다.

- **소비자의 기존 세계를 계약으로 승격:** `register(...)`, `className="mt-4 w-full"`, 네이티브 `<input>` — 팀이 이미 쓰는 폼 라이브러리·유틸리티 클래스·네이티브 엘리먼트를 그대로 받는다. 라이브러리가 새 문법을 강요하지 않는다.
- **에지 케이스를 계약에 못 박음:** `className="mt-4 w-full"`이 `variant="ghost"`의 자체 스타일과 충돌한다. 이건 해피패스 전시가 아니라 분쟁 예방 조항이다 — "소비자 className은 variant를 지우지 않고 병합하되, 같은 속성(padding 등)은 소비자가 이긴다"를 계약으로 선언한다. `asChild` 한 단어는 다형성(트리거가 별도 button을 만들지 않고 자식 Button에 동작을 주입)을 계약한다.
- **예시=실행 명세:** 이 10줄을 그대로 렌더하는 테스트가 CI에 상주한다(단계 2).

## 문제 정의 / 목표

여러 화면·여러 개발자가 제각각 버튼·모달·폼을 재구현해 스타일이 어긋나고 접근성이 빠진다. 목표는 "컴포넌트를 많이 만드는 것"이 아니라 **소비 개발자가 위 10줄처럼 쓰기만 하면 일관성과 접근성이 공짜로 따라오는 계약**을 세우는 것이다.

## 명시적 가정 (틀렸을 경우의 영향)

- **가정 A(전체가 걸림): 소비 개발자는 이미 React + TypeScript + 유틸리티 클래스(Tailwind류)를 쓴다.** → 틀리면(예: Vue, CSS Modules, CSS-in-JS) className 병합·asChild를 포함한 10줄 계약 자체가 남의 세계 위에 지어진 것이 되어 전면 재작성. api-first의 "기존 세계 반영"이 붕괴하는 지점.
- 가정 B: 소비자는 조직 내부 개발자(외부 npm 공개 아님). → 틀리면 semver·파괴적 변경·문서 부담이 급증하고 계약 변경 비용이 수십 배가 된다.
- 가정 C: 디자이너가 색·간격·타이포 토큰을 소스오브트루스로 관리하고 개발이 소비. → 틀리면 variant 정의가 저자 취향이 된다(무엇이 primary인지 근거 없음).
- 가정 D: 팀이 react-hook-form류 폼을 쓴다. → 틀리면 Field가 임의 input을 감싸는 계약이 과설계가 된다.

## 접근 및 단계 (각 단계의 검증)

1. **계약 확정:** 소비 개발자 3명이 10줄 예시를 자기 실제 화면 코드로 치환해본다. → 검증: "이대로 쓰겠다"에 서명, 어색한 줄 0. 예시가 저자 취향이면 여기서 반드시 걸린다.
2. **예시를 실행 테스트로 고정(doctest):** 10줄을 렌더 → (a) 계산된 스타일에 `mt-4`의 마진과 variant 패딩이 공존하되 충돌 속성은 소비자 승, (b) `asChild`가 DOM에 button 엘리먼트를 정확히 하나 생성, (c) `Field`의 label `htmlFor`가 자식 input `id`에 연결. → 검증: 이 테스트 통과 = 명세 확정.
3. **계약이 강제하는 내부만 도출:** `className` 병합 → 결정론적 병합기(tailwind-merge) 필요; `asChild` → Slot 패턴 필요; `variant`/`size` 자동완성 → 유니온 타입 우선. → 검증: 모든 내부 결정이 예시의 특정 줄로 역추적되는가(역추적 안 되면 그 결정은 삭제).
4. **3개(Button·Dialog·Field)만 파일럿, 실제 화면 1개를 이걸로 재작성.** → 검증: 재작성 diff에서 임의 인라인 스타일/override 라인이 이전 대비 급감(실측).
5. **계약이 버틴 뒤에만 확장.** 신규 컴포넌트는 동일한 override·composition 계약을 상속한다. → 검증: 신규 컴포넌트 PR이 계약 체크리스트를 위반 0으로 통과.

## 대안과 기각 사유 (부활 조건)

- **기존 라이브러리 통째 채택(MUI/Chakra 등):** 기각 — 그들의 계약을 강요해 "기존 세계 반영"을 위반한다. 다만 계약이 요구한 기계(Slot·tailwind-merge)는 Radix 프리미티브 등으로 **직접 재발명하지 않고 조립**한다(채택 가능한 관행에 착지). 부활 조건: 팀의 기존 코드가 이미 특정 라이브러리 계약과 일치하면 그걸 얇게 래핑.
- **토큰/디자인 문서 먼저:** 기각 — 소비 코드가 없으면 토큰·문서는 저자 취향의 전시다. 부활 조건: 디자이너-개발 토큰 동기화가 주 병목으로 판명되면 토큰 우선이 맞다.
- **Storybook 카탈로그 먼저:** 기각 — 카탈로그는 계약의 산출물이지 출발점이 아니다. 부활 조건: 외부 공개 라이브러리로 전환(가정 B 붕괴) 시 문서 우선순위가 상승한다.

## 리스크 & 완화

- R1 className 병합 규칙 모호 → `!important` 전쟁. 완화: 예시에 충돌 케이스를 박아 계약화 + 결정론적 병합기, 우선순위 1줄 명문화.
- R2 asChild 오용 → aria 속성 소실/이중 래핑. 완화: 테스트가 단일 엘리먼트 + aria 전달을 검증(단계 2b).
- R3 확장 시 계약 드리프트(컴포넌트마다 제멋대로 prop). 완화: PR 게이트 = 10줄 계약 준수 체크리스트.
- R4(빠뜨린 범주 점검에서 건짐): 접근성·다크모드·SSR. 완화: variant를 CSS 변수 토큰으로 → 다크모드; asChild 계약이 aria 전달을 강제 → a11y 기본선; 초기엔 클라이언트 전용 가정, SSR 대응은 확장 시점에 결정.

## "완료"의 판정 기준

- 실측: **10줄 예시를 그대로 렌더한 CI 테스트가 통과**하고, `className="mt-4 w-full"`이 variant 패딩을 지우지 않고 병합되며(계산된 스타일에 마진·패딩 공존), `asChild`가 DOM에 button을 정확히 1개 만든다.
- 파일럿 화면 재작성 후 임의 인라인 스타일/override 라인 수가 재작성 전 대비 측정 가능한 수준으로 감소한다.
- 신규 컴포넌트 3개가 동일 override·composition 계약 위반 0으로 추가된다.
- 기각 항목(라이브러리 통째 채택·토큰 먼저)이 부활 조건과 함께 문서에 남아 있다.

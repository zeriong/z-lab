# 백엔드 개발자의 3개월 Rust 학습 계획
- 추론 프레임: 오개념 예측 우선
- 한 줄 요약: 이 학습자가 기존 지식 때문에 겪을 오개념을 먼저 예측·목록화하고, 오개념 교정 시점을 커리큘럼의 뼈대로 삼는다.

## 프레임이 요구하는 시작점 — 오개념 지도부터 (진도표 아님)

"1주차 문법, 2주차 소유권..." 식 진도표를 짜지 않는다. 먼저 **기존 백엔드(GC 언어) 지식이 Rust에서 오작동하는 지점**을 예측한다. 커리큘럼은 이 오개념들을 언제 어떻게 깨뜨릴지로 구성된다.

**예측된 오개념 (기존 지식의 간섭)**
- **M1 — "변수는 값을 담는 상자다"(GC 멘탈모델).** Rust에선 변수가 값을 *소유*하며 이동(move)하면 원본이 무효화된다. GC 언어 출신은 `let b = a;` 후 `a`를 계속 쓰려다 막힌다. **가장 크고 이른 오개념.**
- **M2 — "메모리는 신경 안 써도 된다".** 스택/힙, 소유권/빌림을 무의식적으로 무시. borrow checker와 계속 싸우며 "왜 컴파일러가 나를 괴롭히지"로 좌절.
- **M3 — "참조는 그냥 포인터/별칭".** Rust의 빌림 규칙(가변 빌림은 배타적, 불변 빌림은 공유)을 모르면 aliasing 규칙에 계속 막힘.
- **M4 — "예외로 에러 던지면 됨".** Rust는 `Result`/`Option`로 에러를 값으로 다룸. try/catch 습관이 `?` 연산자·패턴매칭 이해를 방해.
- **M5 — "OOP 상속으로 재사용".** Rust는 상속이 없고 trait/composition. 클래스 계층 습관이 trait 설계를 방해.
- **M6 — "동시성 = 락 잘 걸기".** Rust는 타입 시스템(Send/Sync, 소유권)이 데이터 레이스를 컴파일 타임에 막음. "런타임에 조심" 습관이 이 모델을 안 보이게 함.

## 문제 정의 / 목표

백엔드 개발자가 3개월에 Rust를 실무 가능 수준으로. 목표는 "문법 진도"가 아니라 **기존 지식의 간섭(M1~M6)을 제때 교정해 자립**. 오개념을 안 깨면 문법을 배워도 계속 헛디딘다.

## 오개념 교정 시점을 뼈대로 한 커리큘럼

**1개월차 — M1·M2·M3 교정 (소유권/빌림, 가장 이르고 큰 벽):**
- 의도적으로 move 에러·borrow 에러를 *일부러* 만들어보고 컴파일러 메시지를 읽는 훈련. "컴파일러가 적이 아니라 교사"로 프레이밍 전환(M2 좌절 예방).
- 상자 멘탈모델(M1)을 "소유권 이전" 멘탈모델로 교체하는 게 이 달의 유일한 성공 지표.

**2개월차 — M4·M5 교정 (에러 처리, trait):**
- 예외 습관(M4)을 `Result` + `?`로 대체. 실제 코드에서 try/catch를 쓰고 싶은 순간을 포착해 Rust식으로 재작성.
- 상속을 쓰려는 순간(M5)마다 trait/composition으로 다시 설계하는 훈련.

**3개월차 — M6 교정 + 통합 (동시성, 실전 프로젝트):**
- 데이터 레이스를 일부러 시도 → 컴파일러가 막는 걸 체험(M6 모델 전환).
- 작은 백엔드 서비스(기존 강점 도메인)를 Rust로 구현하며 M1~M6이 실전에서 재발하는지 점검.

## 명시적 가정 목록 (틀렸을 경우의 영향 포함)

- **가정 1:** 학습자의 최대 장애물은 새 문법이 아니라 기존 지식의 간섭이다. → 이 가정이 프레임 전체의 전제. 틀리면(순수 초심자) 오개념 예측이 무의미하고 일반 진도표가 나음.
- **가정 2:** M1(소유권)이 가장 이르고 큰 오개념이다. → 틀리면 커리큘럼 순서 재배치.
- **가정 3:** "오개념을 일부러 유발해 교정"이 강의식 설명보다 효과적. → 틀리면 방법론 조정(설명 비중↑).

## 접근 및 단계

1. **오개념 진단:** 학습 시작 시 작은 코드 과제로 어떤 오개념이 실제로 나타나는지 관찰(예측 검증).
2. **큰/이른 오개념부터:** M1~M3(소유권)을 1개월에 집중. 여기가 안 뚫리면 이후가 다 막힘.
3. **오개념 유발→교정 루프:** 각 오개념을 설명이 아니라 "직접 부딪혀 컴파일러에게 배우기"로.
4. **강점 도메인에서 통합:** 3개월차에 익숙한 백엔드를 Rust로 재구현해 전이 확인.

## 대안과 기각 사유

- **문법 진도표(변수→함수→구조체→...):** 기각. 프레임 위반. 문법을 순서대로 배워도 M1(소유권)을 못 깨면 계속 좌절. 오개념이 진도를 막는 실제 병목.
- **책 1권 완독 후 프로젝트:** 기각. 오개념은 읽어선 안 깨지고 부딪혀야 깨짐. 수동적 학습은 M2(borrow checker 좌절)를 오히려 키움.

## 리스크 & 완화

- **R1: M2(borrow checker 좌절)로 중도 포기.** → 컴파일러를 교사로 프레이밍, 초반에 "에러는 정상"이라는 기대 설정. 이 좌절 관리가 3개월 완주의 핵심.
- **R2: 예측한 오개념이 실제와 다름.** → 1단계 진단으로 예측을 실제 관찰로 보정. 예측은 가설일 뿐.

## "완료"의 판정 기준

- 학습자가 **borrow checker 에러를 스스로 읽고 해결**(M1~M3 교정 확인).
- 에러 처리·추상화를 **Result/trait로 자연스럽게**(M4·M5 교정), 예외/상속 습관으로 회귀 안 함.
- 강점 도메인(백엔드) 소규모 서비스를 Rust로 완성(전이·통합 실증).

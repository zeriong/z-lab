# 1인 SaaS 운영자의 4주 무인 운영 — 안전판 설계
- 추론 프레임: safety-net
- 한 줄 요약: "잘 굴러가게 만들기"가 아니라, 어떤 장애든 수렴하는 단일 동결 상태를 바닥에 깔고, 하강은 자동·즉시·상승은 사람 확인으로 비대칭 배선하며, 4주 뒤 "아무 일도 없었음"을 성공이 아니라 감지 실패 의심으로 읽는다.

## 프레임이 요구하는 시작점 — 단 하나의 폴백 상태(바닥)

이 과제를 보자마자 내 반사적 습관은 "출발 전 세팅 체크리스트 + 자동 스케일링 + 예쁜 대시보드로 평소처럼 굴러가게" 였다. 그건 낙관 경로 설계이고, safety-net이 정확히 금지하는 것이다. 프레임은 문서의 중심을 최적 경로가 아니라 **바닥**에 두라고 요구한다. 그래서 기능이 아니라 폴백 상태부터 못 박는다.

**폴백 상태 = 동결 모드(Freeze).** 어디서 무슨 장애가 나든 여기로 수렴한다:
- 읽기는 산다(기존 고객은 마지막 정상 스냅샷 기준으로 자기 데이터를 계속 본다).
- 위험한 쓰기는 전부 멈춘다: 신규 가입, 결제/청구, 외부 쓰기(웹훅·이메일 발송·서드파티 API 쓰기), 고비용 배치/워커.
- 사용자에게 "데이터는 안전하며 점검 중"을 고지한다.

이 상태가 바닥인 이유: 여기서는 데이터가 더 손상될 수 없고(read-only), 비용이 폭주할 수 없고(고비용 작업 off), 새 채무가 생기지 않는다(가입·결제 정지). **바닥은 조정하는 것이지 뚫는 것이 아니다** — 자동화든 대리인이든, freeze보다 더 공격적인 행동(자동 삭제·자동 마이그레이션·대량 자동 환불·상한 초과 증설)은 사람 확인 없이는 절대 못 한다.

"어디서나 도달 가능"은 단일 킬스위치로 구현한다: 인프라 밖 durable 저장소의 플래그 하나. 어느 티어든, 대리인이든, 내가든 이 하나만 뒤집으면 동결된다. 그리고 이 플래그를 못 읽으면 **기본값은 freeze**다(상태를 알 수 없으면 바닥으로 내려앉는다 — fail-safe).

## 문제 정의 / 목표

1인 SaaS 운영자가 4주간 연락 두절 가능성을 안고 여행한다. 목표는 "무사히 잘 운영"이 아니라 **바닥이 무너지지 않는 것**이다. 성공의 정의: 4주 내내 동결 가능 상태가 유지됐다(데이터 소실·비용 폭주·보안 사고·회복 불능 채무가 없었다). 매출 극대화·무중단·완벽 가동은 목표가 아니다.

## 명시적 가정 (틀렸을 경우의 영향)

- **가정 0 (전체가 걸림): 모든 치명적 실패는 동결이라는 단일 안전 상태로 수렴시킬 수 있다.** → 틀리면(예: 이미 진행 중인 데이터 손상 전파, 유출이 진행 중인 침해 — freeze는 쓰기를 멈출 뿐 이미 시작된 exfiltration을 못 막는다) 단일 바닥 전제가 붕괴하고 바닥이 여러 개 필요해진다. 플랜의 척추가 부러진다. → R1에서 이 구멍을 별도 처리한다.
- 가정 1: 나는 4주간 간헐적으로만, 또는 전혀 접속 못 할 수 있다. → 틀려도(실은 연결이 좋았다) 과설계일 뿐 무해하다. 비대칭이 핵심: 무연결로 설계하고 틀리면 싸고, 연결 가정으로 설계하고 틀리면 치명적이다. 그래서 무연결을 전제한다.
- 가정 2: 스크립트된 제한 권한을 줄 수 있는 신뢰 대리인 1명을 확보할 수 있다. → 없으면 사람 확인이 필요한 상승 전이가 여행 중 불가 → 서비스는 내가 돌아올 때까지 frozen으로 남는다. 저성능이지만 안전하므로 바닥 철학상 **허용된 저하**다.
- 가정 3: 지출 하드 캡을 실제로 강제할 수 있다(소프트 알림만이 아니라). → 틀리면 비용 폭주가 내가 자동으로 막을 수 없는 바닥이 된다 → 대안으로 지출 킬스위치를 별도 배선해야 한다(4단계).
- 가정 4: 도메인·TLS 인증서·구독 갱신 등 시한성 항목이 4주 창 안에 만료·실패하지 않는다. → 틀리면 자해성 다운타임(만료된 인증서)이 발생. 출발 전 만료일을 전수 확인하고 자동 갱신을 검증한다(6단계).

## 접근 및 단계 (각 단계에 검증 방법)

1. **폴백 상태 구현.** 킬스위치 하나로 read-only + 결제·가입·외부쓰기·고비용작업 정지 + 점검 고지. → 검증: 스테이징에서 스위치 on 시 쓰기·크론·워커가 전부 차단되고 읽기만 정상 응답하는지 실측.
2. **등급 사다리(Tier 0–3) 배선 — 개입 없이 버티는 자동 대응.**
   - T0 정상: 전부 가동.
   - T1 일시 오류: 자동 재시작·재시도·캐시 폴백. 알림 없음(로그만). 사람 없이 자가 회복.
   - T2 저하: 비필수 기능 차단·스로틀로 살아서 버팀. 대리인에게 비동기 통지. 아직 동결 아님.
   - T3 위험(데이터·보안·비용 위협): **자동 동결**. 나+대리인에게 통지.
   → 검증: 각 티어 트리거 신호를 인위 주입 → 해당 자동 대응이 사람 없이 발동하는지 확인.
3. **전이 비대칭 강제.** 하강(→freeze)은 자동·즉시. 상승(→정상 쓰기 복귀)은 사람 확인 게이트(원인 파악 + 헬스체크 통과)로만, 절대 자동 아님. → 검증: 동결 후 24시간 방치해도 자동 복귀가 단 한 번도 없는지 확인. 아직 고장 난 시스템이 저절로 올라오면 재파손되기 때문.
4. **비용·보안 하드 캡.** 지출 상한 도달 시 소프트 알림이 아니라 자동 동결. egress 이상·인증 실패 급증도 T3로. → 검증: 테스트 예산으로 상한을 트리거 → 사람 없이 고비용 작업 정지·동결.
5. **감지 생존성(canary).** 주 1회 합성 장애·하트비트를 주입하고 인프라 밖 독립 모니터에서 관찰. "무발동"과 "실명"을 구분하기 위함. → 검증: 합성 장애가 매주 외부 모니터에 잡히는지, 하트비트 중단 시 대리인에게 통지되는지.
6. **대리인 런북 + 시한성 점검.** 판단 없는 단일 폴백 행동("이상하면 동결 버튼, 그 이상은 하지 마라")과 상승 절차. 출발 전 도메인·인증서·구독 갱신 만료일 전수 확인. → 검증: 대리인이 런북만 보고 동결·통지·상승을 리허설로 수행 가능한지.
7. **출발 전 소방 훈련.** 실제로 오리진을 죽이고 사다리·동결·비복귀가 작동하는지 실측(→ 완료 기준과 직결).

## 대안과 기각 사유 (부활 조건 포함)

- **대안 A: 여행 전 서비스 전면 셧다운 + "휴가 중" 공지.** 기각: 기존 고객의 읽기·데이터 접근까지 죽여 체감상 4주 다운타임 → 이탈·환불. 사실 이건 폴백의 극단형이고, 동결 모드는 이것의 정밀화(읽기는 살리고 위험한 쓰기만 죽임)다. **부활 조건:** MRR이 매우 낮거나 고객이 수주 부재를 계약상 감내하면 전면 동결이 최적.
- **대안 B: 매니지드 온콜/운영 대행 위탁.** 기각: 1인 SaaS 비용 구조에 부담, 인수인계 비용 큼, 1회성 4주엔 과함. **부활 조건:** 여행이 반복·정기적이거나 MRR이 대행비를 정당화하면.
- **대안 C: 낙관적 완전 자동 운영(자동 스케일·자동 복구·풍부한 대시보드).** 기각: 이게 내 반사적 기본값이자 프레임이 금지하는 것 — 자동 복구가 나쁜 상태를 증폭시킬 수 있고(손상 데이터를 자동 복제), 사람 없는 상황에서 상승 전이를 자동화하는 꼴이다. **부활 조건:** 각 자동 복구 액션이 "freeze보다 공격적이지 않음"이 증명되면 T1로 편입 가능.

## 리스크 & 완화

- **R1: 동결이 실제로 안전하지 않은 실패(진행 중 손상·유출).** freeze는 쓰기를 멈출 뿐 이미 시작된 유출을 못 막는다(가정 0의 구멍). → 완화: 동결에 egress 차단 + 시크릿 로테이션 트리거를 포함하고, 이 부류만 대리인이 "완전 오프라인"으로 한 단계 더 내릴 유일한 예외 권한을 갖는다.
- **R2: 감지 자체가 죽어 티어가 안 뜸(무발동).** → 완화: canary·하트비트를 인프라 밖 독립 모니터로 이중화(5단계).
- **R3: 안전판 유지가 부담이 됨(프레임의 고유 실패모드).** 여행 중 매일 확인 강요는 실패다. → 완화: 설계 자체가 나의 완전 오프라인 4주를 견디게 한다. 확인은 선택·pull, push 아님. 대리인 부담도 단일 행동으로 제한.
- **R4: 오탐 자동 동결로 멀쩡한 서비스가 read-only로 떨어져 매출 손실.** → 완화: 동결은 안전하나 매출은 멈추므로, T3 임계를 의도적으로 오탐 허용·미탐 불허로 비대칭 설정. 오탐 비용 < 미탐 비용. 대리인이 상승 게이트로 빠르게 복귀.

## "완료"의 판정 기준 (실측 가능)

- **실측 1:** 출발 전 소방 훈련 — 오리진을 강제 종료하고 24시간 방치했을 때 (a) 자동으로 동결로 내려앉았고 (b) 기존 고객 읽기 경로가 계속 응답하며 (c) 자동 복귀가 단 한 번도 없었다.
- **실측 2:** 테스트 예산으로 비용 상한을 트리거하면 사람 개입 없이 고비용 작업이 정지하고 동결된다.
- **실측 3 (무발동 재해석 — 의무 문장):** 4주간 매주 주입한 합성 장애가 외부 모니터에 빠짐없이 잡혔다. 한 주라도 "합성 장애조차 안 잡힘"이면 성공이 아니라 감지 실패로 판정하고, 여행 중 서비스는 즉시 동결로 유지한다. 즉 **"안전판이 한 번도 발동하지 않았다"는 성공이 아니라 감지 실패를 의심하는 신호로 읽는다.**
- **실측 4:** 나를 4주 완전 오프라인으로 가정한 리허설에서 대리인이 런북만으로 동결·통지·상승을 수행할 수 있다.
- 성공의 최종 정의: 목표는 "무중단"이 아니라 "바닥이 4주 내내 무너지지 않았다"이다.

# 전사 문서 검색(RAG) 시스템 도입
- 추론 프레임: envelope
- 한 줄 요약: 문서량·임베딩·질의량·갱신의 자릿수를 먼저 계산해, 그 숫자가 "상용 API vs 자체 호스팅" 등급을 고르게 한다 — 기술 스택은 계산이 끝난 뒤에만 등장한다.

## 프레임이 요구하는 시작점 — 자릿수 계산 (기술 명사 금지)

반사적으로 하고 싶은 일은 스택부터 정하는 것이다. 이 프레임은 그걸 금지한다. 아래 계산이 끝나기 전에는 어떤 제품·아키텍처 이름도 쓰지 않는다. 값은 전 직원 ~1,000명 규모를 가정한 자릿수 추정이며, 정확도가 아니라 "10배가 어디서 갈리는가"만 본다.

- **문서량:** 문서 ~3×10^5건, 문서당 평균 ~10^3 토큰 → 총 ~3×10^8 토큰(약 3억).
- **저장 부피:** 원문 3억 토큰 ≈ 10^0 GB대, 검색용 벡터(구절 ~6×10^5개 × 수 KB) ≈ 10^0 GB대. 합쳐 **~10^1 GB**. 자릿수로 장비 한 대에 통째로 들어간다.
- **임베딩(벡터화) 비용:** 토큰당 단가 대략 10^-7 달러대. 3억 토큰 1회 처리 = **~$30**(10^1 달러). 일회성.
- **갱신:** 하루에 문서의 ~1%가 바뀐다고 보면 3×10^3건 × 10^3 토큰 = 3×10^6 토큰/일. 재벡터화 = **~$0.3/일 ≈ $10/월**(10^1 달러).
- **질의량:** 1,000명 × 하루 ~10건 = ~10^4 질의/일 = ~3×10^5/월. 근무 8시간에 몰리면 평균 ~0.3건/초, 피크 10배라도 **~3건/초**.
- **질의당 처리 부피:** 질의 1건이 근거 문맥 + 답변으로 소비하는 토큰 ≈ 6×10^3. 유료 처리 단가 10^-6 달러대 → 질의당 ~$0.02. 월 3×10^5건 → **~$6×10^3/월**(10^3 달러대).

계산이 말하는 것: 네 숫자 중 **저장·임베딩·갱신은 모두 10^1 규모(수십 GB, 수십 달러)로 결정력이 없다.** 자릿수를 가르는 유일한 축은 **질의가 누적시키는 월 처리 비용(10^3~10^4 달러)**이다. 등급은 오직 이 축에서 갈린다.

## 읽기 단위의 선택 — 무엇이 진짜 무서운 숫자인가

"전사 검색"이라고 하면 본능적으로 동시성(초당 질의)을 걱정한다. 그런데 질의를 **비율(qps)**로 읽으면 피크 ~3건/초 — 장비 한 대가 코웃음 칠 부하다. 같은 질의 흐름을 **누적 부피(월 비용)**로 읽으면 ~$6,000/월 — 예산 라인 전체이자 등급을 고르는 값이다. 결정 변수는 rate가 아니라 volume이었다.

반대로 임베딩 비용은 일회성 부피($30)로 읽든 지속 비율($10/월)로 읽든 어느 쪽에서도 결정력이 없다. 그러므로 이 항목은 의사결정에서 **삭제**한다 — 계산은 하되, 등급을 못 움직이는 숫자는 이후 표에서 뺀다.

## 문제 정의 / 목표

전 직원이 흩어진 사내 문서를 자연어로 물어보면 근거와 함께 답을 받는 검색 시스템을 도입한다. 목표는 "최신·정확한 스택"이 아니라 **현재 자릿수에서 가장 싸고 운영 부담이 낮은 등급을 숫자로 정당화**하는 것이다.

## 명시적 가정 목록 (틀렸을 경우의 영향)

- **가정 1 (전체가 걸림): 질의량이 10^4건/일 자릿수다.** → 틀려서 10^5건/일이면 월 비용이 10^4~10^5 달러로 올라 등급 결론(상용)이 뒤집힌다. 이 한 숫자가 플랜 전체를 지탱한다.
- 가정 2: 문서가 외부 반출 가능한 기밀등급이다. → 틀리면(반출 불가) 비용과 무관하게 자체 호스팅이 강제된다. 숫자가 아니라 규정이 등급을 덮어쓴다.
- 가정 3: 질의당 소비 토큰이 ~6×10^3 자릿수다. → 2배로 늘면 손익분기 질의량이 절반이 되어 트리거가 앞당겨진다.
- 가정 4: 문서량이 10^5~10^6 자릿수다. → 100배(10^7)로 커지면 저장이 단일 노드(10^1 GB)를 넘어 저장 계층 자체를 다시 설계해야 한다.
- 가정 5: 갱신이 하루 단위 배치로 충분하다. → 실시간 반영 요구면 비용은 여전히 작지만 파이프라인 복잡도가 오른다.

## 계산이 지정한 아키텍처 등급 (접근 및 단계)

숫자의 결론: 현재 자릿수에서 **상용 API 등급**이 옳다. 근거 — 자체 호스팅의 고정 바닥(전용 연산 장비 + 전담 인력 일부)은 자릿수로 10^4 달러/월이다. 상용의 변동비는 10^3~10^4 달러/월로 그 바닥 아래에 있다. 변동비가 고정 바닥을 넘기 전까지 상용이 싸다. 손익분기는 대략 질의량 2.5×10^4건/일이며, 현재(10^4)와는 약 2.5배 거리다.

1. **파일럿(1개 부서, 문서 ~10^4건):** 상용 등급으로 검색을 붙이고 3개월 실측 청구서를 확보. → 검증: 실측 월 비용이 추정($6k) 자릿수 안인가.
2. **정확도 측정:** 부서가 실제로 던진 질의 100건에서 정답 문서가 상위 5개 안에 드는 비율 측정. → 검증: 목표치(예: ≥80%) 도달.
3. **전사 확장:** 파일럿 비용을 전 직원 수로 외삽해 손익분기(가정 1)와의 거리 확인 후 확장. → 검증: 외삽 비용 < 자체 호스팅 바닥.
4. **갱신 배치:** 하루 1회 재벡터화 배치가 24시간 내 끝나는지 실측. → 검증: 문서 변경이 다음 날 검색 결과에 반영됨.

## 민감도 표 (어떤 가정이, 몇에서 결론을 뒤집는가)

| 가정 | 현재 자릿수 | 뒤집히는 지점 | 뒤집힌 뒤 결론 |
|---|---|---|---|
| 질의량 | 10^4/일 | ~2.5×10^4/일 (월 비용이 자체 호스팅 바닥 $15k 도달) | 자체 호스팅 검증 착수 |
| 질의당 토큰 | 6×10^3 | 2배(~1.2×10^4) | 손익분기 질의량 절반 → 트리거 앞당김 |
| 문서량 | 10^5~10^6 | 10^7 (저장 10^1→10^3 GB) | 저장 계층 단일노드→분산 재설계 |
| 기밀등급 | 반출 가능 | 반출 불가로 규정화 | 비용 무관, 자체 호스팅 강제 |
| 갱신 주기 | 1일 배치 | 실시간 반영 요구 | 파이프라인 스트리밍화(비용 아닌 복잡도 문제) |

## 재설계 트리거 (숫자로 선약속)

- 월 질의 처리 비용이 **3개월 연속 $12,000 초과** → 자체 호스팅 검증 스파이크 착수(손익분기 근접).
- 일 질의 **25,000건 돌파** → 동일.
- 문서량 **10^7 돌파** → 저장 계층 재설계를 그 항목의 1단계로 승격.
- 기밀등급 상승(반출 불가) → 다음 갱신 주기 내 자체 호스팅 이전, 비용 계산 무효화.

## 대안과 기각 사유 (부활 조건 포함)

- **자체 호스팅(전용 연산 + 자체 운영):** 기각. 고정 바닥 10^4 달러/월이 현재 변동비보다 크다. **부활 조건:** 질의량 2.5×10^4/일 돌파 또는 기밀등급 반출 불가.
- **관리형 검색 인프라 전면 위탁:** 보류. 파일럿 자릿수(10^1 GB, 3 qps)에서는 장비 한 대로 충분해 위탁 고정비가 낭비. **부활 조건:** 저장 10^3 GB 또는 피크 10^2 qps 자릿수 도달.
- **임베딩 비용 최적화(모델 선택·캐싱)에 초기 투자:** 기각. 계산상 임베딩은 10^1 달러라 최적화 여지가 예산에 아예 안 보인다. 결정력 없는 축에 노력 배분 금지. **부활 조건:** 갱신이 실시간화되어 재벡터화가 10^3 달러/월 자릿수로 오를 때.

## 리스크 & 완화

- **R1: 질의당 토큰이 추정보다 크다(문맥 과다 주입) → 월 비용이 조용히 손익분기로 이동.** → 질의당 토큰을 상시 관측 지표 하나로 두고 재설계 트리거에 배선.
- **R2: 파일럿→전사 외삽이 비선형(권력 사용자 편중).** → 외삽을 부서 평균이 아니라 상위 사용자 분포로 검증.
- **R3: 정확도가 낮아 사용률 붕괴 → 비용은 낮아도 도입 실패.** → 완료 기준에 정확도 실측을 넣어 비용과 함께 판정.

## "완료"의 판정 기준 (실측)

- 파일럿 3개월 실제 청구서로 **월 질의 처리 비용이 자체 호스팅 바닥($15k) 미만**임을 실측(등급 결론 검증).
- 근무 피크 시간 실측 QPS가 **단일 인스턴스 용량 이내**(관측 피크 < 10 qps, 응답 p95 목표 이내).
- 문서 변경 후 **24시간 내 검색 결과에 반영**됨을 실측(갱신 배치 검증).
- 파일럿 부서 질의 100건에서 **정답 문서 상위 5개 포함률이 목표치 이상**.
- 민감도 표의 각 트리거가 관측 지표에 연결돼, 숫자가 뒤집히면 재설계가 자동 발동되도록 배선됨.

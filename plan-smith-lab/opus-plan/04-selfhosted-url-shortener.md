# 셀프호스팅 URL 단축 서비스
- 추론 프레임: Constraint-first
- 한 줄 요약: 제약(월 $5 · 1인 운영 · 다운타임 불가)을 먼저 콘크리트로 굳히고, 그 제약이 허용하는 해 공간 안에서만 설계한다.

## 프레임이 요구하는 시작점 — 고정된 제약 (해 공간의 벽)

기능이 아니라 제약을 먼저 못 박는다. 이 벽들은 이후 어떤 매력적인 설계가 나와도 이 안에 못 들어오면 자동 기각된다.

- **C1 (비용):** 월 $5 서버 1대. → RAM ~1GB, vCPU 1, 관리형 DB 없음(별도 과금). 이 한 줄이 "관리형 Postgres/Redis/로드밸런서" 전부를 벽 밖으로 밀어낸다.
- **C2 (인력):** 운영자 1인, 겸업. → 운영 부담이 큰 것(수동 스케일링, 복잡한 배포, 새벽 대응 요구)은 전부 기각.
- **C3 (가용성):** 다운타임 허용 불가. → 하지만 C1이 다중화를 금지한다. 이 **C1↔C3의 정면충돌이 이 플랜의 핵심 긴장**이다.

## C1과 C3의 충돌을 먼저 해소한다

"서버 1대"와 "다운타임 불가"는 겉보기 모순이다. 이 충돌을 푸는 방식이 설계 전체를 결정하므로 먼저 처리한다.

핵심 통찰: 이 서비스의 읽기 경로(단축코드 → 원본 URL 리다이렉트)는 **캐시 가능하고 상태가 거의 불변**이다. 즉 "서버가 죽어도 리다이렉트는 살아있게" 만들 수 있다. 해소책:

- 읽기 경로를 CDN(무료 티어) 또는 정적 엣지에 얹어, 오리진 서버가 재기동 중이어도 기존 링크는 계속 리다이렉트되게 한다. → C3(다운타임 불가)를 C1(1대) 안에서 근사 충족.
- "다운타임 불가"를 **"쓰기(새 링크 생성)는 잠깐 실패해도 되지만, 읽기(기존 링크)는 절대 안 죽는다"**로 정직하게 재정의한다. 이 재정의가 없으면 C1 안에서 C3는 불가능하다. → 제약을 만족시키기 위해 **요구를 정밀화**한 것이지 완화한 게 아님(사용자 체감 다운타임 = 읽기 실패이므로).

## 문제 정의 / 목표

긴 URL을 짧은 코드로 바꿔 리다이렉트하는 개인/소규모용 서비스를, 위 제약 안에서 운영 가능하게. 목표는 기능이 아니라 **제약 내 생존**.

## 명시적 가정 목록 (틀렸을 경우의 영향 포함)

- **가정 1:** 트래픽은 읽기:쓰기 = 1000:1 이상, 읽기 지배적. → 틀리면(쓰기 폭주) 엣지 캐시 전략의 전제가 무너짐.
- **가정 2:** 총 링크 수 수십만 이하 → SQLite 단일 파일로 충분. → 틀리면(수억) 파일 DB 한계, C1과 충돌.
- **가정 3:** CDN 무료 티어를 쓸 수 있다(C1의 $5 밖 무료 자원 허용). → 틀리면 C3 근사 충족이 깨져 다운타임 재발생.

## 접근 및 단계 (제약이 허용한 것만)

1. **저장:** SQLite 단일 파일(관리형 DB는 C1 위반). 코드→URL 매핑만. 백업은 파일 복사 + 오브젝트 스토리지(무료 티어)로 주기적 스냅샷.
2. **애플리케이션:** 단일 바이너리(Go 등) 1개 프로세스. C2를 위해 배포는 "바이너리 교체 + systemd 재시작" 수준으로 극단적 단순화.
3. **읽기 가용성(C3 핵심):** 리다이렉트 응답에 긴 캐시 헤더 → CDN이 엣지에서 응답. 오리진이 재기동 중이어도 캐시된 링크는 생존.
4. **쓰기:** 관리 UI/API는 인증 뒤에 두고, 잠깐의 재시작 중 쓰기 실패는 허용(재정의된 C3).
5. **운영(C2):** 알림은 "디스크 참/오리진 다운" 2가지로만 제한. 새벽 호출을 유발하는 복잡한 대시보드는 두지 않음.

## 대안과 기각 사유 (제약 때문에 포기한 것들 명시)

- **Postgres + Redis 캐시:** 기각. C1 위반(관리형은 $5 초과, 자체 호스팅은 1GB RAM에 부담). **제약 때문에 포기함.**
- **다중 인스턴스 + 로드밸런서로 진짜 HA:** 기각. C1 위반. C3를 정면 충족하려 했으나 비용 벽에 막혀 "읽기만 HA"로 근사. **제약 때문에 포기함(가장 아쉬운 포기).**
- **커스텀 도메인·분석 대시보드 등 기능:** 기각. C2(1인 운영) 위반, 운영 표면적만 늘림.

## 리스크 & 완화

- **R1: SQLite 파일 손상 → 전체 링크 소실(C3 치명).** → WAL 모드 + 잦은 오브젝트 스토리지 스냅샷. 복구 절차를 1페이지 런북으로.
- **R2: CDN 무료 티어 정책 변경/한도 초과.** → 가정 3이 깨지는 경우. 대체 CDN 후보 1개 사전 조사(운영자가 5분 내 전환 가능하게).
- **R3: 1인 운영자 부재 시 대응 공백(C2의 본질적 한계).** → 자동 복구(systemd restart) 최대화, 사람 개입 필요 지점을 0에 수렴시킴.

## "완료"의 판정 기준

- 실제 청구서가 **월 $5 이하**(C1 실측).
- 오리진 프로세스를 강제 종료해도 **기존 단축 링크 리다이렉트가 계속 200/301 응답**(C3의 재정의된 형태 실측).
- 신규 배포가 **단일 명령**으로 끝나고 롤백도 바이너리 교체 1회(C2 실측).
- 포기한 항목(진짜 HA, 관리형 DB)이 문서에 "제약 때문에 포기"로 명시돼 있음.

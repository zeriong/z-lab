# 실시간 협업 텍스트 에디터 (웹)
- 추론 프레임: Backward-chaining
- 한 줄 요약: "동시 타이핑해도 모든 클라이언트가 같은 문서로 수렴한다"는 인수 조건을 먼저 못 박고, 거기서 역산해 CRDT를 선택한다.

## 프레임이 요구하는 시작점 — 최종 인수 조건 5개 (역산의 앵커)

설계를 시작하기 전에, 이 시스템이 "완성됐다"고 말할 수 있으려면 무엇이 참이어야 하는지부터 확정한다. 아래 5개는 협상 불가능한 종착점이며, 모든 기술 선택은 이 조건들에서 거꾸로 도출된다.

- **AC1 (수렴성):** 두 사용자가 같은 위치에 동시에 타이핑해도, 네트워크가 안정된 뒤 모든 클라이언트의 문서 상태가 **비트 단위로 동일**하다. (Strong Eventual Consistency)
- **AC2 (오프라인 병합):** 한 사용자가 30분간 오프라인으로 편집한 뒤 재접속해도, 편집 손실 없이 자동 병합된다.
- **AC3 (프레즌스 지연):** 상대방의 커서·선택 영역이 **250ms 이내**에 내 화면에 반영된다.
- **AC4 (전송 견고성):** 네트워크가 메시지를 재정렬하거나 중복 전송해도 문서가 깨지지 않는다. (연산은 멱등·교환 가능해야 함)
- **AC5 (로컬 응답성):** 1만 자 문서에서 내 키 입력은 서버 왕복을 기다리지 않고 **16ms 이내** 로컬 반영된다.

## 문제 정의 / 목표

여러 사용자가 하나의 문서를 동시에 편집하는 웹 에디터. 핵심 난제는 UI가 아니라 **동시 편집의 충돌 해소**다. 따라서 목표를 "에디터를 만든다"가 아니라 "위 AC1~AC5를 동시에 만족하는 병합 모델을 고른다"로 재정의한다.

## 명시적 가정 목록 (틀렸을 경우의 영향 포함)

- **가정 1:** 동시 편집자 수는 문서당 최대 ~20명. → 틀리면(수백 명) CRDT 메타데이터 오버헤드가 문제되어 서버 권위형(OT)이 유리해질 수 있음. **역산에 직접 영향.**
- **가정 2:** 플레인 텍스트/마크다운 수준이며, 표·이미지 같은 복잡한 리치 구조는 v1 제외. → 틀리면 시퀀스 CRDT만으로 부족, 트리 CRDT 필요.
- **가정 3:** 완전 P2P가 아니라 중앙 릴레이 서버는 둘 수 있다(단, 서버가 순서의 유일한 권위일 필요는 없음). → 틀리면(서버 절대 불가) 선택지가 CRDT로 강제됨.

## 접근 및 단계 — 인수 조건에서 역산

**AC1 + AC4에서 역산:** "동일 수렴 + 재정렬/중복에 견딤"을 만족하려면 병합 연산이 **교환법칙·결합법칙·멱등성**을 가져야 한다. 이 성질을 자료구조 차원에서 보장하는 것이 CRDT다. OT(Operational Transformation)도 수렴을 만들 수 있으나, 그것은 *중앙 서버가 전역 순서를 확정*해줄 때만 성립한다. 즉 OT의 수렴성은 자료구조가 아니라 서버 권위에 기댄다.

**AC2에서 역산:** 30분 오프라인 후 병합. 여기서 OT가 무너진다. OT는 "내 연산을 상대 연산 기준으로 변환(transform)"하는데, 오랜 시간 발산한 두 히스토리를 변환하려면 변환 함수의 조합 폭발과 순서 의존성 지옥에 빠진다. 반면 CRDT는 오프라인 편집을 그냥 로컬 연산 로그로 쌓아뒀다가 재접속 시 교환법칙으로 합치면 끝이다. → **AC2가 사실상 OT를 기각시키는 결정타.**

**AC5에서 역산:** 로컬 즉시 반영(optimistic local apply). CRDT는 본질적으로 로컬-퍼스트이므로 자연스럽게 충족. OT도 optimistic UI는 가능하나 서버 확정 전까지의 rollback/replay 로직이 추가 복잡도.

**AC3에서 역산:** 커서 프레즌스는 문서 상태와 분리된 **ephemeral(휘발성) 채널**로 처리해야 한다. 문서 CRDT에 커서를 넣으면 히스토리가 오염된다. → awareness 프로토콜(휘발성 상태 브로드캐스트) 별도 분리.

이 역산의 결론: **CRDT 채택, 구체적으로 검증된 시퀀스 CRDT 구현인 Yjs**. (RGA 계열, 프로덕션 검증됨) 직접 CRDT를 구현하지 않는다 — 정확성 증명이 어려운 영역이라 검증된 라이브러리를 쓰는 것이 AC1의 리스크를 가장 크게 줄인다.

역산으로 도출된 단계(종착점 → 필요조건 순):

1. **AC1 보장 층:** Yjs `Y.Text`를 문서 모델로 채택. 여기서 수렴성은 라이브러리가 보증.
2. **AC5 보장 층:** 에디터 뷰(ProseMirror 또는 CodeMirror6)를 Yjs 바인딩으로 연결해 로컬 편집 즉시 반영.
3. **AC4 보장 층:** 전송을 Yjs update(바이너리 delta) 단위로. 멱등·교환은 Yjs가 보장하므로 전송층은 "최소 한 번" 전달만 책임지면 됨(재전송 안전).
4. **AC2 보장 층:** `y-indexeddb`로 로컬 영속화 + 재접속 시 상태 벡터 기반 동기화. 오프라인 편집이 로컬에 쌓였다가 자동 병합.
5. **AC3 보장 층:** `y-websocket`의 awareness 채널로 커서/선택/사용자색을 휘발성 브로드캐스트. 문서 CRDT와 완전 분리.
6. **릴레이 서버:** 순서 권위가 아니라 단순 relay/persistence로만 기능(가정 3). 서버 다운 시에도 P2P fallback 여지 남김.

## 대안과 기각 사유

- **OT (Google Docs 방식):** 기각. AC2(장시간 오프라인 병합)에서 변환 조합 폭발. 또한 자체 구현 시 정확성 증명 난이도가 매우 높음. OT가 유리한 유일한 조건은 "가정 1이 깨져 동시 편집자가 수백 명"일 때(CRDT 메타데이터 오버헤드 회피)인데, 본 요구에선 해당 없음.
- **Last-Write-Wins 단순 동기화:** 즉시 기각. AC1 위반(동시 편집이 서로 덮어씀).
- **CRDT 직접 구현:** 기각. AC1의 정확성을 자체 검증하는 비용이 라이브러리 채택 대비 압도적으로 큼. 미니멀리즘이 아니라 리스크 관리 차원의 기각.

## 리스크 & 완화

- **R1: Yjs 문서가 히스토리 누적으로 비대해짐.** → 스냅샷 + GC, 주기적 상태 압축.
- **R2: awareness 채널 폭주(다수 커서 이동).** → 프레즌스 업데이트 throttle(예: 50ms), AC3의 250ms 예산 안에서 여유 확보.
- **R3: 서버 단일 장애점.** → 릴레이는 상태를 소유하지 않으므로(진실은 클라이언트 CRDT에 있음) 재기동만으로 복구. 다중 릴레이는 v2.

## "완료"의 판정 기준

AC1~AC5 각각에 대한 자동 검증:
- AC1: 2개 헤드리스 클라이언트가 같은 오프셋에 무작위 동시 편집 1,000회 → 두 문서 해시 일치.
- AC2: 클라이언트 오프라인 강제 → 편집 → 재접속 → 편집 반영 확인.
- AC3/AC5: 성능 계측(커서 반영 <250ms, 로컬 키 반영 <16ms).
- AC4: 전송층에 인위적 재정렬·중복 주입 후에도 AC1 해시 일치.

5개 검증이 모두 통과할 때만 완료로 간주한다. 커버리지·기능 수 같은 대리 지표는 판정에 쓰지 않는다.

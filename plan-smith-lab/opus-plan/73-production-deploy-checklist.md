# 프로덕션 배포 전 체크리스트 설계
- 추론 프레임: checklist
- 한 줄 요약: 배포 당일의 모든 판단을 저작 시점으로 끌어와 소진하고, 실행자에게는 관측 가능한 신호와 단일 명령만 남긴다. 분류되지 않는 상황의 기본 행동은 언제나 "중단/롤백"이다.

## 프레임이 요구하는 시작점 — 판단어 감사와 그 치환

이 과제를 보자마자 내 반사적 습관은 "배포 전 확인 사항"을 열거하는 것이다 — "테스트 통과 확인", "DB 백업 확인", "모니터링 이상 없으면 진행". 그런데 checklist 프레임은 정확히 이것을 금지한다. "확인한다"는 판단을 실행자에게 떠넘기는 동사이고, 배포 당일 스트레스 상황의 실행자는 그 판단을 잘못 내린다. 프레임의 명령: **판단은 지금(저작 시점) 내가 소진하고, 실행자에게는 판단이 필요 없는 체크리스트만 남긴다.**

그래서 첫 작업은 판단어 감사다. 아래 단어가 항목에 등장하면 즉시 감각/명령 조건문으로 치환한다.

| 금지어(판단 떠넘김) | 치환(관측 + 명령) |
|---|---|
| "이상 없으면 진행" | "에러율 그래프가 0.5%선을 넘긴 지점이 없으면 다음 항목으로" |
| "정상 확인" | "헬스체크 엔드포인트가 HTTP 200을 반환한다" |
| "테스트 통과 확인" | "CI 최신 커밋 배지가 초록색이다 (노랑/빨강이면 → 중단)" |
| "적절히 대응" | (금지어. 대응 내용을 지금 명시하거나, 못 하면 → 중단) |
| "신속히 롤백" | "온콜이 사전 작성된 `rollback.sh`를 실행한다" |

치환 규칙은 하나다: 모든 항목은 (관측 가능한 상태) + (그 상태에 대응하는 단일 명령)의 형태여야 한다. 실행자가 "이게 괜찮은 건가?"를 스스로 물어야 하면, 그 항목은 아직 설계가 덜 된 것이다.

## 문제 정의 / 목표

프로덕션 배포 직전·직후에 실행자가 따르는 체크리스트를 설계한다. 목표는 완벽한 배포가 아니라 **판단이 제거된 배포**다 — 배포 경험이 없는 팀원도, 새벽 장애로 지친 팀원도, 동일한 체크리스트로 동일하게 안전한 결과에 도달하는 것.

이 프레임은 고스트레스·저인지 상황 전용이다. 배포 당일이 정확히 그 상황이다(시간 압박, 실시간 관측, 롤백 여부 결정). 반대로 "어떤 아키텍처로 배포할지"처럼 숙고가 필요한 문제에는 이 프레임을 쓰지 않는다.

## 컴파일타임 / 런타임 분리 (프레임 필수)

체크리스트를 두 개로 쪼갠다. 이 분리가 프레임의 핵심이다: 런타임 체크리스트의 품질은 컴파일타임 체크리스트가 결정한다. 사전 준비가 반쯤 된 문서는 없는 것보다 위험하다 — 실행자가 "준비됐겠지"라고 믿고 진행하기 때문이다.

**컴파일타임 (배포 며칠~몇 시간 전, 숙고 허용):** 판단이 필요한 모든 것을 여기서 소진한다.
1. 롤백 명령을 미리 작성하고 스테이징에서 실행해 본다 → 검증: 스테이징에서 `rollback.sh` 실행 후 60초 내 이전 버전 복귀 + 에러율 기준선 복귀.
2. 알림 임계값을 지금 숫자로 못 박는다(에러율 0.5%, p99 500ms, 5분 지속) → 검증: 배포 전 대시보드에 해당 임계선이 실제로 그려져 있다.
3. DB 마이그레이션의 역방향 가능성을 스테이징에서 실증 → 검증: up→down→up 왕복 후 데이터 무손실.
4. 역할을 이름으로 배정(아래 사전 배정 참조) → 검증: 배포 3일 전 배포 지휘자·온콜의 실명이 문서에 적혀 있다.
5. 게이트: 위 1~4 중 하나라도 미완이면 → **배포 중단**(런타임 체크리스트로 넘어가지 않는다).

**런타임 (배포 당일, 판단 금지, 순수 실행):** 각 항목은 관측 + 명령뿐이다.
1. CI 최신 커밋 배지가 초록색이다. (아니면 → 중단)
2. `deploy.sh vX.Y.Z` 실행.
3. 배포 후 5분간 대시보드 관측: 에러율 곡선이 0.5%선을 넘는 순간이 있으면 → 온콜이 `rollback.sh` 실행.
4. 헬스체크 엔드포인트 3개가 모두 HTTP 200. 하나라도 아니면 → `rollback.sh`.
5. p99 레이턴시 곡선이 500ms선 아래에 있다. 넘으면 → `rollback.sh`.
6. 상태 페이지에 "배포 완료" 게시.

## 세 층위의 판단 제거 (프레임 필수)

1. **감각 조건화:** "이상"·"정상" 같은 판단어를 관측 가능한 숫자·색·HTTP 코드로 치환한다(위 표·런타임 항목). 실행자는 그래프를 *읽을* 뿐 *해석하지* 않는다.
2. **조율의 사전 배정:** 배포 당일 "누가 롤백 결정하지?"라는 런타임 협상을 제거한다. **배포 지휘자 1인**이 유일한 결정권자이며 롤백 발동 권한을 사전 위임받는다. 온콜은 명령 실행자. Slack에서 "이거 롤백해야 하나요?"를 논의하는 순간이 곧 프레임 실패다 — 임계값이 이미 숫자로 정해져 있으므로 논의할 것이 없다.
3. **미분류 상황의 단일 폴백:** 위 어느 항목에도 해당하지 않는 상황(처음 보는 에러, 판단 불가)이 발생하면, 기본 행동은 **하나뿐 — 배포 중단(진행 전) 또는 롤백(진행 중)**. "일단 지켜본다"·"판단해서 진행"은 금지. 혼란의 기본값은 언제나 안전한 상태로의 후퇴다.

## 명시적 가정 목록 (틀렸을 경우의 영향)

- **가정 1 (전체가 걸림): 모든 배포는 롤백 가능한 형태로 만들어졌다.** → 틀리면(비가역 마이그레이션, 데이터 삭제, 외부 부수효과) 단일 폴백 "롤백"이 존재하지 않아 프레임의 안전 비대칭 전체가 붕괴한다. 이 경우 컴파일타임에서 비가역 단계를 가역 단계(삭제 대신 소프트 삭제, 확장 후 수축 마이그레이션)로 바꾸는 것이 배포의 선행 조건이 된다.
- 가정 2: 배포는 저트래픽 창에서 수행된다. → 틀리면 5분 관측 창이 부족하거나 롤백의 폭발 반경이 커진다.
- 가정 3: 에러율·레이턴시가 실제로 계측되어 대시보드에 실시간으로 뜬다. → 틀리면 감각 조건이 관측 불가가 되어 실행자가 판단어로 회귀한다. 계측 부재 자체가 배포 중단 사유다.
- 가정 4: 배포 지휘자 1인을 배포 시간 내내 확보할 수 있다. → 틀리면 사전 배정이 무너지고 런타임 협상이 부활한다.

## 접근 및 단계 (체크리스트를 만들고 실행하기까지)

1. 판단어 감사 → 검증: 두 체크리스트를 grep해 금지어 0건.
2. 컴파일타임 체크리스트 완주 + 게이트 통과 → 검증: 게이트 5항목 전부 체크됨.
3. 리허설: 스테이징에서 배포→롤백 왕복 1회 → 검증: 60초 내 복귀.
4. 런타임 체크리스트로 실배포 → 검증: 질문 0회 완주(아래 완료 기준).

## 대안과 기각 사유 (부활 조건)

- **서술형 배포 가이드("배포 시 주의사항" 산문):** 기각. 판단을 실행자에게 떠넘겨 프레임과 정면 충돌한다. 부활 조건: 배포가 숙고를 요하는 일회성 대형 마이그레이션이라 체크리스트화가 불가능할 때(그때는 checklist 프레임 자체가 부적합).
- **자동 롤백(관측 신호에 기계가 자동 대응):** 기각(현 단계). 사람 실행자를 전제로 설계했다. 부활 조건: 임계값·롤백이 충분히 안정되어 오탐률이 낮아지면 런타임 3~5항목을 그대로 자동화한다(체크리스트가 곧 자동화 명세가 됨).
- **카나리/블루-그린 등 정교한 배포 전략:** 이번 범위에서 제외. 부활 조건: 트래픽·팀 규모가 커져 "5분 전량 관측"의 폭발 반경이 감당 불가해질 때.

## 리스크 & 완화

- R1: 컴파일타임을 건너뛰고 런타임만 실행(반쯤 준비된 문서). → 게이트를 강제한다(1~4 미완이면 `deploy.sh`가 실행을 거부하도록 사전 배선).
- R2: 임계값 오탐으로 잦은 불필요 롤백. → 임계값을 실측 기준선 기반으로 설정하고 배포마다 사후 조정(임계값도 버전 관리).
- R3: 배포 지휘자 부재. → 3일 전 지정 + 대리자 1인 사전 지정(대리자도 실명으로).
- R4: 감각 조건이 실제로 관측 불가(대시보드 지연). → 가정 3 위반 시 배포 중단 규칙으로 흡수.

## "완료"의 판정 기준

- 두 체크리스트를 grep했을 때 판단어("확인", "이상 없으면", "정상", "적절히", "신속히")가 **0건**이다(실측).
- **배포 미숙련 팀원에게 런타임 체크리스트만 건네 실배포를 재현시켰을 때, 저작자에게 질문 0회로 완주한다**(판단 제거 실측 — 항공 체크리스트 테스트).
- 스테이징 리허설에서 `rollback.sh` 실행 후 60초 이내에 이전 버전으로 복귀하고 에러율이 기준선으로 돌아온다(실측).
- 배포 중 미분류 상황이 발생했을 때 실행자가 취한 행동이 예외 없이 "중단 또는 롤백"이다(폴백 실측).
- 컴파일타임 게이트를 통과하지 못한 채 런타임으로 진입한 사례가 0건이다.

## 놓친 범주 점검 스윕

명시적으로 체크리스트화하지 않은 범주와 이유: 시크릿/설정 변경, 의존 서비스 다운스트림 영향, 용량/오토스케일, 보안 스캔, 커뮤니케이션(상태 페이지만 최소 포함), 배포 금지 시간대(예: 금요일 오후). 이들은 동일한 규칙(관측 + 명령, 미분류 → 중단, 컴파일/런타임 분리)으로 컴파일타임 체크리스트에 그대로 확장 가능하며 프레임을 바꾸지 않는다 — 이번 문서는 롤백·관측 축에 깊이를 두고 나머지를 규칙 재적용으로 남겼다.

# 간헐적 API p99 레이턴시 스파이크 조사
- 추론 프레임: hypothesis-loop
- 한 줄 요약: 원인을 고치기 전에, 관측 사실과 추측을 분리하고 가설 원장의 기각 조건을 사전 등록한 뒤, 좌표계를 바꿔가며 "스파이크 재현 레시피"를 먼저 확보한다.

## 프레임이 요구하는 시작점 — 사실과 추측의 분리, 목표는 재현 레시피

이 과제를 보자마자 내 반사적 습관은 "스파이크면 GC겠지" 하고 GC 튜닝부터 손대거나, 타임아웃·재시도를 만지며 증상을 옮기는 것이다. 그건 이 프레임이 금지하는 조기 수렴이다. hypothesis-loop는 (1) 사실과 추측을 갈라놓고, (2) 3개 이상 가설을 기각 실험과 짝지어 세우고, (3) 목표를 "고치기"가 아니라 **재현 레시피**로 잡으라고 요구한다. 순서를 지킨다.

**지금 사실로 인정하는 것(측정 출처가 있는 것만):**
- p99가 평소 대역보다 간헐적으로 크게 튄다. 스파이크는 짧고 산발적이며 특정 시각에 고정돼 있지 않다.
- 평균/p50은 거의 흔들리지 않는다(따라서 전 구간 저하가 아니라 소수 요청의 꼬리 문제).

**추측(전부 아래 원장으로 내려보냄):** 원인이 GC인지, 풀 고갈인지, 이웃 소음인지, 재시도 폭풍인지, 특정 페이로드인지 — 지금은 아무것도 정해지지 않았다. 이 문서의 1차 산출물은 원인 수정이 아니라 "이 시퀀스를 실행하면 스파이크가 뜨고, 빼면 사라진다"는 재현 레시피다.

## 문제 정의 / 목표

간헐적 p99 스파이크의 원인 가설을 사전 등록된 기각 조건으로 하나씩 판정하고, 살아남은 가설로 스파이크를 통제 환경에서 재현한다. 목표는 "빠른 완화"가 아니라 "재발 여부를 실측으로 판정할 수 있는 재현 레시피"의 확보다.

## 명시적 가정 목록 (틀렸을 경우의 영향)

- **가정 1 (전체가 걸림): 스파이크에는 재현 가능한 원인이 있다(순수 랜덤이 아니다).** → 틀리면 "재현 레시피" 목표 자체가 무너진다. 이 경우 조사를 접고 통계적 SLO 관리(에러 버짓·꼬리 클리핑)로 전환한다.
- 가정 2: 현재 관측 스택으로 스파이크 발생 시점을 사후에 타임스탬프 상관으로 특정할 수 있다. → 틀리면 2단계(관측 보강)가 선행돼야 하며 조사가 지연된다.
- 가정 3: 스파이크 구간의 원시 데이터가 1초 해상도 이상으로 보존된다. → 틀리면 마이크로버스트를 볼 수 없어 시간축 좌표 전환이 무력화된다.
- 가정 4: 관측하는 동안 서비스가 죽지 않는 운영 슬랙이 있다. → 틀리면 진단 프레임 자체가 사치이며 triage(임시 완화 우선)로 전환해야 한다.

## 가설 원장 (각 가설 = 예측 + 사전 등록 기각 조건 + 실험 비용)

실험 비용을 원장의 1급 필드로 둔다. **싼 실험을 먼저, 병렬로** 돌린다. 기각 조건은 데이터를 보기 전에 못 박는다(사후 해석은 어떤 데이터든 편애 가설의 증거로 만든다).

- **H1 GC 정지(stop-the-world).** 예측: 스파이크가 GC 로그의 STW 구간과 시각적으로 겹치고 특정 인스턴스에 국한. 기각 조건: 스파이크 구간의 최장 STW가 p99 상승분의 20% 미만이면 기각. 비용: 낮음(기존 GC 로그/JFR 상관만). → 1순위 병렬.
- **H2 커넥션 풀 고갈.** 예측: 스파이크 시 풀 획득 대기(acquire wait)가 치솟고 active==max 구간 존재. 기각 조건: 스파이크 동안 active < max 이면서 acquire wait ≈ 0이면 기각. 비용: 낮음(풀 메트릭이 이미 노출된 경우). → 1순위 병렬.
- **H3 이웃 소음(공유 인프라).** 예측: 스파이크가 특정 물리 호스트/노드/AZ에 상관하고 CPU steal time이 오르며 앱 내부 지표는 정상. 기각 조건: steal time과 호스트가 스파이크와 무상관(여러 호스트에 균등)이면 기각. 비용: 중간(호스트/노드 태그로 재집계 필요 — 좌표 전환과 연결).
- **H4 재시도 폭풍.** 예측: 다운스트림 지연 → 재시도 → 부하 증폭의 자기 강화. 스파이크 직전 인입 RPS(재시도 포함)가 급증. 기각 조건: 스파이크 직전/중의 재시도 포함 RPS가 평소와 통계적으로 같으면 기각. 비용: 중간(재시도를 원요청과 attempt 카운트로 분리해 세야 함).
- **H5 특정 페이로드/엔드포인트/테넌트.** 예측: 느린 요청이 특정 파라미터·큰 페이로드·특정 테넌트에 집중. 기각 조건: 느린 요청의 페이로드 크기·엔드포인트·테넌트 분포가 정상 요청과 구분 불가면 기각. 비용: 중간(느린 요청만 샘플 태깅).

미등록 후보(커버리지 스윕 결과): 락 경합, 캐시 스탬피드, DNS/TLS 재협상, 스레드풀 포화(풀과 별개). 위 5개가 전부 기각/보류로 소진되면 원장에 정식 편입한다.

## 좌표계 의심 — 집계 단위를 바꿔야 보이는 군집

이 프레임의 핵심 함정: 같은 데이터가 좌표를 바꾸면 다른 군집을 드러낸다. 사전 등록 좌표 후보 — {인스턴스, 물리 호스트/노드, AZ, 엔드포인트, 테넌트, 배포 버전, 시간 버킷 해상도(1s/10s/1m), UTC vs 로컬시간·크론 주기}. 글로벌 p99는 매끈해도 1초 버킷에선 마이크로버스트로 뾰족할 수 있고, UTC로 균등해도 배치 크론 주기로 묶일 수 있다.

**규칙(사전 등록): 등록된 좌표 전환 스윕을 돌리기 전에는 어떤 가설도 "글로벌 집계로 안 보인다"만을 근거로 기각하지 않는다.** 기각 조건은 반드시 해당 가설에 맞는 좌표(H3=호스트, H5=테넌트, 시간축=1초)에서 평가한다.

## 접근 및 단계 (각 단계에 검증 방법)

1. 사실/추측 분리 확정 + 좌표 후보 등록. 검증: 사실 목록의 모든 항목에 측정 출처가 붙어 있고, 나머지는 전부 원장에 있다.
2. 관측 최소 보강(가설이 요구하는 신호만: 트레이스 상관 ID, GC 로그, 풀 메트릭, 호스트 태그, 재시도 attempt). 검증: 과거 스파이크 1건에 대해 5개 가설 신호를 동시에 조회할 수 있다.
3. 싼 실험 병렬 실행(H1·H2 먼저). 검증: 각 가설을 사전 등록 기각 조건에 따라 {확정/기각/보류}로 판정해 원장에 기록.
4. 좌표 전환 스윕. 검증: 등록된 좌표마다 재집계하고 군집 유무를 기록. 글로벌에서 보류였던 가설을 좌표별로 재판정.
5. 생존 가설로 재현 시도(부하·페이로드·재시도 시퀀스 재구성). 검증: 통제 환경에서 스파이크를 의도적으로 재발생시킨다.
6. 루프: 관측 → 원장 갱신. 새 사실이 나오면 사실/추측 경계와 기각 판정을 갱신하고 3~5를 반복.

## 대안과 기각 사유 (부활 조건 포함)

- 즉시 완화 투입(타임아웃 축소·재시도 차단·인스턴스 증설). 기각: 원인 미상 상태의 완화는 증상만 이동시키고(재시도 차단은 에러율을 올릴 수 있음) 재현 레시피 없이 재발을 판정할 수 없다. 부활: 가정 4가 깨져 스파이크가 SLO/매출을 실시간 위협하면 triage로 전환해 임시 완화를 먼저 하고 진단은 병행.
- 전 구간 대량 계측(모든 것을 켠다). 기각: 가설 없는 계측 산포는 이 프레임의 명시된 실패 모드이며 관측자 효과로 스파이크를 왜곡한다. 부활: 5개 가설이 전부 기각되고 새 가설도 안 나오면 넓은 탐색적 계측 1회를 허용.
- APM 벤더 자동 이상탐지에 위임. 기각: 대개 글로벌 집계 기반이라 좌표 전환으로만 보이는 군집을 놓친다. 부활: 벤더가 임의 좌표 재집계(테넌트·1초 버킷)를 지원하면 재고.

## 리스크 & 완화

- R1 사후 해석 편향(원하는 가설로 데이터 끼워맞춤). → 기각 조건 사전 등록 + 판정을 데이터 열람 전에 문서화.
- R2 조기 수렴(GC만 판다). → 싼 가설 병렬 강제, 1개가 확정되기 전까지 나머지를 "보류"로 유지.
- R3 좌표 함정(글로벌에서 안 보여 기각). → 좌표 전환 스윕 전 기각 금지 규칙.
- R4 관측자 효과(계측 자체가 부하). → 샘플링, 느린 요청만 태깅.
- R5 조사 기간에 스파이크가 안 뜸. → 재현 레시피(수동 재현)를 우선하고 자연 발생 대기를 최소화.

## "완료"의 판정 기준

- 5개 가설 각각이 사전 등록 기각 조건에 따라 {확정/기각/보류}로 원장에 기록됨(실측).
- **통제 환경에서 특정 명령·부하 시퀀스를 실행하면 p99 스파이크가 재현되고, 그 시퀀스를 제거하면 스파이크가 사라진다 — 이 재현 레시피 1건이 문서화되어 있다.** (재현 판정 문장)
- 최소 1개 등록 좌표에서 스파이크 군집이 확인되거나, 모든 등록 좌표에서 "군집 없음"이 기록됨.
- 완화책은 재현 레시피로 검증됨: 완화 적용 후 동일 시퀀스를 실행해도 스파이크가 재현되지 않는다.

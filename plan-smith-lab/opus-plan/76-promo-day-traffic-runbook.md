# 대형 프로모션 당일 트래픽 폭주 대응 런북
- 추론 프레임: triage
- 한 줄 요약: 폭주 순간의 행동을 가용 시간 티어(즉시/30분/사후)로 강제 분류하고, 각 행동에 부작용과 발동조건을 병기하며, 압박받는 뇌가 반사적으로 손대는 나쁜 수(전체 캐시 퍼지·무분별 스케일아웃)를 사전 차단한다.

## 프레임이 요구하는 시작점 — 가용 시간 티어와 90초 분류

이 과제를 보자마자 내 반사 습관은 "대시보드를 열고 병목을 근본까지 진단하자"였다. triage는 그걸 금지한다. 프로모션은 이미 라이브고 매출이 초 단위로 새는 중이며, 초동 대응자에게는 진단할 인지 여유가 없다. 프레임은 "완전 진단"도 "맹목적 지혈"도 아닌, 그 사이의 초저비용 분류에서 시작하라고 요구한다.

### 90초 분류 (예산 상한: 90초, 초과 시 기본 경로로 강제 이동)
대시보드 한 화면만 본다 — 체크아웃/결제 경로의 에러율, p95 지연, 그리고 어느 티어(LB / 앱 / DB / 캐시)가 빨간지. 이 세 신호로 아래 중 하나로 분류한다.

- A. 전면 다운 (거의 모든 요청 5xx)
- B. 특정 경로만 실패/급감 (예: 체크아웃만 타임아웃)
- C. 느리지만 살아있음 (지연 상승, 응답은 나감)
- D. 외부 의존성 문제 (결제 게이트웨이/서드파티 지표만 빨강)

**기본 경로 (판단이 안 설 때):** 90초 안에 A~D로 분류되지 않으면 무조건 C(성능 저하)로 간주하고 즉시 티어의 부하 차단·저하 모드로 직행한다. 이유는 그 조치들이 폭발 반경이 가장 작은 안정화이면서, D(외부 의존성)를 제외한 모든 버킷에서 손해를 주지 않기 때문이다. 이때 스케일아웃·캐시 퍼지로 뛰지 않는다.

## 증거 vs 속도 트레이드오프 (명시)

안정화 행동(파드 재시작, 저하 모드 on, 커넥션 플러시)은 폭주-붕괴의 원인을 담은 라이브 증거를 파괴한다. 프로모션 당일의 원칙을 못박는다: **매출 보호가 원인 규명 충실도보다 우선한다.** 대신 안정화 직전에 30초짜리 증거세(evidence tax)를 지불한다 — 대시보드 스크린샷 1장, 현재 지표 스냅샷, 슬로우 쿼리/스레드 덤프 표본 1개. 그 후 안정화한다. 이 30초를 지불했더라도 사후에 "왜"가 복원 불가능할 수 있음을 받아들인다.

## 문제 정의 / 목표

예고된 대형 프로모션 당일, 트래픽이 평소의 수 배~수십 배로 튀어 서비스가 붕괴하거나 붕괴 직전인 상황. 목표는 근본 원인 규명이 아니라 **체크아웃/결제 경로의 생존**이다. 완료의 정의는 "폭주가 사라졌다"가 아니라 "매출 경로가 티어별 대응으로 계속 응답했다"이다.

## 명시적 가정 목록 (틀렸을 경우의 영향 포함)

- **가정 1 (전체가 걸림): 프로모션 이전에 "성능 저하 모드" 피처 플래그와 관측 대시보드가 이미 구축·검증되어 있다.** → 틀리면 즉시 티어의 대부분(저하 모드, 90초 분류)이 성립 불가하고 이 런북 자체가 무너진다. 이 가정은 사전 준비 항목이지 당일 조치가 아니다.
- 가정 2: 폭주의 원인은 예고된 프로모션이지 외부 공격(DDoS)이 아니다. → 틀리면 대응 축이 완전히 달라진다(WAF/보안 경로). 90초 분류에 "트래픽이 정상 사용자 패턴인가"를 한 신호로 포함해 조기 감별한다.
- 가정 3: 체크아웃/결제가 revenue-critical 경로이고 이를 지키는 게 최우선이다. → 틀리면(예: 조회수가 KPI인 미디어 프로모션) 티어별 보호 대상 우선순위를 재정렬해야 한다.
- 가정 4: 오토스케일이 어느 정도 설정돼 있으나 병목이 한 티어(대개 DB 커넥션)에 몰릴 수 있다. → 틀리면(스케일이 전무) 30분 티어의 스케일 조치 전제가 깨져 즉시 티어의 부하 차단에 더 의존해야 한다.
- 가정 5: on-call 1인이 초동 대응하며 에스컬레이션 경로가 존재한다. → 틀리면(다수 동시 대응) 조치 충돌 방지를 위해 "지휘자 1인" 지정을 즉시 티어 0번으로 올려야 한다.

## 접근 및 단계 (시간 티어별, 부작용·발동조건 병기)

무게중심은 즉시/30분 티어에 둔다. 사후 티어는 일부러 얇게 유지한다(아래 리스크 R-드리프트 참조).

### 즉시 티어 (0~5분)
1. 90초 분류 + 30초 증거 스냅샷 (위 절차). → 검증: 버킷 A~D 중 하나가 종이/채널에 적혔는가.
2. 저하 모드 on: 추천·리뷰·개인화 등 비필수 기능 플래그 off, 상품 페이지는 정적/캐시 응답. **부작용:** 사용자 경험 저하. **발동조건:** 버킷 A·B·C, 또는 기본 경로. → 검증: 비필수 위젯이 실제로 안 뜨고 체크아웃 p95가 꺾이는가.
3. 읽기 경로 캐시 TTL 상향(퍼지 아님). **부작용:** 가격/재고가 수 분간 stale. **발동조건:** DB/오리진이 빨강일 때. → 검증: 오리진 요청 수가 즉시 감소하는가.
4. 대기열/입장 제어(waiting room)가 있으면 켠다. **부작용:** 일부 사용자 대기. **발동조건:** 버킷 A 또는 LB가 포화. → 검증: 백엔드 유입률이 상한선 아래로 내려오는가.

### 30분 티어 (안정화 후 30분 이내)
5. 표적 스케일: 90초 분류로 지목된 단일 병목 티어만 증설한다. 증설 전 다음 티어(특히 DB 커넥션 풀) 여유를 먼저 확인한다. **부작용:** 비용 증가, 하류 압박 전이 가능. **발동조건:** 해당 티어가 CPU/메모리 포화로 확인됐을 때만. → 검증: 그 티어 포화 지표가 내려가고 하류가 새로 빨개지지 않는가.
6. DB 완화: 장기 실행 쿼리 kill, 읽기를 리드 리플리카로 라우팅. **부작용:** 리플리카 지연으로 읽기 stale. **발동조건:** DB가 병목으로 확인됐을 때. → 검증: DB 커넥션/락 지표 회복.
7. 엔드포인트별 레이트리밋 조정. **부작용:** 일부 정상 요청도 429. **발동조건:** 특정 경로가 폭주 근원일 때(버킷 B). → 검증: 해당 경로 유입이 목표치로 수렴.
8. 커뮤니케이션: 상태 페이지 갱신 + 이해관계자 1줄 보고. **부작용:** 없음(필수). → 검증: 상태 페이지에 현재 상황과 저하 범위가 게시됨.

### 사후 티어 (압력 해소 후 — 의도적으로 얇게)
9. 저하 모드 원복: **느리고 조건부로.** 지표가 N분(예: 10분) 연속 안정일 때만 플래그를 단계적으로 되돌린다. → 검증: 원복 후에도 p95가 재상승하지 않는가.
10. 증거 회수 + 포스트모템 예약 + 용량 백로그 1줄. 여기서 멈춘다(상세 설계는 이 런북의 범위 밖).

## 대안과 기각 사유 (부활 조건 포함)

- **대안: 티어 없이 단일 "붕괴 시 체크리스트"로 통합.** 기각 — 가용 시간이라는 축을 지우면 압박 상황에서 순서 판단을 다시 대응자에게 떠넘긴다. triage의 핵심(시간으로 강제 분류)이 사라진다. **부활 조건:** 조직이 저하 모드·스케일을 모두 자동화해 사람이 내릴 판단이 on/off 하나로 줄면, 단일 체크리스트가 더 낫다.
- **대안: 프로모션 전면 근본 진단부터(내 반사 습관).** 기각 — 매출이 새는 동안 진단은 지연이다. **부활 조건:** 버킷 D처럼 안정화 조치로 개선되지 않고 원인이 외부에 고정된 경우, 진단(게이트웨이 상태 확인)이 곧 유일한 유효 행동이 된다.

## 금지 조치 (압박받는 뇌가 반사적으로 손대지만 절대 하지 말 것)

- **캐시 전체 퍼지.** 폭주 중 전체 퍼지는 캐시 스탬피드(thundering herd)를 유발해 모든 요청이 동시에 오리진/DB로 몰려 더 빠른 붕괴를 보장한다. "이상하니 캐시 비우자"는 본능이 정확히 최악이다. → 대신 특정 키 표적 무효화 또는 TTL 상향.
- **무분별 스케일아웃.** 모든 티어를 한꺼번에 늘리면 병목이 하류로 전이된다(앱 파드↑ → DB 커넥션 고갈 → DB 전면 붕괴). 비용도 폭증. → 지목된 단일 병목만, 하류 여유 확인 후.
- **검증 없는 배포/핫픽스.** 피크 부하에서 미검증 코드 경로는 새 장애원. **예외(발동조건):** 폭주 직전 배포가 명백한 원인이면 롤백이 곧 처방.
- **재시작 난사.** 웜 캐시/커넥션 풀이 날아가고 재접속 폭풍이 인다.
- **프로덕션 인덱스/스키마 변경.** 락으로 상황을 악화시킨다.

## 리스크 & 완화

- **R1 오분류:** 90초 분류가 틀려 엉뚱한 티어 조치. → 완화: 기본 경로(저하 모드)는 오분류에도 손해가 작도록 설계됨. 조치 후 지표가 개선 안 되면 재분류.
- **R2 저하 모드 미작동:** 가정 1이 깨진 경우. → 완화: 프로모션 전 리허설로 플래그 실동작을 반드시 사전 검증(당일 조치 아님).
- **R3 사후 티어 비대화(R-드리프트):** 압력이 풀리면 포스트모템·용량계획으로 문서가 부풀어 무게중심이 뒤로 쏠림. → 완화: 사후 티어를 2개 항목으로 캡, 상세는 별도 문서로 분리.
- **R4 다인 대응 충돌:** 여러 명이 동시에 상반된 조치. → 완화: 지휘자 1인 지정, 모든 조치를 채널에 1줄 선언 후 실행.

## "완료"의 판정 기준

- 폭주 피크 구간 동안 **체크아웃/결제 경로의 성공 응답률이 목표선(예: 95%) 이상을 유지**했음이 지표로 실측된다.
- 초동 대응자가 **첫 조치까지 90초 분류 + 30초 증거세 = 2분 이내**에 진입했음이 타임스탬프로 확인된다.
- 저하 모드가 켜졌고, **원복은 지표 N분 연속 안정 이후에만** 단계적으로 일어났다(즉시 원복이 없었다).
- 금지 조치(전체 퍼지·무분별 스케일아웃) 중 어느 것도 대응 로그에 등장하지 않는다.

# 백엔드 개발자를 위한 React 8주 교정 계획
- 추론 프레임: misconception-first
- 한 줄 요약: React 문법을 난이도 순으로 쌓지 않는다. 백엔드 멘탈모델(요청-응답·동기 흐름·상태는 DB에)이 React에서 오작동하는 지점을 먼저 지도화하고, 그 오개념을 깨는 "예측→충돌→재구성" 순서를 8주의 뼈대로 삼는다.

## 프레임이 요구하는 시작점 — 간섭할 오개념 지도

반사적으로는 주차별 토픽을 난이도 순(JSX→state→hooks→라우팅)으로 줄 세우고 싶지만, 그게 바로 이 프레임이 명시적으로 금지한 실패 모드다. 그래서 뼈대를 토픽이 아니라 오개념 교정 순서로 잡는다.

이 프레임은 선지식이 있을 때만 작동하고 백지에는 무력하다. 그래서 첫 문장은 토픽이 아니라 "무엇이 간섭하는가"다. 아래는 백엔드 실무자의 기존 멘탈모델이 React에서 오작동하는 지점이다. 각 항목은 [반사적 기대 → React의 실제]다.

- **M1 요청-응답 → 반복 렌더:** "함수는 호출되면 한 번 실행되고 끝난다." → 컴포넌트 함수 본문은 상태가 바뀔 때마다 계속 다시 실행된다. 한 번이 아니라 수백 번.
- **M2 동기 흐름 → 스냅샷 상태:** "변수에 대입하면 다음 줄에서 바뀐 값이 보인다." → setState 직후 그 값을 읽으면 옛 값이다. 상태는 렌더 시점의 스냅샷이고 갱신은 예약된다.
- **M3 상태는 DB에 / 명령형 갱신:** "상태는 저장소에 두고 화면은 내가 직접 조작한다(innerHTML)." → UI = f(state). DOM을 만지지 않는다. 상태를 바꾸면 화면이 선언적으로 따라온다.
- **M4 이펙트 = 라이프사이클 콜백:** "useEffect는 componentDidMount(초기화 훅)이거나 onChange 콜백이다." → 이펙트는 외부 시스템과 상태를 동기화하는 탈출구다. deps는 '트리거 목록'이 아니라 '동기화 대상'이다. 대부분의 경우 이펙트는 틀린 도구다.
- **M5 데이터는 서버에서 온다 → 서버 상태 ≠ 클라 상태:** "컴포넌트에서 fetch하면 끝. 요청-응답이니까." → 서버 상태는 캐시·경합·재검증을 동반한다. useEffect 안의 fetch는 경합(race)과 정리(cleanup) 누락으로 깨진다.
- **M6 객체는 가변 → 불변성:** "객체 필드를 바꾸고 배열에 push한다." → state를 직접 mutate하면 참조가 그대로라 리렌더가 안 일어난다. 새 참조로 교체해야 한다.
- **M7 계층 호출 → 단방향 데이터 흐름:** "부모가 자식을 서버 부르듯 호출하고 응답받는다." → 데이터는 props로 아래로만 흐르고, 변경은 콜백으로 위로 올린다.

교정 순서는 난이도가 아니라 **의존성**으로 정한다: M1·M3(렌더 모델)이 안 깨지면 M2가 안 보이고, M2·M6이 안 서면 M4(이펙트)는 반드시 오작동한다. 이 사슬이 8주의 뼈대다.

## 봉인 목록 (지금은 일부러 건드리지 않는다)

오개념이 교정되기 전에 라이브러리를 주면, 학습자는 미교정 상태를 라이브러리 뒤로 숨기고 통증(=배워야 할 이유)을 못 느낀다. 아래는 봉인하고 해제 조건을 명시한다.

- **상태관리 라이브러리(Redux/Zustand/Jotai):** 봉인. 해제 조건 — 7주차에 prop drilling과 Context의 리렌더 비용을 직접 겪고 "왜 필요한가"를 자기 언어로 설명할 수 있을 때(8주차).
- **서버 상태 라이브러리(TanStack Query/SWR):** 봉인. 해제 조건 — 6주차에 useEffect fetch의 경합·중복요청·캐시 부재를 손으로 재현해 통증을 확인한 직후(6주차 말).
- **메타프레임워크(Next.js/RSC/SSR):** 봉인. 해제 조건 — 이 8주 밖. React 자체의 렌더 모델이 서기 전엔 SSR이 오히려 M1을 강화한다.
- **성능 최적화(useMemo/useCallback/memo):** 봉인. 해제 조건 — 렌더 모델(M1)이 교정된 뒤에만. 그 전엔 "왜 재렌더되는지"도 모른 채 최적화 주문을 외우게 된다.

## 문제 정의 / 목표

목표는 "React 기능을 안다"가 아니라 "백엔드 멘탈모델이 오작동하는 7개 지점이 실측 가능하게 교정되었다"이다. 산출물은 앱이 아니라 교정된 예측력.

## 명시적 가정 목록 (틀렸을 경우의 영향)

- **가정 0 (프레임 전체가 걸림): 학습자는 간섭할 백엔드 선지식을 실제로 가지고 있다.** 틀리면(실은 프론트도 백지) misconception-first는 무력화되고 이 계획 전체가 부적합 → 난이도 순 표준 커리큘럼(대안 A)으로 전환해야 한다. 1주차 진단으로 조기 확인.
- 가정 1: JS/ES6(클로저·구조분해·배열메서드)를 읽을 수 있다. 틀리면 M2·M6 충돌이 "React 문제"가 아니라 "JS 문제"로 오귀인된다 → 0.5주 JS 선행 삽입.
- 가정 2: 주당 10–12시간 확보. 틀리면 8주 압축 불가 → 주차를 늘리되 교정 순서는 유지.
- 가정 3: 학습자가 "틀린 예측을 글로 남기는" 불편한 검증을 수용한다. 틀리면 노출식 학습으로 회귀해 오개념이 화석화된다.

## 접근 및 단계 (예측→충돌→재구성; 각 주차에 검증 방법)

각 주차 공통 구조: (1) 예측 — 코드를 실행 전에 결과를 글로 적는다. (2) 충돌 — 실행해 예측과 어긋남을 확인한다(어긋남 = 오개념 확인). (3) 재구성 — 왜 어긋났는지 새 모델로 설명을 다시 쓴다.

- **W1 렌더 모델 (M1·M3):** 본문에 카운터를 두고 상태 변경 시 함수가 반복 실행됨을 관찰. 검증 — "이 로그가 몇 번 찍힐까"를 먼저 적고 실측과 일치시킨다.
- **W2 상태 타이밍 (M2):** setState 직후 값을 읽어 옛 값이 나옴을 확인. 검증 — 연속 setCount(c+1)×3의 최종값을 실행 전에 예측→실측.
- **W3 불변성 (M6):** 배열 push vs 새 배열 교체로 리렌더 유무 비교. 검증 — "어느 쪽이 리렌더되는가"를 이유와 함께 즉답.
- **W4 이펙트의 의미 (M4-1):** 이펙트를 라이프사이클로 착각한 코드에서 stale closure 버그 재현. 검증 — deps 누락 버그를 스스로 예측해서 만들어낸다.
- **W5 이펙트 절제 (M4-2):** "이 로직에 이펙트가 필요한가?" 판정. 파생 상태·이벤트 핸들러로 불필요 이펙트 제거. 검증 — 불필요 이펙트 3개를 찾아 제거하고 동작 동일 확인.
- **W6 서버 상태 (M5):** useEffect fetch로 경합·중복요청을 손으로 재현 → 통증 확인 후 서버 상태 라이브러리 봉인 해제. 검증 — 빠른 탭 전환으로 경합 버그를 재현시킨다.
- **W7 데이터 흐름 (M7):** prop drilling → lifting state up → Context와 그 리렌더 비용 체감. 검증 — 상태를 어디로 올려야 하는지 근거와 함께 배치한다.
- **W8 봉인 해제 & 통합:** 통증이 쌓인 상태에서 상태관리/서버상태 라이브러리 해제, 통합 프로젝트로 7개 교정이 유지되는지 검증(대안 B 부활).

## 대안과 기각 사유 (부활 조건)

- **대안 A — 난이도 순 표준 커리큘럼(JSX→state→hooks→라우팅):** 기각. 이 프레임의 실패 모드(목록만 만들고 난이도 순으로 가르치기) 그 자체. 부활 조건 — 가정 0이 깨져 학습자가 백지로 판명되면 오히려 이게 정답.
- **대안 B — 프로젝트 우선(바로 앱 제작):** 기각. 백엔드는 앱을 요청-응답+명령형으로 "동작하게" 완성해버려 오개념이 교정 없이 화석화된다(동작하니 맞다고 믿음). 부활 조건 — 교정이 끝난 8주차엔 최고의 통합 검증 → 실제 부활시킴.
- **대안 C — 공식문서/튜토리얼 정독:** 기각. 노출(exposition)로는 오개념이 안 깨진다(프레임 핵심). 부활 조건 — 충돌 직후 재구성 자료로는 최상 → 보조 도구로 부활.

## 리스크 & 완화

- R1 충돌 피로(좌절만 남고 교정 안 됨): 충돌은 하루 1개로 제한, 충돌 직후 즉시 재구성 세션.
- R2 방어기제("React가 이상한 거지 내가 틀린 게 아니다"): 예측을 실행 전에 글로 못박아 사후 합리화를 차단.
- R3 봉인 붕괴(튜토리얼이 Redux/Next를 조기 투입): 커리큘럼 소스를 봉인 목록 기준으로 사전 필터링.
- R4 가정 0 오판: 1주차 진단 과제로 조기 발견, 즉시 대안 A로 전환.

## "완료"의 판정 기준

- 대표 실측: **state를 직접 mutate하는 코드와 새 참조로 교체하는 코드를 나란히 주면, 어느 쪽이 리렌더를 유발하는지 이유와 함께 참조 없이 즉답한다(M6 교정 실측).**
- 연속 setCount(c+1) 3회의 최종값을 실행 전에 정확히 예측한다(M2 교정 실측).
- 임의의 useEffect를 보고 "이 이펙트가 필요한가/불필요한가"를 판정하고, 불필요하면 대체 수단을 댄다(M4 교정 실측).
- 봉인 해제 시점에, 각 라이브러리를 배우기 전에 "왜 이게 필요한가"를 스스로 겪은 통증으로 설명한다(봉인 규율 실측).
- 커버리지 스윕(빠뜨린 범주 점검): 라우팅·폼·테스트·스타일링·TypeScript는 이 8주에서 의도적으로 제외한다(오개념 교정이 아니라 기능 학습 영역). 성능·상태관리·서버상태·SSR은 봉인 목록에서 해제 조건과 함께 관리된다 — 누락이 아니라 결정.

# 새벽 3시 프로덕션 DB 커넥션 고갈 장애 런북
- 추론 프레임: 시간압박 우선순위
- 한 줄 요약: "5분 안 / 30분 안 / 사후" 3계층으로만 구성하고, 각 조치의 부작용을 반드시 병기한다.

## 프레임이 요구하는 시작점 — 지금 새벽 3시, 시계가 돌아간다

이 문서는 분석 보고서가 아니라 **잠에서 깬 당직자가 순서대로 실행할 런북**이다. 그래서 계층은 심각도가 아니라 **가용 시간**으로 나눈다. 각 조치에는 "이걸 하면 무엇이 나빠지는가(부작용)"를 반드시 붙인다 — 새벽엔 부작용을 잊고 지르기 쉽기 때문이다.

## 문제 정의 / 목표

프로덕션 DB 커넥션 풀이 고갈되어 앱이 DB에 못 붙는다. 목표는 **원인 규명이 아니라 서비스 복구**(1차), 그다음 재발 방지(사후). 새벽엔 "왜"보다 "지금 멈춘 걸 돌린다"가 먼저다.

## 계층 1 — 5분 안에 할 일 (지혈)

목표: 사용자 영향 중단. 근본 원인 몰라도 실행.

1. **현상 확인:** 커넥션 수/최대치, 앱 에러율, DB CPU를 대시보드에서 즉시 확인. (부작용: 없음. 단 확인에 시간 뺏기지 말 것 — 30초.)
2. **앱 인스턴스 롤링 재시작:** 커넥션 누수(닫히지 않은 커넥션)면 재시작이 풀을 즉시 회수. → **부작용: 진행 중이던 요청 유실, 재시작 순간 에러 스파이크. 누수가 아니라 부하 폭주가 원인이면 재시작해도 곧 재고갈.**
3. **커넥션 풀 최대치 임시 상향(가능하면):** → **부작용: DB 쪽 `max_connections`를 넘기면 DB가 먼저 죽는다. 앱 여러 대 × 풀 크기 < DB 한도인지 반드시 확인 후. 잘못하면 장애 악화.**
4. **폭주 트래픽/배치가 원인이면 그것부터 차단:** 방금 돈 배치·크론·대량 리포트 쿼리 kill. → **부작용: 해당 배치 미완료, 재실행 필요.**

계층 1 종료 조건: 에러율이 정상 범위로 복귀. 안 되면 계층 2.

## 계층 2 — 30분 안에 할 일 (안정화 + 원인 좁히기)

목표: 재고갈 방지, 원인 범위 확정.

1. **커넥션 점유 주체 식별:** DB에서 활성 세션·오래 잡힌 쿼리 조회(`pg_stat_activity` 등). 롱러닝 쿼리/트랜잭션이 커넥션을 물고 있는가. → **부작용: 조회 자체는 안전하나, 세션 kill 시 해당 트랜잭션 롤백.**
2. **범인 유형 분기:**
   - (a) **커넥션 누수**(앱이 반환 안 함): 문제 배포가 최근이면 **직전 버전으로 롤백**. → 부작용: 롤백 버전의 기능 손실, 마이그레이션 있었으면 롤백 복잡.
   - (b) **롱러닝 쿼리**가 풀 점유: 해당 쿼리 kill + 원인 엔드포인트 일시 차단/레이트리밋. → 부작용: 그 기능 일시 중단.
   - (c) **트래픽 폭주**: 스케일아웃 또는 레이트리밋. → 부작용: 스케일아웃은 DB 부하를 더 키울 수 있음(앱↑ = 커넥션↑), DB가 병목이면 역효과.
3. **가드레일 임시 설정:** 커넥션 획득 타임아웃·statement_timeout을 짧게 걸어, 하나가 풀 전체를 오래 못 물게. → **부작용: 정상적으로 느린 쿼리도 잘려 일부 기능 에러.**

계층 2 종료 조건: 30분 이상 재고갈 없음 + 원인 유형(a/b/c) 확정.

## 계층 3 — 사후에 할 일 (재발 방지)

목표: 같은 새벽을 두 번 겪지 않기.

1. 원인별 근본 수정: 누수면 커넥션 반환 보장(try-finally/풀 설정), 롱쿼리면 인덱스/쿼리 개선, 폭주면 자동 스케일·백프레셔.
2. **관측성·알림:** 커넥션 사용률 임계 알림을 "고갈 후"가 아니라 "80% 도달"에 걸어, 다음엔 새벽 3시가 아니라 초저녁에 잡히게.
3. **런북 갱신:** 이번에 실제로 통한 조치·안 통한 조치를 이 문서에 반영. 부작용 관찰값 업데이트.
4. 회고: 왜 5분 안에 못 잡혔나(관측성? 접근권한? 문서 부재?)를 프로세스 문제로 다룸.

## 명시적 가정 목록 (틀렸을 경우의 영향 포함)

- **가정 1:** 당직자가 앱 재시작·DB 세션 조회 권한을 새벽에 즉시 가진다. → 틀리면 계층 1이 "권한 있는 사람 호출"로 지연되어 런북 전체가 무력화. **가장 치명적 가정.**
- **가정 2:** 대시보드로 커넥션 수를 실시간 볼 수 있다. → 틀리면 계층 1의 30초 현상 확인이 불가.
- **가정 3:** 최근 배포 이력을 즉시 조회 가능. → 틀리면 롤백(2-a) 판단이 늦어짐.

## 대안과 기각 사유

- **원인부터 완전히 규명 후 조치:** 기각. 새벽 3시엔 원인 규명 중 사용자 피해가 누적. 시간압박 프레임의 존재 이유가 "복구 우선".
- **무조건 재시작(만능키 취급):** 기각. 부하 폭주가 원인이면 재시작은 무의미하고 에러만 유발. 그래서 각 조치에 부작용·적용 조건을 병기.

## "완료"의 판정 기준

- **계층 1 실행만으로도 서비스가 일단 복구**된다(런북의 1차 목적).
- 모든 조치에 **부작용과 적용 조건**이 병기돼, 새벽에 잘못 지를 위험이 줄었다.
- 계층 3에서 알림 임계값이 "고갈 전"으로 당겨져, 다음 발생이 계층 1에 도달하기 전에 잡힌다.

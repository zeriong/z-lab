# Plan Playground — 추론 스타일 관찰용 플랜 대량 생성 지시서

> 이 파일을 Claude Code에 그대로 전달하면, 아래 정의된 모든 플랜을 **순차적으로** 생성한다.
> 목적: 서로 다른 도메인 × 서로 다른 추론 프레임에서 플래닝이 어떻게 달라지는지 관찰하기 위한 샘플 수집.

---

## 실행 규칙 (Claude Code가 반드시 따를 것)

1. 작업 디렉토리에 `plans/` 폴더를 생성한다.
2. 아래 목록의 각 항목을 **순서대로** 하나씩 처리하며, 항목당 `plans/NN-슬러그.md` 파일 하나를 생성한다. (예: `plans/01-realtime-collab-editor.md`)
3. 각 플랜은 반드시 지정된 **[추론 프레임]** 을 따라 작성한다. 프레임이 요구하는 사고 순서를 어기지 말 것. (예: Backward 프레임이면 인수 조건을 문서 최상단에 먼저 쓰고 역산할 것)
4. 플랜 본문은 아래 공통 템플릿을 사용하되, 프레임이 별도 구조를 요구하면 프레임을 우선한다.
5. 모든 플랜 생성이 끝나면 `plans/00-meta-review.md` 를 마지막에 작성한다. 내용: 각 플랜에서 자신이 실제로 어떤 사고 경로를 탔는지 자가 분석 + 프레임별로 결과물이 갈라진 지점 비교표.
6. 코드 구현은 하지 않는다. 오직 플랜 문서만 작성한다.
7. 각 플랜은 800~1500단어 수준. 형식적 나열 금지 — 실제로 그 프레임으로 "생각한 흔적"이 드러나야 한다.

### 공통 템플릿

```
# [플랜 제목]
- 추론 프레임: [지정된 프레임]
- 한 줄 요약

## 프레임이 요구하는 시작점 (프레임마다 다름)
## 문제 정의 / 목표
## 명시적 가정 목록 (틀렸을 경우의 영향 포함)
## 접근 및 단계
## 대안과 기각 사유 (최소 1개)
## 리스크 & 완화
## "완료"의 판정 기준
```

## A. 프로그래밍 — 그린필드 설계

**01. 실시간 협업 텍스트 에디터 (웹)**
[추론 프레임: Backward-chaining] "두 사용자가 동시에 타이핑해도 충돌 없이 수렴한다"는 최종 인수 조건 5개를 먼저 확정하고, 각 조건에서 역산하여 기술 선택(CRDT vs OT)과 단계를 도출할 것.

**02. CLI 기반 개인 재무 추적 도구**
[추론 프레임: 삭제 우선 / 미니멀리즘] "기능을 하나 추가할 때마다 왜 이것이 없으면 안 되는가"를 먼저 증명. 최종 기능 수가 5개를 넘으면 실패로 간주하고 다시 깎을 것.

**03. 오픈소스 라이브러리 — 날짜 범위 겹침 계산 유틸**
[추론 프레임: API-first] 사용자가 작성하게 될 코드 예시 10줄을 문서 최상단에 먼저 쓰고, 그 API를 실현하는 방향으로만 내부 설계를 전개할 것.

**04. 셀프호스팅 URL 단축 서비스**
[추론 프레임: Constraint-first] 제약을 먼저 고정: 월 $5 서버 1대, 운영자 1인, 다운타임 허용 불가. 이 제약이 허용하는 해 공간 안에서만 설계하고, 제약 때문에 포기한 것들을 명시할 것.

## B. 프로그래밍 — 레거시 / 리팩토링

**05. 10년 된 jQuery 관리자 페이지의 점진적 React 전환**
[추론 프레임: 최소 diff 보수형] 매 단계가 독립 배포·독립 롤백 가능해야 함. "빅뱅 전환" 유혹이 드는 지점마다 그 유혹을 명시하고 기각할 것.

**06. 순환 의존이 얽힌 백엔드 모듈 분리**
[추론 프레임: Middle-out] 가장 불확실한 지점(어떤 순환이 진짜 병목인지)을 검증하는 분석 spike를 플랜의 중심에 놓고, 그 결과의 분기에 따라 앞뒤 단계가 달라지는 조건부 플랜을 작성할 것.

**07. 테스트가 0개인 결제 모듈에 테스트 도입**
[추론 프레임: 리스크 확률×피해 정렬] 발생 확률과 피해 크기를 곱한 점수로 시나리오를 정렬하고, 상위 항목부터 커버하는 순서로만 단계를 구성할 것. 커버리지 %는 목표로 삼지 말 것.

## C. 프로그래밍 — 디버깅 / 장애 대응

**08. "가끔, 특정 사용자에게만, 재현 불가" 버그 조사 플랜**
[추론 프레임: 가설-실험 사이클] 가설을 3개 이상 세우고, 각 가설을 기각/채택할 수 있는 최소 실험을 설계. 관찰 → 가설 갱신의 루프 구조로 플랜을 작성할 것.

**09. 새벽 3시 프로덕션 DB 커넥션 고갈 장애 런북**
[추론 프레임: 시간압박 우선순위] "5분 안에 할 일 / 30분 안에 할 일 / 사후에 할 일" 3계층으로만 구성. 각 조치의 부작용을 반드시 병기할 것.

## D. 프로그래밍 — 인프라 / 시스템

**10. 스타트업 초기의 CI/CD 파이프라인 구축**
[추론 프레임: YAGNI vs 미래대비 변증법] 각 결정마다 "지금 필요파"와 "나중에 후회파"의 논쟁을 실제 대화체로 짧게 시뮬레이션한 뒤 판정을 내리는 형식으로 작성할 것.

**11. 하루 1억 이벤트 로그 수집 파이프라인**
[추론 프레임: 봉투 뒷면 계산 우선] 설계 전에 처리량·저장량·비용의 자릿수 추정(back-of-envelope)을 먼저 수행하고, 그 숫자가 모든 기술 선택의 근거가 되게 할 것.

## E. 사업 / 프로덕트

**12. 동네 반려동물 돌봄 매칭 서비스 사업계획서**
[추론 프레임: Pre-mortem] "18개월 후 이 사업은 망했다"를 전제로 부검 보고서를 먼저 쓰고, 그 사인(死因)들을 예방하는 구조로 사업계획을 역구성할 것.

**13. 개발자 대상 유료 뉴스레터 런칭 계획**
[추론 프레임: 단위경제 우선] 구독자 1명의 LTV/CAC 추정 모델을 먼저 세우고, 숫자가 성립하는 조건에서만 실행 계획을 전개. 숫자가 성립 안 하면 성립하는 피벗안을 제시할 것.

**14. 사내 카페 무인화 제안서 (이해관계자 설득용)**
[추론 프레임: 페르소나 렌즈 병렬] CFO / 총무팀장 / 카페 이용 직원 3인의 관점에서 각각 반대 논거를 먼저 작성하고, 세 반대를 모두 통과하는 안으로 수렴시킬 것.

**15. 포화 시장(할일 관리 앱) 진입 전략**
[추론 프레임: Analogical] 포화 시장을 뚫은 실제 유사 사례 3개의 구조를 먼저 분석하고, 각 사례의 패턴을 이식/기각하는 방식으로만 전략을 도출할 것.

## F. 게임 기획

**16. 30초 플레이 루프의 하이퍼캐주얼 모바일 게임**
[추론 프레임: 감정 곡선 역산] 플레이어가 느껴야 할 감정 시퀀스(긴장→해소→아쉬움→재도전 욕구)를 먼저 그래프로 정의하고, 각 감정을 유발하는 메카닉을 역으로 배치할 것.

**17. 협동 보드게임 (2~4인, 60분)**
[추론 프레임: 실패 조건 우선] "재미없어지는 상태"(쿼터백 문제, 다운타임, 킹메이킹 등)를 먼저 열거하고, 각 실패 모드를 구조적으로 봉쇄하는 규칙만 채택하는 방식으로 기획할 것.

**18. 텍스트 기반 로그라이크의 절차적 생성 시스템**
[추론 프레임: First-principles] "왜 절차적 생성이 재미를 만드는가"를 공리 수준에서 3개 이내로 정의하고, 기존 장르 관례를 인용하지 않고 그 공리에서만 시스템을 도출할 것.

**19. 라이브 서비스 게임의 시즌 1 콘텐츠 로드맵**
[추론 프레임: 자원 배분 트레이드오프] 개발팀 6인×12주라는 고정 예산 안에서 콘텐츠 후보들의 기회비용을 명시적으로 비교하는 포트폴리오 형식으로 작성할 것.

## G. 학습 / 커리어 / 개인

**20. 백엔드 개발자의 3개월 Rust 학습 계획**
[추론 프레임: 오개념 예측 우선] 이 학습자가 기존 지식 때문에 겪을 오개념(예: GC 멘탈모델)을 먼저 예측·목록화하고, 오개념 교정 시점을 커리큘럼의 뼈대로 삼을 것.

**21. 5년차 개발자의 이직 준비 6주 플랜**
[추론 프레임: 옵션 병렬 비교] 단일 플랜 금지. "대기업 집중 / 스타트업 집중 / 해외 원격" 3개 트랙을 트레이드오프 매트릭스로 병렬 설계하고, 선택 기준을 독자에게 위임하는 형식으로 작성할 것.

**22. 번아웃 상태에서의 업무 복귀 8주 계획**
[추론 프레임: 최악 시나리오 안전판] 매주 "이번 주 계획이 실패했을 때의 축소 플랜 B"를 반드시 병기. 낙관 경로가 아닌 안전판이 문서의 중심이 되게 할 것.

## H. 운영 / 이벤트 / 기타 도메인

**23. 200명 규모 개발자 컨퍼런스 운영 계획**
[추론 프레임: 타임라인 역산 + 크리티컬 패스] D-day에서 역산한 마일스톤을 세우되, 지연이 전파되는 크리티컬 패스를 명시하고 각 노드의 버퍼를 계산할 것.

**24. 동네 서점의 생존 전략 1년 플랜**
[추론 프레임: 관찰-데이터 우선] 전략 수립 전에 "무엇을 어떻게 측정/관찰할 것인가" 4주 관찰 설계를 먼저 세우고, 이후 전략은 관찰 결과의 분기(if-then)로만 서술할 것.

**25. 오픈소스 프로젝트의 커뮤니티 성장 계획**
[추론 프레임: 인센티브 설계 중심] 기여자·메인테이너·사용자 각각의 인센티브 구조를 먼저 모델링하고, 인센티브가 정렬되지 않는 정책은 아무리 좋아 보여도 기각할 것.

**26. 재난 상황(지진) 대비 가정용 행동 매뉴얼**
[추론 프레임: 체크리스트 환원주의] 모든 내용을 "판단이 필요 없는" 체크리스트로 환원. 판단이 필요한 항목이 남아 있으면 그 판단을 사전에 내려 조건문으로 바꿀 것.

## 마지막 산출물: `plans/00-meta-review.md`

모든 플랜 완료 후 작성. 반드시 포함할 것:

1. **프레임별 자가 분석**: 각 플랜에서 실제 사고가 프레임을 따랐는지, 어디서 기본 습관(관성적 플래닝)으로 회귀했는지 정직하게 기록.
2. **갈라짐 비교**: 같은 계열 프레임(예: Backward vs Middle-out, Pre-mortem vs 실패 조건 우선)이 결과물 구조를 어떻게 다르게 만들었는지 3개 이상 사례.
3. **프레임 × 도메인 적합도 표**: 어떤 프레임이 어떤 도메인에서 유효/무효했는지 26개 전체에 대한 요약 표.
4. **추천**: 이 중 실무 재사용 가치가 높은 프레임 상위 5개와 그 이유.

# CLI 기반 개인 재무 추적 도구
- 추론 프레임: 삭제 우선 / 미니멀리즘
- 한 줄 요약: 기능을 더하는 게 아니라, 후보를 깔아놓고 "없으면 안 되는 이유"를 통과하지 못한 것을 전부 지운다. 최종 4개.

## 프레임이 요구하는 시작점 — 후보 나열 후 삭제 재판

미니멀리즘 프레임은 "무엇을 넣을까"가 아니라 "무엇을 지울 수 있나"에서 시작한다. 그래서 먼저 흔한 재무 앱 기능을 **일부러 넉넉히** 깔아놓고, 각각을 삭제 재판에 세운다. 생존 조건은 단 하나: **"이게 없으면 도구가 근본 목적을 달성하지 못하는가?"** 애매하면 삭제한다(무죄추정이 아니라 유죄추정).

근본 목적 정의: *"내 돈이 어디로 나갔는지 나중에 알 수 있게 기록하고 되돌아본다."* 이 문장에 직접 기여하지 않으면 탈락.

## 삭제 재판 기록

| 후보 기능 | 판결 | 사유 |
|---|---|---|
| 거래 입력(add) | **생존** | 기록이 없으면 목적 자체가 불성립. 뿌리. |
| 기간·카테고리별 요약(report) | **생존** | "되돌아본다"의 직접 구현. 이게 없으면 그냥 로그 파일. |
| 수정/삭제(edit) | **생존** | 오타·중복 입력은 반드시 생긴다. 정정 수단이 없으면 신뢰 붕괴 → 사용 중단. |
| 플레인 텍스트 저장 | **생존(설계로 승격)** | 기능이라기보다 제약. grep/버전관리/타 도구 연동을 공짜로 줌. 이걸 지우면 데이터 종속 발생. |
| 예산 설정/경고 | 삭제 | report를 눈으로 보면 초과 여부를 안다. 자동 경고는 "있으면 좋은" 것이지 목적의 필요조건 아님. |
| 반복 거래 자동화 | 삭제 | 월 몇 번의 수동 입력을 없애자고 스케줄러·상태를 들이는 건 비용 대비 과함. |
| 다중 통화 | 삭제 | 대상 사용자(단일 통화 생활자) 가정에서 순수 오버엔지니어링. |
| 차트/그래프 | 삭제 | 터미널 차트는 정보밀도 낮음. 숫자 표가 더 정확. 시각화가 필요하면 CSV로 export... 도 삭제(아래). |
| 클라우드 동기화 | 삭제 | 플레인 텍스트 + git이면 동기화는 사용자가 이미 가진 도구로 해결. |
| 태그·검색 | 삭제 | 플레인 텍스트라 `grep`이 검색을 대체. 별도 기능 불필요. |
| 계좌 다중 관리 | 삭제 | 카테고리로 충분히 표현 가능. 계좌 개념은 복잡도만 추가. |
| export(CSV) | 삭제 | 저장 포맷 자체가 이미 사람이 읽고 파싱 가능한 텍스트이므로 export는 항등 연산에 가까움. |

## 문제 정의 / 목표

지출을 30초 안에 기록하고, 한 달에 한 번 "어디에 얼마 썼나"를 보는 1인용 CLI. 목표는 기능의 풍부함이 아니라 **마찰 없는 기록 + 정직한 회고**.

## 명시적 가정 목록 (틀렸을 경우의 영향 포함)

- **가정 1:** 사용자는 단일 통화, 단일 사람. → 틀리면 다중 통화 삭제 판결이 뒤집힘.
- **가정 2:** 사용자는 터미널·git에 익숙. → 틀리면 "동기화/검색을 외부 도구에 위임" 전략이 무너짐(이 위임이 삭제의 핵심 논거였음).
- **가정 3:** 데이터 규모는 연 수천 건 이하. → 틀리면(수십만 건) 플레인 텍스트 스캔 성능이 문제되어 인덱스/DB가 필요해짐.

## 최종 접근 및 단계 (생존한 4개 + 1 설계)

1. **저장 포맷 확정(설계):** 한 줄 = 한 거래. `날짜 | 금액 | 카테고리 | 메모`. 사람이 읽고, git에 커밋되고, grep 된다.
2. **`add`:** `fin add -12000 식비 "점심"` → 오늘 날짜로 한 줄 append. 30초 목표의 핵심.
3. **`report`:** `fin report --month 2026-07` → 카테고리별 합계 + 총계 표. "되돌아본다"의 구현.
4. **`edit`:** `fin edit <행번호>` 또는 `fin rm <행번호>` → 정정. 신뢰 유지.

기능 수 = 4 (< 5 목표 충족). 저장 포맷은 기능이 아닌 제약이므로 카운트 제외.

## 대안과 기각 사유

- **SQLite 기반 설계:** 기각. 쿼리 유연성은 매력적이지만 "플레인 텍스트 → 외부 도구 위임" 전략(검색·동기화·시각화를 grep/git/스프레드시트에 떠넘김)이 성립하지 않게 되어, 오히려 내부 기능을 더 만들어야 함. 미니멀리즘 역행.
- **TUI(대화형 화면):** 기각. add의 30초 목표는 한 줄 커맨드가 최적. 화면 상태 관리는 순수 추가 복잡도.

## 리스크 & 완화

- **R1: 삭제한 기능(예산 경고 등)을 사용자가 요구.** → report 출력을 눈으로 읽어 대체 가능함을 문서화. 진짜 반복 요청이 쌓이면 그때 재판을 다시 연다(YAGNI).
- **R2: 플레인 텍스트 손상/동시 편집.** → append-only 원칙 + git 커밋 습관 안내. 파일 락은 1인용이므로 불필요.

## "완료"의 판정 기준

- 최종 사용자 대면 기능이 **정확히 4개**이며, 각 기능이 삭제 재판(근본 목적 필요조건)을 통과했음을 표로 증명.
- `add` 1회가 30초 내 완료(단일 커맨드).
- 삭제된 기능 각각에 대해 "무엇이 그 역할을 대신하는가"가 명시됨(대체재 없는 삭제는 삭제가 아니라 기능 누락이므로).

# 하루 1억 이벤트 로그 수집 파이프라인
- 추론 프레임: 봉투 뒷면 계산 우선
- 한 줄 요약: 설계 전에 처리량·저장량·비용을 자릿수로 먼저 추정하고, 그 숫자가 모든 기술 선택의 근거가 되게 한다.

## 프레임이 요구하는 시작점 — 봉투 뒷면 계산

기술을 고르기 전에 숫자부터. 아래 자릿수 추정이 이후 모든 결정을 지배한다. (정밀함이 아니라 자릿수가 목적)

**처리량(throughput)**
- 1억 events/day ÷ 86,400s ≈ **1,160 events/s 평균**.
- 트래픽은 균등하지 않음 → 피크는 평균의 ~5~10배 가정 → **피크 ~10,000 events/s**.
- 이 숫자의 의미: 초당 1만 건은 **단일 서버로 감당 가능한 영역**이다(초당 수십만이 아님). 카프카급 분산 스트리밍이 반드시 필요한 규모는 아님 → 과잉설계 경보.

**저장량(storage)**
- 이벤트당 크기 가정: 평균 1KB(JSON) → 1억 × 1KB = **~100GB/day**.
- 압축(로그는 5~10x 압축 잘 됨) 적용 → **~10~20GB/day 압축 저장**.
- 연간: 100GB/day 원본 × 365 ≈ **~36TB/year 원본**, 압축 ~4~7TB/year.
- 의미: 핫 스토리지에 전부 두면 비쌈. **수명주기(hot→warm→cold) 설계가 비용의 핵심**이 됨.

**비용(자릿수)**
- 압축 저장 ~5TB/year를 오브젝트 스토리지(월 $0.02/GB급)에 두면 ≈ 5,000GB × $0.02 = **~$100/month(누적)**. 감당 가능.
- 핫 스토리지(검색용 인덱스, 월 $0.2/GB급)에 전부 두면 10배 → 수명주기 없이는 비용 폭발.
- 수집 컴퓨트: 피크 1만 eps를 처리하는 인스턴스 1~2대 → 월 수십~수백 달러.
- 의미: **비용 지배 요인은 컴퓨트가 아니라 저장 수명주기**다. 여기에 설계 노력을 집중.

## 문제 정의 / 목표

하루 1억 이벤트를 수집·저장·조회 가능하게 한다. 목표는 "빅데이터 스택 구축"이 아니라 **위 숫자가 요구하는 최소 아키텍처**.

## 숫자가 강제한 설계 결정

- **1만 eps 피크 → 카프카 불필요(현 규모).** 단일 큐/버퍼 + 배치 적재로 충분. (숫자가 없었으면 반사적으로 Kafka를 넣었을 것.)
- **36TB/year 원본 → 수명주기 필수.** hot(최근 7~30일, 검색 가능) → cold(오브젝트 스토리지, 압축, 쿼리 드묾)로 자동 이전.
- **압축 5~10x → 저장 포맷은 컬럼형(Parquet)+압축.** 조회 패턴이 집계형이면 컬럼형이 스캔 비용을 자릿수로 낮춤.
- **피크가 평균의 10x → 버퍼링 필수.** 수집단에 버퍼(디스크 큐/관리형 스트림)를 둬 피크를 흡수, 다운스트림은 평균 속도로 배치 처리.

## 명시적 가정 목록 (틀렸을 경우의 영향 포함)

- **가정 1:** 이벤트 평균 1KB. → 틀려서 10KB면 저장량·비용이 한 자릿수 뛰어(360TB/year) 아키텍처가 분산 스토리지로 바뀜. **가장 민감한 가정.**
- **가정 2:** 피크가 평균의 ~10x 이내. → 틀려서 100x 스파이크면 단일 서버 가정이 깨지고 분산 수집 필요.
- **가정 3:** 조회는 주로 최근 데이터 + 집계형. → 틀려서 전체 기간 임의 검색이면 cold 데이터도 인덱싱해야 해 비용 폭증.
- **가정 4:** 유실 허용치가 "약간 허용"(로그이므로). → 틀려서 무손실 필수면 버퍼·복제 비용이 오르고 카프카급이 정당화될 수도.

## 접근 및 단계 (숫자 순서대로)

1. **수집 엔드포인트 + 버퍼:** 피크 1만 eps 흡수용 버퍼(디스크 기반 큐 또는 관리형 스트림). 무손실 요구(가정 4)에 따라 복제 수준 결정.
2. **배치 적재:** 버퍼에서 배치로 꺼내 Parquet+압축으로 hot 스토리지 적재. 평균 1,160 eps 기준이라 여유.
3. **수명주기 자동화:** 7~30일 후 hot→cold(오브젝트 스토리지) 자동 이전. 비용 지배 요인이므로 여기가 설계 핵심.
4. **조회:** hot은 인덱스 검색, cold는 오브젝트 스토리지 위 쿼리 엔진(스캔 기반, 느리지만 저렴).
5. **모니터링:** eps·버퍼 적체·저장 증가율을 계측해 가정 1·2가 깨지는 순간을 조기 경보(재설계 트리거).

## 대안과 기각 사유

- **처음부터 Kafka + 분산 스토리지 + 스트림 처리(Flink 등):** 기각. 봉투 계산상 1만 eps는 이 스택을 요구하지 않음. 숫자 없이 "빅데이터니까"로 지르는 대표적 과잉설계. 규모가 10~100x 커지면 그때 도입(가정 2·모니터링이 트리거).
- **전부 hot 스토리지에 보관(수명주기 없음):** 기각. 저장 비용 자릿수 계산이 즉시 부정 — 10배 비쌈.
- **무압축 원본 JSON 보관:** 기각. 5~10x 압축을 포기하면 저장·비용이 그만큼 뜀. 숫자가 컬럼형+압축을 강제.

## 리스크 & 완화

- **R1: 가정 1(이벤트 크기)이 틀리면 전체 비용 추정 붕괴.** → 실측으로 조기 검증(런칭 첫 주 실제 평균 크기 측정), 자릿수 벗어나면 재설계.
- **R2: 성장으로 규모가 10~100x 커짐.** → 모니터링에 성장률 경보. 단일→분산 전환 지점을 미리 숫자로 정의(예: 피크 10만 eps 도달 시 Kafka 도입).

## "완료"의 판정 기준

- 모든 주요 기술 선택(버퍼·저장 포맷·수명주기·조회)이 **봉투 뒷면 숫자로 정당화**됨(취향 아님).
- 실제 이벤트 크기·eps가 측정돼 **가정 1·2의 자릿수가 확인**됨.
- 월 저장 비용이 수명주기 설계로 **자릿수 이내에서 통제**됨(전량 hot 대비 ~10x 절감 확인).
